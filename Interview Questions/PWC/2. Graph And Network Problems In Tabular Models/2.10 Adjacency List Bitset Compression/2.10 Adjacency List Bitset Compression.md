# Adjacency List Bitset Compression

## Question
Construct a Power Query pipeline to compress adjacency lists into compact bitset fingerprints and implement set operations in DAX for fast neighborhood comparisons.

## Answer
WeŌĆÖll build a solution that transforms graph adjacency lists into compact integer bitmasks using Power Query (M), then leverage DAX bitwise functions to perform fast set operations (intersection, union, Jaccard similarity) on those bitmasks. This approach is ideal when the graph has at most 64 nodes, as the bitmask fits in a 64ŌĆæbit integer ŌĆō the maximum size supported by DAX bitwise functions. For larger graphs we discuss extensions at the end.


## 1. Power Query Pipeline: From Adjacency Lists to Bitset Fingerprints

Assume we have an `Edges` table with two columns: `Source` and `Target` (each representing a node ID). WeŌĆÖll create a nodeŌĆætoŌĆæbitŌĆæposition mapping and then compute a bitmask for every nodeŌĆÖs neighbourhood.

### Step 1: Create a node index table
- Collect all distinct node IDs from both `Source` and `Target`.
- Assign a unique 0ŌĆæbased index to each node.

```powerquery
let
    // Assume "Edges" is already loaded
    SourceNodes = Table.Distinct(Table.SelectColumns(Edges, {"Source"})),
    TargetNodes = Table.Distinct(Table.SelectColumns(Edges, {"Target"})),
    AllNodes = Table.Distinct(Table.Combine({SourceNodes, TargetNodes})),
    #"Renamed Columns" = Table.RenameColumns(AllNodes, {{"Source", "Node"}}), // adjust if needed
    #"Added Index" = Table.AddIndexColumn(#"Renamed Columns", "BitIndex", 0, 1, Int64.Type)
in
    #"Added Index"
```

Name this query `NodeIndex`.

### Step 2: Map each edge to the targetŌĆÖs bit index
- Merge `Edges` with `NodeIndex` to get the bit index of the `Target` node.

```powerquery
let
    EdgesWithIndex = Table.NestedJoin(Edges, {"Target"}, NodeIndex, {"Node"}, "TargetIndex", JoinKind.LeftOuter),
    #"Expanded TargetIndex" = Table.ExpandTableColumn(EdgesWithIndex, "TargetIndex", {"BitIndex"}, {"TargetBitIndex"}),
    #"Removed Columns" = Table.RemoveColumns(#"Expanded TargetIndex",{"Target"}) // optional
in
    #"Removed Columns"
```

### Step 3: Group by source and compute the bitmask
- Group the edges by `Source`.
- For each group, start with `0` and set the bit at each neighbourŌĆÖs index using `Number.BitwiseShiftLeft` and `Number.BitwiseOr`.

```powerquery
let
    Grouped = Table.Group(EdgesWithIndex, {"Source"}, {
        {"NeighbourIndices", each List.Transform([TargetBitIndex], (idx) => idx), type list},
        {"Bitset", each List.Accumulate([TargetBitIndex], 0,
            (state, idx) => Number.BitwiseOr(state, Number.BitwiseShiftLeft(1, idx))
        ), type number}
    })
in
    Grouped
```

Name this query `NodeBitsets`. It now contains for each node its `Bitset` (an integer) and optionally the list of neighbour indices for debugging.

*Optional:* You can also store the neighbour count (popcount) as a separate column, which is useful in DAX for quick cardinality comparisons.

```powerquery
// Add a column with neighbour count
#"Added Popcount" = Table.AddColumn(Grouped, "NeighbourCount", each List.NonNullCount([NeighbourIndices]), Int64.Type)
```


## 2. DAX Set Operations Using Bitsets

Now we have a table `NodeBitsets` with columns: `Source`, `Bitset`, `NeighbourCount`. We can use DAX measures to compare the neighbourhoods of two selected nodes.

### Helper: Bit population count in DAX
DAX does not have a builtŌĆæin function to count set bits, but we can count bits by iterating over the 64 possible bit positions. Create a calculated table `BitPositions` with a single column `Bit` containing numbers 0..63.

```dax
BitPositions = GENERATESERIES(0, 63, 1)
```

### Measures for two selected nodes
We assume you have two slicers that each select a single node from `NodeBitsets`. WeŌĆÖll use `SELECTEDVALUE` to capture the bitset of each selected node.

```dax
// Bitset of first selected node
Bitset1 = SELECTEDVALUE( NodeBitsets[Bitset] )

// Bitset of second selected node
Bitset2 = SELECTEDVALUE( NodeBitsets[Bitset] )

// Intersection bitmask (bits set in both)
IntersectionMask = BITAND( [Bitset1], [Bitset2] )

// Intersection size
IntersectionCount =
VAR AndMask = [IntersectionMask]
RETURN
    COUNTROWS(
        FILTER(
            BitPositions,
            BITAND( AndMask, BITLSHIFT( 1, BitPositions[Bit] ) ) > 0
        )
    )

// Union bitmask (bits set in either)
UnionMask = BITOR( [Bitset1], [Bitset2] )

// Union size
UnionCount =
VAR OrMask = [UnionMask]
RETURN
    COUNTROWS(
        FILTER(
            BitPositions,
            BITAND( OrMask, BITLSHIFT( 1, BitPositions[Bit] ) ) > 0
        )
    )

// Jaccard similarity (intersection / union)
JaccardIndex = DIVIDE( [IntersectionCount], [UnionCount], 0 )
```

These measures are **dynamic** ŌĆō they react to slicer selections and compute the results on the fly. Because they only scan 64 rows, performance is excellent even in large reports.

### Example usage
- Place two slicers (or a single slicer with a ŌĆ£Select first nodeŌĆØ and ŌĆ£Select second nodeŌĆØ field) bound to `NodeBitsets[Source]`.
- Create card visuals showing the intersection count, union count, and Jaccard index.
- Optionally show the bitmasks in binary format using a custom DAX expression.


## 3. Handling More Than 64 Nodes

If your graph exceeds 64 nodes, a single 64ŌĆæbit integer is insufficient. You can extend the approach by:

- **Using multiple integers**: Partition the node indices into chunks of 64 and store an array of integers. In Power Query, you can generate a record or a list of numbers. In DAX, you would need separate columns for each chunk (e.g., `BitsetChunk0`, `BitsetChunk1`, ŌĆ”). Bitwise operations must then be performed per chunk and results aggregated ŌĆō this becomes more complex but is still feasible.

- **Storing the bitset as a binary value** and using custom DAX to iterate over bytes. However, DAX does not natively handle binary data in bitwise operations, so this route is not recommended.

- **Alternative: Precompute all pairwise similarities** in Power Query if the number of nodes is manageable (e.g., a few thousand). This creates a static similarity matrix that can be quickly queried in DAX. The pipeline would compute the Jaccard index for every node pair using the bitsets and store it in a separate table. Then DAX can simply look up the value for the two selected nodes.

For most practical graphs, the 64ŌĆænode limit is a genuine constraint. If your node count is larger, consider the preŌĆæcomputation approach or using dedicated graph databases.


## 4. Complete Power Query Script (M)

Below is a consolidated script that reads an `Edges` table and outputs the `NodeBitsets` table:

```powerquery
let
    // Assume Edges is a table with columns: Source, Target
    SourceNodes = Table.Distinct(Table.SelectColumns(Edges, {"Source"})),
    TargetNodes = Table.Distinct(Table.SelectColumns(Edges, {"Target"})),
    AllNodes = Table.Distinct(Table.Combine({SourceNodes, TargetNodes})),
    #"Renamed Columns" = Table.RenameColumns(AllNodes, {{"Source", "Node"}}),
    NodeIndex = Table.AddIndexColumn(#"Renamed Columns", "BitIndex", 0, 1, Int64.Type),

    // Map each edge to target's bit index
    EdgesWithTargetIndex = Table.NestedJoin(Edges, {"Target"}, NodeIndex, {"Node"}, "TargetIndex", JoinKind.LeftOuter),
    #"Expanded TargetIndex" = Table.ExpandTableColumn(EdgesWithTargetIndex, "TargetIndex", {"BitIndex"}, {"TargetBitIndex"}),

    // Group by source and compute bitset
    Grouped = Table.Group(#"Expanded TargetIndex", {"Source"}, {
        {"NeighbourIndices", each List.Transform([TargetBitIndex], (idx) => idx), type list},
        {"Bitset", each List.Accumulate([TargetBitIndex], 0,
            (state, idx) => Number.BitwiseOr(state, Number.BitwiseShiftLeft(1, idx))
        ), type number}
    }),

    // Add neighbour count
    NodeBitsets = Table.AddColumn(Grouped, "NeighbourCount", each List.NonNullCount([NeighbourIndices]), Int64.Type)
in
    NodeBitsets
```
