## 12. **Running Total**  
    ```DAX 
    Running Total = 
    CALCULATE([Total Sales], FILTER(ALLSELECTED('Date'), 'Date'[Date] <= MAX('Date'[Date])) 
    ```  
    *Custom running aggregates without time intelligence.*

*   **Overview of the Measure**
    *   **Purpose:** This DAX formula creates a cumulative sum (Running Total) of sales up to the current point in time.
    *   **Approach:** It manually calculates the accumulation rather than using built-in Time Intelligence functions (like `TOTALYTD`).
    *   **Flexibility:** This pattern works on any column (not just dates) and respects user slicers while ignoring specific visual filters that would break the accumulation.

*   **Component Breakdown**
    *   **`Running Total =`**: This names the new measure so it can be dragged into visuals.
    *   **`CALCULATE(...)`**: The most powerful function in DAX; it modifies the **Filter Context**. It tells DAX to evaluate `[Total Sales]` under a new set of rules defined in the following arguments.
    *   **`[Total Sales]`**: This is your base measure (e.g., `SUM(Sales[Amount])`). It is the value being aggregated cumulatively.
    *   **`FILTER(...)`**: A table function that iterates through a table row-by-row. It creates a **Row Context** and returns only the rows that meet the specified condition.
    *   **`ALLSELECTED('Date')`**: This provides the table for `FILTER` to scan.
        *   It removes filters coming from the *visual* (like rows/columns in a matrix).
        *   It *keeps* filters coming from *external slicers* (like a Year Slicer).
        *   This ensures the running total resets based on slicers but accumulates across the visual's axis.
    *   **`'Date'[Date] <= MAX('Date'[Date])`**: This is the logical condition inside `FILTER`.
        *   `'Date'[Date]` (left side): Refers to the date in the current row being scanned by `FILTER`.
        *   `MAX('Date'[Date])` (right side): Refers to the date in the **current filter context** (the specific cell or bar in your chart being calculated).
        *   **Logic:** "Keep all dates that are earlier than or equal to today's date."

*   **Step-by-Step Execution Logic**
    *   **Step 1:** DAX looks at a specific cell in your report (e.g., Sales for March 3rd).
    *   **Step 2:** `MAX('Date'[Date])` captures "March 3rd" as the current endpoint.
    *   **Step 3:** `ALLSELECTED('Date')` creates a temporary list of all dates visible based on your slicers (ignoring that the visual is currently filtered to just March 3rd).
    *   **Step 4:** `FILTER` walks through that temporary list. It keeps Jan 1, Jan 2, ... up to March 3rd. It discards March 4th and onward.
    *   **Step 5:** `CALCULATE` applies this new filtered list to `[Total Sales]`.
    *   **Step 6:** The result is the sum of sales from the beginning of time (or slicer start) up to March 3rd.

*   **Understanding Context Transition**
    *   **Filter Context:** The filters applied by visuals, slicers, and rows (e.g., "Show me 2023").
    *   **Row Context:** Created by `FILTER`, allowing comparison of one row against another.
    *   **Context Transition:** When `MAX('Date'[Date])` is called inside the `CALCULATE` filter argument, DAX automatically transforms the current Row Context (from the visual axis) into a Filter Context. This is how the formula knows "which date" it is currently calculating for.

*   **Why "Without Time Intelligence"?**
    *   **Standard Time Intelligence:** Functions like `TOTALYTD` require a marked "Date Table" and rely on specific calendar hierarchies.
    *   **This Custom Pattern:** Does not require a marked date table. It works on any numeric or date column.
    *   **Granularity:** It works seamlessly at Day, Month, Quarter, or Year levels without changing the formula, whereas Time Intelligence functions sometimes require specific adjustments for different granularities.

*   **The Role of `ALLSELECTED` vs. `ALL`**
    *   **If you used `ALL('Date')`:** The running total would ignore *everything*, including Year Slicers. If you sliced for 2023, the running total might start accumulating from 2020 data if it exists in the model.
    *   **Because we use `ALLSELECTED('Date')`:** The running total respects the user's selection. If the user slices for "2023", the running total starts at Jan 1, 2023, not Jan 1, 2020.

*   **Performance Considerations**
    *   **Computationally Expensive:** `FILTER` iterates row-by-row. On very large datasets (millions of rows), this can be slower than built-in Time Intelligence.
    *   **Storage Engine vs. Formula Engine:** This formula forces more work onto the Formula Engine. Built-in Time Intelligence functions are often optimized to run closer to the Storage Engine.
    *   **Optimization:** For massive models, consider using `TOTALYTD` if possible, or ensure your Date table is optimized and relationships are active.

*   **Common Pitfalls & Troubleshooting**
    *   **Blank Values:** If there are gaps in your date table (missing dates), the running total might look jagged or incorrect. Ensure a contiguous Date table.
    *   **Multiple Date Columns:** If your model has multiple active date relationships, ensure `CALCULATE` is referencing the correct Date table used in your visual axis.
    *   **Syntax Error:** The formula provided in the prompt is missing one closing parenthesis at the very end. The correct syntax is:
        `CALCULATE([Total Sales], FILTER(ALLSELECTED('Date'), 'Date'[Date] <= MAX('Date'[Date])))`

*   **When to Use This Pattern**
    *   **Non-Standard Calendars:** When your fiscal year doesn't match standard Time Intelligence expectations.
    *   **Non-Date Running Totals:** When you need a running total based on Order ID, Step Number, or any sequential numeric column.
    *   **Specific Slicer Behavior:** When you need the running total to reset dynamically based on complex slicer interactions that `TOTALYTD` cannot handle.

*   **Summary of Value**
    *   **Control:** Gives you precise control over how the accumulation behaves.
    *   **Resilience:** Works even if the Date Table isn't formally marked in the model.
    *   **User Experience:** Ensures slicers behave intuitively (resetting the start point of the accumulation) while maintaining the cumulative effect across the visual axis.
