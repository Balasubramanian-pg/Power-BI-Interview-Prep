# Custom Connectors and Non-OAuth2 APIs in Power BI

## Overview

Custom connectors in Power BI extend connectivity to data sources not natively supported, including APIs that use authentication methods other than OAuth2. The Power Query SDK enables developers to build connectors that handle various authentication types including API keys, username/password, Windows authentication, and custom schemes.

```
Custom Connector (.mez) → Power BI Desktop → Authentication UI → API Calls → Data
         ↓                        ↓                  ↓              ↓
    Development              Installation        User Creds     Non-OAuth2 API
    (VS Code)                (Custom Connectors Folder)
```

## Supported Authentication Types for Non-OAuth2 APIs

Power Query connectors can support multiple authentication kinds defined in the `Authentication` record :

| Authentication Kind | Use Case | Implementation |
|--------------------|----------|----------------|
| **Key** | API Key authentication | Simple key in header/query |
| **UsernamePassword** | Basic authentication | Username + password credentials |
| **Windows** | Integrated Windows auth | Windows domain credentials |
| **Anonymous** | Public APIs | No credentials required |
| **OAuth** | Custom OAuth flows | Full OAuth implementation |
| **Aad** | Microsoft Entra ID | Specialized OAuth for Azure |

### Authentication Record Definition

```powerquery-m
Authentication = [
    Key = [
        KeyLabel = "API Key",  // Optional custom label
        Label = "API Key Authentication"
    ],
    UsernamePassword = [
        UsernameLabel = "Username",
        PasswordLabel = "Password",
        Label = "Basic Authentication"
    ],
    Windows = [],  // Supports Windows integrated auth
    Anonymous = []
]
```

## Building Connectors for Non-OAuth2 APIs

### 1. API Key Authentication Pattern

For APIs that require an API key in headers or query parameters :

```powerquery-m
// Data source kind definition
MyAPI = [
    Authentication = [
        Key = []
    ],
    Label = "My Custom API"
];

// Connection function
[DataSource.Kind="MyAPI", Publish="MyAPI.Publish"]
shared MyAPI.Contents = () =>
    let
        apiKey = Extension.CurrentCredential()[Key],
        headers = [
            #"X-API-Key" = apiKey,
            Accept = "application/json"
        ],
        source = Web.Contents("https://api.example.com/v1/data", [
            Headers = headers,
            ManualCredentials = true  // Prevents default auth header
        ]),
        json = Json.Document(source)
    in
        json;
```

**Key points** :
- `Extension.CurrentCredential()[Key]` retrieves the stored API key
- `ManualCredentials = true` prevents Power Query from adding default Authorization header
- Key value is also available in `Password` field but `Key` is preferred

### 2. Username/Password (Basic) Authentication

```powerquery-m
[DataSource.Kind="MyAPI", Publish="MyAPI.Publish"]
shared MyAPI.Contents = () =>
    let
        creds = Extension.CurrentCredential(),
        username = creds[Username],
        password = creds[Password],
        
        // Create Basic Auth header
        authString = Text.Combine({username, password}, ":"),
        encodedAuth = "Basic " & Binary.ToText(Text.ToBinary(authString), BinaryEncoding.Base64),
        
        headers = [
            Authorization = encodedAuth,
            Accept = "application/json"
        ],
        
        source = Web.Contents("https://api.example.com/v1/data", [
            Headers = headers,
            ManualCredentials = true
        ])
    in
        Json.Document(source);
```

### 3. Multiple Authentication Types

Connectors can support multiple authentication methods :

```powerquery-m
MyAPI = [
    Authentication = [
        Key = [],
        UsernamePassword = [],
        Windows = [ SupportsAlternateCredentials = true ],
        Anonymous = []
    ],
    Label = "Multi-Auth API"
];
```

The Power Query UI will present all supported options to users.

## Handling Non-Standard Authentication Schemes

### Custom Header Authentication

Some APIs use custom authentication schemes (e.g., `X-Auth-Token`, `App-ID`):

```powerquery-m
shared MyCustomAuthAPI.Contents = () =>
    let
        creds = Extension.CurrentCredential(),
        
        // Handle different auth types in same connector
        headers = 
            if creds[AuthenticationKind] = "Key" then
                [ #"X-App-Token" = creds[Key] ]
            else if creds[AuthenticationKind] = "UsernamePassword" then
                let
                    auth = "Basic " & 
                        Binary.ToText(
                            Text.ToBinary(creds[Username] & ":" & creds[Password]),
                            BinaryEncoding.Base64
                        )
                in
                    [ Authorization = auth ]
            else
                [ Accept = "application/json" ],
        
        request = Web.Contents("https://api.example.com/data", [
            Headers = headers,
            ManualCredentials = true
        ])
    in
        Json.Document(request);
```

### Multiple Credential Requirements

Some APIs require **both** API key and username/password. This is a limitation in the current authentication framework :

> Community discussion highlights: "The `Extension.CurrentCredential()` function only appears to be able to return the most recently entered credential. Am I missing something?"

**Workaround options** :
1. **Embed the API key** in the connector code (less secure)
2. **Combine credentials** into a single field (e.g., username:password format)
3. **Use a two-step process** - first authenticate to get token, then use token

```powerquery-m
// Workaround for APIs requiring multiple credentials
shared ComplexAPI.Contents = () =>
    let
        // Assume API key is stored as Key
        apiKey = Extension.CurrentCredential()[Key],
        
        // Prompt for additional credentials via custom UI
        username = "user@example.com",  // Would need custom input mechanism
        password = "password",
        
        // Combine both in request
        headers = [
            #"X-API-Key" = apiKey,
            #"X-User" = username,
            #"X-Pass" = password
        ],
        
        result = Web.Contents("https://api.example.com/auth", [
            Headers = headers,
            ManualCredentials = true
        ])
    in
        result;
```

## Accessing Credentials in M Code

The `Extension.CurrentCredential()` function returns a record with authentication-specific fields :

| Field | Authentication Kind | Description |
|-------|---------------------|-------------|
| `AuthenticationKind` | All | Type of credential used |
| `Username` | UsernamePassword, Windows | Username value |
| `Password` | Key, UsernamePassword, Windows | Password or Key value |
| `Key` | Key | API key value |
| `access_token` | OAuth | OAuth access token |
| `Properties` | OAuth | Additional properties (refresh_token, etc.) |
| `EncryptConnection` | All | Whether encrypted connection required |

### Example: Accessing Different Credential Types

```powerquery-m
shared AdaptiveAPI.Contents = () =>
    let
        creds = Extension.CurrentCredential(),
        authKind = creds[AuthenticationKind],
        
        result = 
            if authKind = "Key" then
                CallAPIWithKey(creds[Key])
            else if authKind = "UsernamePassword" then
                CallAPIWithBasicAuth(creds[Username], creds[Password])
            else if authKind = "Windows" then
                CallAPIWithWindowsAuth(creds[Username], creds[Password])
            else
                error "Unsupported authentication type",
        
        // Log which auth method was used (optional)
        _ = Diagnostics.LogValue("Auth method used", authKind)
    in
        result;
```

## Installation and Security

### Custom Connector Deployment 

1. **Build the connector** using Power Query SDK in VS Code
2. **Package** as `.mez` file
3. **Install** by copying to:
   ```
   [Documents]\Microsoft Power BI Desktop\Custom Connectors\
   ```
4. **Restart** Power BI Desktop
5. **Configure security** if needed:
   - **Recommended**: Sign connector with trusted certificate 
   - **Alternative**: Enable "Allow any extension" in Options > Security

### Security Settings Impact 

| Setting | Behavior | Use Case |
|---------|----------|----------|
| **Recommended** (Default) | Only signed/certified connectors load | Production, enterprise |
| **Allow any extension** | All connectors load (warning shown) | Development, trusted sources only |

> **Important**: Uncertified extensions can ignore privacy levels and send credentials over HTTP. Only enable permissive mode when you fully trust the connector source .

### Signing Custom Connectors 

```powershell
# Sign .mez file with code signing certificate
signtool sign /tr http://timestamp.sectigo.com /td SHA256 /fd SHA256 /a YourConnector.mez
```

- Timestamping preserves signature after certificate expiry
- Signed connectors load under recommended security settings
- Unsigned connectors trigger "Uncertified Connectors" dialog

## Advanced Scenarios

### Custom OAuth for Non-Standard APIs

Even for non-OAuth2 APIs, you can implement custom OAuth flows :

```powerquery-m
OAuth = [
    StartLogin = (resourceUrl, state, display) =>
        let
            authUrl = "https://api.example.com/oauth/authorize?" & 
                      Uri.BuildQueryString([
                          response_type = "code",
                          client_id = "your_client_id",
                          redirect_uri = "https://oauth.powerbi.com/views/oauthredirect.html",
                          state = state
                      ])
        in
            authUrl,
    
    FinishLogin = (context, callbackUri, state) =>
        let
            code = Uri.Parts(callbackUri)[Query][code],
            tokenResponse = Web.Contents("https://api.example.com/oauth/token", [
                Content = Text.ToBinary(Uri.BuildQueryString([
                    grant_type = "authorization_code",
                    code = code,
                    redirect_uri = "https://oauth.powerbi.com/views/oauthredirect.html",
                    client_id = "your_client_id"
                ])),
                Headers = [#"Content-Type" = "application/x-www-form-urlencoded"]
            ]),
            tokens = Json.Document(tokenResponse)
        in
            tokens
];
```

> **Note**: Never include client secrets in connectors—they run on client machines where secrets could be discovered . Use PKCE flow instead.

### Windows Authentication 

```powerquery-m
Authentication = [
    Windows = [ SupportsAlternateCredentials = true ]
];
```

The `SupportsAlternateCredentials` flag enables "Connect using alternative credentials" option, allowing explicit Windows credentials for impersonation testing .

## Troubleshooting Common Issues

| Symptom | Likely Cause | Resolution  |
|---------|--------------|-------------------------------------|
| Connector not in Get Data | Wrong folder or blocked file | Check path with `[Environment]::GetFolderPath('MyDocuments')`. Unblock file properties. |
| Uncertified dialog every launch | Unsigned connector with recommended security | Sign connector or switch to permissive mode |
| "Field 'Key' not found" error | Accessing wrong credential field | Check `AuthenticationKind` and use correct field  |
| Credential dialog loops | Unsupported auth kind or incorrect Authentication section | Verify Authentication record matches API requirements |
| Silent failure | M script error | Enable tracing (File > Options > Diagnostics). Check logs in `%LOCALAPPDATA%\Microsoft\Power BI Desktop\Traces` |
| DirectQuery fails | Multiple custom sources | Only one custom connector per DirectQuery model  |
| Load error after update | Cached previous version | Restart Desktop. Clear cache in File > Options > Data Load if needed |

## Best Practices Summary

### Security
✓ **Never embed secrets** - Connectors run client-side, secrets can be discovered   
✓ **Use PKCE for OAuth** - Avoid client secrets in public clients   
✓ **Sign connectors** - Enable recommended security settings   
✓ **Validate inputs** - Check credential fields before use  
✓ **Handle errors gracefully** - Provide meaningful error messages  

### Authentication Design
✓ **Support multiple auth types** - Give users flexibility   
✓ **Use ManualCredentials** - Prevent default auth header interference   
✓ **Document auth requirements** - Clear instructions for users  
✓ **Test all auth paths** - Verify each supported authentication kind  
✓ **Consider token refresh** - For OAuth, implement Refresh function   

### Development
✓ **Start with SDK samples** - GitHub has working examples   
✓ **Use VS Code SDK** - New features target this platform   
✓ **Test in Desktop first** - Verify before service deployment  
✓ **Enable tracing** - Debug with detailed logs   
✓ **Version control** - Track changes to .pq files  

## Limitations and Considerations

1. **Single credential limitation** - Cannot easily support APIs requiring multiple credential types simultaneously 
2. **DirectQuery restriction** - Only one custom connector per model in DirectQuery mode 
3. **Client-side execution** - Secrets cannot be protected 
4. **Conditional Access issues** - Custom OAuth for Microsoft Entra ID may fail with Conditional Access policies 
5. **Refresh limitations** - Some OAuth flows may not support scheduled refresh 

## Summary

**Key Points for Custom Connectors with Non-OAuth2 APIs:**

1. **Multiple auth types supported** - Key, UsernamePassword, Windows, Anonymous 
2. **Extension.CurrentCredential()** - Access stored credentials in M code 
3. **ManualCredentials option** - Control header behavior 
4. **Sign connectors** - Enable recommended security settings 
5. **Multiple credentials limitation** - Workarounds needed for APIs requiring both key and password 
6. **Test thoroughly** - Verify all authentication paths
7. **Document auth requirements** - Clear user instructions

**The Golden Rule:**
> Design connectors to handle one authentication kind at a time. For APIs requiring multiple credentials, consider token-based approaches or embedding one credential in the connector (with appropriate security warnings).
