# Custom M Functions: Parameterized Data Sources and Error Handling

## Overview

Custom M functions in Power Query allow you to encapsulate reusable logic for connecting to and transforming data sources. They accept parameters and return a value (typically a table), making them ideal for managing multiple similar data sources with consistent logic.

```
(parameter1 as type, parameter2 as type) as returnType =>
    let
        // Function body
    in
        result
```

## Parameterized Data Source Functions

### Basic Pattern: Environment-Specific Connections

```m
// fn_GetDatabaseConnection
(ServerName as text, DatabaseName as text, TableName as text) as table =>
    let
        Source = Sql.Database(ServerName, DatabaseName),
        Table = Source{[Schema="dbo",Item=TableName]}[Data]
    in
        Table
```

Usage:
```m
// Development
DevSales = fn_GetDatabaseConnection("dev-server", "SalesDB", "FactSales")

// Production
ProdSales = fn_GetDatabaseConnection("prod-server", "SalesDB", "FactSales")
```

## Advanced Parameter Patterns

### Pattern 4: Optional Parameters with Defaults

```m
// fn_GetDataWithOptions
(
    SourceType as text, 
    optional Path as text, 
    optional Schema as text, 
    optional Table as text,
    optional RowLimit as number
) as table =>
    let
        // Set defaults
        ActualPath = if Path = null then "C:\default" else Path,
        ActualSchema = if Schema = null then "dbo" else Schema,
        ActualLimit = if RowLimit = null then 1000 else RowLimit,
        
        // Get data based on source type
        SourceData = 
            if SourceType = "Database" then
                let
                    Source = Sql.Database(ActualPath, "DatabaseName"),
                    Table = Source{[Schema=ActualSchema,Item=Table]}[Data]
                in
                    Table
            else if SourceType = "Excel" then
                Excel.Workbook(File.Contents(ActualPath), null, true)
            else
                error "Unsupported source type: " & SourceType,
        
        // Apply limit if specified
        Limited = if ActualLimit > 0 then Table.FirstN(SourceData, ActualLimit) else SourceData
    in
        Limited
```

## Error Handling in Custom Functions

### Pattern 1: Try-Otherwise Pattern

```m
// fn_SafeFileLoad
(FilePath as text) as table =>
    let
        Result = 
            try
                Excel.Workbook(File.Contents(FilePath), null, true)
            otherwise
                // Return empty table with same structure
                #table(
                    {"Column1", "Column2"}, 
                    {}
                )
    in
        Result
```

### Pattern 2: Detailed Error Information

```m
// fn_GetDataWithErrorHandling
(Server as text, Database as text, Query as text) as table =>
    let
        Attempt = try
            let
                Source = Sql.Database(Server, Database),
                Result = Value.NativeQuery(Source, Query)
            in
                Result
        otherwise
            let
                ErrorDetails = 
                    "Error connecting to " & Server & "/" & Database & 
                    " at " & DateTime.ToText(DateTime.LocalNow()) &
                    " - " & (try Diagnostics.ActivityId() otherwise "No Activity ID"),
                
                // Log to file (conceptual - actual logging requires additional setup)
                LogEntry = 
                    try 
                        // Write to log file
                        Text.Write(ErrorDetails, "C:\logs\pq_errors.txt") 
                    otherwise 
                        null,
                
                // Return error table with details
                ErrorTable = #table(
                    {"Error", "Timestamp", "Server", "Database"},
                    {{ErrorDetails, DateTime.LocalNow(), Server, Database}}
                )
            in
                ErrorTable
    in
        Attempt
```

### Pattern 3: Retry Logic

```m
// fn_GetDataWithRetry
(GetDataFunction as function, MaxRetries as number) as any =>
    let
        RetryLoop = List.Generate(
            () => [Attempt = 1, Result = null, Error = null],
            each [Attempt] <= MaxRetries and [Result] = null,
            each 
                let
                    NextTry = 
                        try 
                            [Result = GetDataFunction(), Error = null]
                        otherwise 
                            [Result = null, Error = Error.Record(
                                "RetryableError", 
                                "Attempt " & Text.From([Attempt]) & " failed",
                                null
                            )]
                in
                    [Attempt = [Attempt] + 1, Result = NextTry[Result], Error = NextTry[Error]],
            each [Result]
        ),
        
        // Get first successful result or last error
        Results = List.RemoveNulls(RetryLoop),
        FinalResult = 
            if List.Count(Results) > 0 then
                Results{0}
            else
                error "All retry attempts failed"
    in
        FinalResult

// Usage
fn_GetDataWithRetry(
    () => fn_GetDatabaseConnection("server", "db", "table"),
    3
)
```

### Pattern 4: Graceful Degradation

```m
// fn_GetDataWithFallback
(PrimarySource as function, FallbackSource as function) as table =>
    let
        PrimaryAttempt = try PrimarySource() otherwise null,
        
        Result = 
            if PrimaryAttempt <> null then
                PrimaryAttempt
            else
                let
                    FallbackAttempt = try FallbackSource() otherwise null,
                    
                    Final = 
                        if FallbackAttempt <> null then
                            FallbackAttempt
                        else
                            #table({"Status"}, {{"All data sources unavailable"}})
                in
                    Final
    in
        Result
```

## Null Propagation Strategies

### Pattern 1: Null Handling with Defaults

```m
// fn_TransformWithNulls
(InputTable as table, ColumnName as text) as table =>
    let
        // Replace nulls before processing
        WithDefaults = Table.TransformColumns(
            InputTable,
            {
                {
                    ColumnName, 
                    each if _ = null then 0 else _, 
                    type number
                }
            }
        ),
        
        // Continue with transformations
        Result = Table.AddColumn(
            WithDefaults,
            "Processed",
            each [ColumnName] * 1.1
        )
    in
        Result
```

### Pattern 2: Conditional Null Propagation

```m
// fn_MergeWithNullHandling
(LeftTable as table, RightTable as table, KeyColumns as list) as table =>
    let
        // Left outer join preserving all left rows
        Merged = Table.Join(
            LeftTable, 
            KeyColumns, 
            RightTable, 
            KeyColumns, 
            JoinKind.LeftOuter
        ),
        
        // Handle nulls in right table columns
        RightColumns = Table.ColumnNames(RightTable),
        ColumnsToFix = List.RemoveItems(RightColumns, KeyColumns),
        
        WithDefaults = List.Accumulate(
            ColumnsToFix,
            Merged,
            (state, column) =>
                let
                    // Replace nulls with appropriate default
                    SampleValue = try List.FirstN(List.RemoveNulls(Table.Column(RightTable, column)), 1) otherwise null,
                    Default = 
                        if SampleValue = null then
                            null
                        else if Value.Is(SampleValue, type number) then
                            0
                        else if Value.Is(SampleValue, type text) then
                            ""
                        else if Value.Is(SampleValue, type date) then
                            null
                        else
                            null,
                    
                    NewColumn = Table.ReplaceValue(
                        state,
                        null,
                        Default,
                        Replacer.ReplaceValue,
                        {column}
                    )
                in
                    NewColumn
        )
    in
        WithDefaults
```

### Pattern 3: Propagating Nulls with Warnings

```m
// fn_AnalyzeNulls
(InputTable as table, ColumnsToCheck as list) as table =>
    let
        // Calculate null counts
        NullAnalysis = List.Accumulate(
            ColumnsToCheck,
            InputTable,
            (state, column) =>
                let
                    NullCount = List.Count(List.Select(Table.Column(state, column), each _ = null)),
                    TotalRows = Table.RowCount(state),
                    NullPercent = if TotalRows > 0 then NullCount / TotalRows else 0,
                    
                    // Add warning column
                    WithWarning = Table.AddColumn(
                        state,
                        column & "_Warning",
                        each if NullPercent > 0.05 then "Warning: " & Text.From(NullCount) & " nulls" else "OK"
                    )
                in
                    WithWarning
        )
    in
        NullAnalysis
```

## Real-World Example: Complete Parameterized Function

```m
// fn_GetSalesData - Complete production function
(
    Environment as text,
    StartDate as date,
    optional EndDate as date,
    optional Region as text,
    optional IncludeNulls as logical
) as table =>
    let
        // Default values
        ActualEndDate = if EndDate = null then Date.From(DateTime.LocalNow()) else EndDate,
        ActualIncludeNulls = if IncludeNulls = null then false else IncludeNulls,
        
        // Environment configuration
        Config = 
            if Environment = "Prod" then
                [
                    Server = "prod-sql-01",
                    Database = "SalesDB",
                    QueryTimeout = 120,
                    RetryCount = 3
                ]
            else
                [
                    Server = "dev-sql-01", 
                    Database = "SalesDev",
                    QueryTimeout = 60,
                    RetryCount = 1
                ],
        
        // Build query with parameters
        BaseQuery = 
            "SELECT * FROM FactSales " &
            "WHERE OrderDate >= '" & Date.ToText(StartDate) & "' " &
            "AND OrderDate <= '" & Date.ToText(ActualEndDate) & "' ",
            
        RegionFilter = 
            if Region <> null and Region <> "" then
                "AND Region = '" & Region & "' "
            else
                "",
                
        NullFilter = 
            if ActualIncludeNulls then
                ""  // Include all rows
            else
                "AND ProductID IS NOT NULL ",
                
        FinalQuery = BaseQuery & RegionFilter & NullFilter,
        
        // Execute with retry and error handling
        GetData = () =>
            let
                Source = Sql.Database(Config[Server], Config[Database], 
                    [CommandTimeout = Config[QueryTimeout]]),
                Result = Value.NativeQuery(Source, FinalQuery)
            in
                Result,
        
        ResultWithRetry = 
            try
                List.Last(
                    List.Generate(
                        () => [Attempt = 1, Data = null],
                        each [Attempt] <= Config[RetryCount] and [Data] = null,
                        each 
                            let
                                NextTry = 
                                    try 
                                        [Data = GetData(), Error = null]
                                    otherwise 
                                        [Data = null, Error = "Attempt failed"]
                            in
                                [Attempt = [Attempt] + 1, Data = NextTry[Data]],
                        each [Data]
                    )
                )
            otherwise
                // Return empty table with error column
                #table(
                    {"Error", "Environment", "Query"},
                    {{"All retry attempts failed", Environment, FinalQuery}}
                ),
        
        // Post-processing - handle nulls in key columns
        FinalTable = 
            if Table.HasColumns(ResultWithRetry, {"Error"}) then
                ResultWithRetry
            else
                let
                    // Replace nulls in numeric columns with 0
                    NumericColumns = Table.SelectColumnsOfType(ResultWithRetry, type number),
                    
                    WithDefaults = List.Accumulate(
                        Table.ColumnNames(NumericColumns),
                        ResultWithRetry,
                        (state, col) =>
                            Table.ReplaceValue(state, null, 0, Replacer.ReplaceValue, {col})
                    )
                in
                    WithDefaults
    in
        FinalTable
```

## Best Practices Summary

### Function Design
✓ **Document parameters** - Include types and descriptions
✓ **Provide defaults** - Use optional parameters with sensible defaults
✓ **Single responsibility** - Each function does one thing well
✓ **Return consistent types** - Same structure regardless of success/failure

### Error Handling
✓ **Use try-otherwise** - Gracefully handle expected errors
✓ **Provide context** - Include meaningful error messages
✓ **Log failures** - Capture details for debugging
✓ **Fail fast vs degrade** - Choose based on criticality

### Null Propagation
✓ **Decide strategy upfront** - Remove, replace, or propagate
✓ **Document null behavior** - Make it explicit in function docs
✓ **Check early** - Validate inputs before processing
✓ **Use appropriate defaults** - 0 for numbers, "" for text, null for dates

### Performance
✓ **Minimize external calls** - Batch where possible
✓ **Use lazy evaluation** - Let Power Query optimize
✓ **Cache configurations** - Avoid repeated lookups
✓ **Test with realistic data** - Validate memory usage

## Common Pitfalls to Avoid

✗ **Ignoring nulls** - Leads to cryptic errors downstream
✗ **Too many retries** - Can mask real problems
✗ **Generic error messages** - Hard to debug
✗ **Side effects** - Functions should be deterministic
✗ **Hard-coded paths** - Use parameters or config
✗ **Missing type declarations** - Can cause performance issues

## Summary

**Key Points for Custom M Functions:**

1. **Parameterize everything** - Environment, paths, credentials
2. **Handle errors explicitly** - Use try-otherwise patterns
3. **Propagate nulls intentionally** - Don't let them surprise you
4. **Provide context in errors** - Include timestamp, environment, operation
5. **Retry transient failures** - Network issues happen
6. **Test all paths** - Success, failure, nulls, edge cases
7. **Document behavior** - Make it clear how nulls and errors are handled

**The Golden Rule:**
> A well-designed M function hides complexity while providing clear, predictable behavior for all inputs—including the unexpected ones.
