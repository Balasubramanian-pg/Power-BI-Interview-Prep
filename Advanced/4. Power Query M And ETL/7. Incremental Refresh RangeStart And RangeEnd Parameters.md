# Incremental Refresh: RangeStart and RangeEnd Parameters in Power Query

## Overview

Incremental refresh in Power BI allows large datasets to refresh efficiently by only processing new or changed data rather than the entire dataset. The `RangeStart` and `RangeEnd` parameters are special query parameters that define the date/time window for each incremental refresh partition.

```
Parameter Definition → Filter Query → Partitioning → Refresh Schedule
       ↓                      ↓              ↓              ↓
  RangeStart/           WHERE Date        Power BI      Only new/
  RangeEnd              BETWEEN           creates        changed
  (datetime)            parameters        partitions     data loads
```

## Understanding RangeStart and RangeEnd

### What They Are

`RangeStart` and `RangeEnd` are **parameters** defined in Power Query that:
- Are automatically managed by Power BI during incremental refresh
- Represent the date/time window for the current partition
- Get different values for each partition during refresh
- Must be of type `datetime` or `date`

### How They Work

```
Full Dataset: 2020-01-01 to 2024-12-31
                    ↓
            Partitioned by year
                    ↓
Partition 1: 2020-01-01 to 2020-12-31 (RangeStart=2020-01-01, RangeEnd=2021-01-01)
Partition 2: 2021-01-01 to 2021-12-31 (RangeStart=2021-01-01, RangeEnd=2022-01-01)
Partition 3: 2022-01-01 to 2022-12-31 (RangeStart=2022-01-01, RangeEnd=2023-01-01)
Partition 4: 2023-01-01 to 2023-12-31 (RangeStart=2023-01-01, RangeEnd=2024-01-01)
Partition 5: 2024-01-01 to 2024-12-31 (RangeStart=2024-01-01, RangeEnd=2025-01-01)
```

## Setting Up Parameters

### Step 1: Create Parameters in Power Query

```m
// Create RangeStart parameter
// Name: RangeStart
// Type: Date/Time/Datetime
// Current Value: 2015-01-01 (or your start date)
// Suggested Values: Any

// Create RangeEnd parameter
// Name: RangeEnd
// Type: Date/Time/Datetime
// Current Value: 2025-01-01 (or future date)
// Suggested Values: Any
```

### Step 2: Apply Filter in Your Query

```m
let
    Source = Sql.Database("server", "database"),
    SalesTable = Source{[Schema="dbo",Item="FactSales"]}[Data],
    
    // Filter using RangeStart and RangeEnd
    FilteredRows = Table.SelectRows(
        SalesTable,
        each [OrderDate] >= RangeStart and [OrderDate] < RangeEnd
    )
in
    FilteredRows
```

### Important Filter Logic

The standard pattern is **inclusive of start, exclusive of end**:

```m
// CORRECT: >= RangeStart, < RangeEnd
each [DateColumn] >= RangeStart and [DateColumn] < RangeEnd

// WRONG: Using <= RangeEnd can cause duplicate data
each [DateColumn] >= RangeStart and [DateColumn] <= RangeEnd
```

This ensures partitions don't overlap:
- Partition 1: Jan 1 00:00 to Feb 1 00:00 (includes all of January)
- Partition 2: Feb 1 00:00 to Mar 1 00:00 (includes all of February)

## Database-Specific Implementations

### SQL Server / Azure SQL

```m
let
    Source = Sql.Database("server", "database"),
    
    // Using native query with parameters
    Query = "
        SELECT * FROM FactSales 
        WHERE OrderDate >= @RangeStart 
        AND OrderDate < @RangeEnd
    ",
    
    // Value.NativeQuery with parameters
    Result = Value.NativeQuery(
        Source,
        Query,
        [
            RangeStart = RangeStart,
            RangeEnd = RangeEnd
        ],
        [EnableFolding = true]
    )
in
    Result
```

### Snowflake

```m
let
    Source = Snowflake.Databases("account", "warehouse"),
    Database = Source{[Name="Database"]}[Data],
    Schema = Database{[Name="Schema"]}[Data],
    Table = Schema{[Name="FactSales"]}[Data],
    
    // Filter with parameters
    Filtered = Table.SelectRows(
        Table,
        each [OrderDate] >= RangeStart and [OrderDate] < RangeEnd
    )
in
    Filtered
```

### OData / REST APIs

```m
let
    BaseUrl = "https://api.example.com/sales",
    
    // Build URL with date filters
    QueryString = Uri.BuildQueryString([
        $filter = "OrderDate ge " & DateTime.ToText(RangeStart, "yyyy-MM-dd") & 
                  " and OrderDate lt " & DateTime.ToText(RangeEnd, "yyyy-MM-dd"),
        $orderby = "OrderDate"
    ]),
    
    FullUrl = BaseUrl & "?" & QueryString,
    
    Source = OData.Feed(FullUrl)
in
    Source
```

## Configuring Incremental Refresh in Power BI

### Step 1: Enable Incremental Refresh

1. In Power BI Desktop, go to **Table tools** > **Incremental refresh**
2. Toggle **Incremental refresh** to On

### Step 2: Define Refresh Settings

| Setting | Description | Example |
|---------|-------------|---------|
| **Archive data starting** | How far back to keep data | `5 years` |
| **Incrementally refresh data starting** | Refresh window | `30 days` |
| **Detect data changes** | Column to check for modifications | `LastModifiedDate` |
| **Only refresh complete periods** | Wait for period to end | Check for daily |

### Step 3: Choose Table for Incremental Refresh

Select the table that contains your date-filtered query and apply the settings.

## Advanced Patterns

### Pattern 1: Handling Multiple Date Columns

```m
let
    Source = Sql.Database("server", "database"),
    SalesTable = Source{[Schema="dbo",Item="FactSales"]}[Data],
    
    // Filter on both OrderDate and ShipDate
    Filtered = Table.SelectRows(
        SalesTable,
        each 
            ([OrderDate] >= RangeStart and [OrderDate] < RangeEnd) or
            ([ShipDate] >= RangeStart and [ShipDate] < RangeEnd)
    )
in
    Filtered
```

### Pattern 2: Dynamic Range Based on Today

```m
let
    // Override parameters for initial load
    EffectiveStart = if RangeStart = null then #date(2015,1,1) else RangeStart,
    EffectiveEnd = if RangeEnd = null then Date.AddDays(DateTime.Date(DateTime.LocalNow()), 1) else RangeEnd,
    
    Source = Sql.Database("server", "database"),
    SalesTable = Source{[Schema="dbo",Item="FactSales"]}[Data],
    
    Filtered = Table.SelectRows(
        SalesTable,
        each [OrderDate] >= EffectiveStart and [OrderDate] < EffectiveEnd
    )
in
    Filtered
```

### Pattern 3: Change Detection Optimization

For tables with modification tracking:

```m
let
    Source = Sql.Database("server", "database"),
    
    // Use LastModifiedDate for change detection
    Query = "
        SELECT * FROM FactSales 
        WHERE OrderDate >= @RangeStart 
        AND OrderDate < @RangeEnd
        AND LastModifiedDate >= @LastRefreshTime
    ",
    
    // Get last refresh time from system
    LastRefreshTime = try #"LastRefreshParameter" otherwise #datetime(1900,1,1,0,0,0),
    
    Result = Value.NativeQuery(
        Source,
        Query,
        [
            RangeStart = RangeStart,
            RangeEnd = RangeEnd,
            LastRefreshTime = LastRefreshTime
        ]
    )
in
    Result
```

### Pattern 4: Partitioned Table with Multiple Sources

```m
let
    // Function to get data for a specific period
    GetDataForPeriod = (start as datetime, end as datetime) as table =>
        let
            Source = Sql.Database("server", "database"),
            Query = "
                SELECT * FROM FactSales 
                WHERE OrderDate >= '" & DateTime.ToText(start, "yyyy-MM-dd") & "'
                AND OrderDate < '" & DateTime.ToText(end, "yyyy-MM-dd") & "'
            ",
            Result = Value.NativeQuery(Source, Query)
        in
            Result,
    
    // Get data for current partition
    CurrentData = GetDataForPeriod(RangeStart, RangeEnd)
in
    CurrentData
```

### Pattern 5: Handling Time Zones

```m
let
    // Convert UTC to local if needed
    ConvertToUTC = (localDateTime as datetime) as datetime =>
        // Assuming source is UTC, adjust if needed

    SourceStart = ConvertToUTC(RangeStart),
```

## Common Issues and Solutions

### Issue 1: Duplicate Data

**Problem**: Overlapping partitions cause duplicate rows.

**Solution**: Ensure filter uses `>= Start AND < End` pattern.

### Issue 2: Missing Data at Boundaries

**Problem**: Data at partition boundaries missing.

**Solution**: Verify time component handling:

```m
// If date only (no time), adjust end date
let
    // For date-only columns, use next day for end
    Filtered = Table.SelectRows(
        SalesTable,
        each [OrderDate] >= RangeStart and [OrderDate] < Date.AddDays(RangeEnd, 1)
    )
in
    Filtered
```

### Issue 3: Slow Initial Load

**Problem**: First refresh processing all historical data is slow.

**Solution**: 
- Start with smaller historical window
- Use data compression
- Optimize source query with indexes

### Issue 4: Query Folding Broken

**Problem**: Filter not folding to source.

**Solution**: Ensure filter expression supports folding:

```m
// GOOD - Folds to SQL
Table.SelectRows(Source, each [Date] >= RangeStart and [Date] < RangeEnd)

// BAD - May not fold
Table.SelectRows(Source, each Date.From([Date]) >= Date.From(RangeStart))
```

## Performance Optimization

### Index Strategy

Ensure your source table has appropriate indexes:

```sql
-- SQL Server index for incremental refresh
CREATE INDEX IX_FactSales_OrderDate 
ON FactSales(OrderDate) 
INCLUDE (/* other frequently queried columns */);

-- Include LastModifiedDate for change detection
CREATE INDEX IX_FactSales_LastModified 
ON FactSales(LastModifiedDate) 
INCLUDE (OrderDate);
```

### Partition Strategy Recommendations

| Data Volume | Historical Range | Incremental Window | Partition Size |
|-------------|------------------|-------------------|----------------|
| < 10M rows | Full history | 30 days | Yearly |
| 10M-100M rows | 3-5 years | 14 days | Monthly |
| 100M-1B rows | 1-2 years | 7 days | Monthly |
| > 1B rows | 1 year | 1 day | Daily |

## Best Practices Summary

### Do's
✓ **Use inclusive/exclusive pattern** - `>= Start AND < End`
✓ **Match data types** - Parameter type must match column type
✓ **Test with manual ranges** - Verify filtering works
✓ **Optimize source indexes** - Index on date column
✓ **Monitor refresh times** - Track partition performance
✓ **Plan partition strategy** - Balance size and count
✓ **Document date boundaries** - Note which dates are included

### Don'ts
✗ **Don't use <= End** - Causes overlapping partitions
✗ **Don't ignore time component** - Date-only vs datetime matters
✗ **Don't skip change detection** - For large tables with updates
✗ **Don't make too many partitions** - Overhead of many small partitions
✗ **Don't forget to test** - Verify with production-like data

## Summary

**Key Points for Incremental Refresh Parameters:**

| Aspect | Recommendation |
|--------|----------------|
| **Parameter Names** | `RangeStart`, `RangeEnd` (case-sensitive) |
| **Filter Logic** | `>= RangeStart AND < RangeEnd` |
| **Data Types** | Must match column being filtered |
| **Partition Strategy** | Based on data volume and query patterns |
| **Change Detection** | Use LastModified column for updates |
| **Testing** | Manual parameter override before publishing |
| **Monitoring** | Track refresh times per partition |

**The Golden Rule:**
> `RangeStart` and `RangeEnd` define the boundaries of each partition. Always use inclusive start and exclusive end to prevent overlap. The filter must fold to the source for incremental refresh to work efficiently.
