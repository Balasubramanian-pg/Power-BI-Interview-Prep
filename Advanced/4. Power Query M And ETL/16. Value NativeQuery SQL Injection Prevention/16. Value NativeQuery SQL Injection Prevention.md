# Value.NativeQuery SQL Injection Prevention in Power Query

## Overview

`Value.NativeQuery` is a powerful M function that executes native queries against a data source, but it introduces significant security risks if not used properly. SQL injection attacks can occur when queries are constructed using string concatenation with user-supplied input.

```
User Input → String Concatenation → Malicious SQL → Data Breach/Modification
     ↓                ↓                    ↓
  Parameter       WHERE clause =      '; DROP TABLE Users; --
  "Mike"          'Mike'               Becomes part of query
```

## The SQL Injection Risk

### How Injection Happens

```m
// VULNERABLE - NEVER DO THIS
let
    UserInput = "Mike",  // Could be from parameter, slicer, or external source
    DangerousQuery = 
        "SELECT * FROM Employees WHERE FirstName = '" & UserInput & "'",
    Result = Value.NativeQuery(Source, DangerousQuery)
in
    Result

// If UserInput contains: "'; DROP TABLE Employees; --"
// Resulting query: SELECT * FROM Employees WHERE FirstName = ''; DROP TABLE Employees; --'
// Database executes: SELECT, then DROP TABLE
```

### Why String Concatenation Is Dangerous

| Input | Concatenated Query | Result |
|-------|-------------------|--------|
| `Mike` | `WHERE FirstName = 'Mike'` | Safe |
| `O'Malley` | `WHERE FirstName = 'O'Malley'` | Syntax error (but not malicious) |
| `'; DROP TABLE Users; --` | `WHERE FirstName = ''; DROP TABLE Users; --'` | **Drops Users table** |
| `' OR '1'='1` | `WHERE FirstName = '' OR '1'='1'` | **Returns all rows** |
| `' UNION SELECT username, password FROM Admins--` | `WHERE FirstName = '' UNION SELECT username, password FROM Admins--'` | **Exfiltrates sensitive data** |

## Safe Parameterization with Value.NativeQuery

### Pattern 1: Positional Parameters (List)

```m
let
    Source = Sql.Database("server", "database"),
    
    // SAFE: Using positional parameters with ?
    SafeQuery = Value.NativeQuery(
        Source,
        "SELECT * FROM Employees WHERE FirstName = ? AND Department = ?",
        {"Mike", "Sales"},  // List of parameters in order
        [EnableFolding = true]
    )
in
    SafeQuery
```

**How it works:**
- Parameters are sent separately from the query
- Database engine treats them as data, not executable code
- `?` placeholders are replaced with parameter values safely

### Pattern 2: Named Parameters (Record)

```m
let
    Source = Sql.Database("server", "database"),
    
    // SAFE: Using named parameters with @prefix
    SafeQuery = Value.NativeQuery(
        Source,
        "SELECT * FROM Employees WHERE FirstName = @FirstName AND Department = @Dept",
        [
            FirstName = "Mike",
            Dept = "Sales"
        ],
        [EnableFolding = true]
    )
in
    SafeQuery
```

**Note:** Parameter prefix (`@`, `:`, `?`) depends on the database provider:
- **SQL Server**: `@parameter`
- **Oracle**: `:parameter` or `?` (positional) 
- **PostgreSQL**: `@parameter` or `$1`, `$2`
- **ODBC**: `?` (positional)

### Pattern 3: Mixed Parameters

```m
let
    Source = Sql.Database("server", "database"),
    
    // Multiple parameters of different types
    DynamicReport = Value.NativeQuery(
        Source,
        "SELECT ProductID, Name, ListPrice 
         FROM Products 
         WHERE Category = @Category 
           AND ListPrice BETWEEN @MinPrice AND @MaxPrice
           AND Name LIKE @SearchPattern",
        [
            Category = "Electronics",
            MinPrice = 100,
            MaxPrice = 1000,
            SearchPattern = "%Laptop%"
        ],
        [EnableFolding = true]
    )
in
    DynamicReport
```

## Provider-Specific Parameter Syntax

### SQL Server

```m
// SQL Server uses @named parameters
Value.NativeQuery(
    Source,
    "SELECT * FROM Orders WHERE OrderDate >= @StartDate AND CustomerID = @CustomerID",
    [
        StartDate = #date(2024, 1, 1),
        CustomerID = 12345
    ]
)
```

### Oracle 

```m
// Oracle supports both named and positional
// Named parameters with colon
Value.NativeQuery(
    Source,
    "SELECT * FROM employees WHERE name = :name AND id = :id",
    [name = "Mike", id = 817]  // Record for named
)

// Positional parameters with ?
Value.NativeQuery(
    Source,
    "SELECT * FROM employees WHERE name LIKE ? AND id = ?",
    {"Mike%", 817}  // List for positional
)
```

### ODBC / Generic

```m
// ODBC uses ? positional parameters
Value.NativeQuery(
    Source,
    "SELECT * FROM Customers WHERE Country = ? AND City = ?",
    {"USA", "Seattle"}
)
```

## Common Anti-Patterns to Avoid

### Anti-Pattern 1: Dynamic Table/Column Names

Parameters cannot be used for database objects (tables, columns, schemas):

```m
// THIS STILL REQUIRES CONCATENATION - but validate input!
DynamicTableQuery = 
let
    // User input for table name - DANGEROUS
    TableName = "Employees",  // Could be "Employees; DROP TABLE..."
    
    // Must concatenate for table name
    Query = "SELECT * FROM " & TableName & " WHERE Department = @Dept",
    
    // Parameters only safe for values
    Result = Value.NativeQuery(
        Source,
        Query,
        [Dept = "Sales"],
        [EnableFolding = true]
    )
in
    Result
```

**Mitigation for dynamic objects:**
- **Whitelist allowed values** - Compare against known safe list
- **Use schema queries** - Query system tables to validate existence
- **Escape identifiers** - Use database-specific quoting functions

```m
// Safer approach with whitelist
SafeTableQuery = 
let
    AllowedTables = {"Employees", "Customers", "Products"},
    RequestedTable = "Employees",  // From user
    
    ValidatedTable = 
        if List.Contains(AllowedTables, RequestedTable) then
            RequestedTable
        else
            error "Invalid table name",
    
    Query = "SELECT * FROM " & ValidatedTable & " WHERE Department = @Dept",
    Result = Value.NativeQuery(Source, Query, [Dept = "Sales"])
in
    Result
```

### Anti-Pattern 2: Mixing Concatenation with Parameters

```m
// BAD - Partially parameterized
let
    UserFilter = "Active = 1",  // User-supplied WHERE clause
    Query = "SELECT * FROM Users WHERE " & UserFilter & " AND Department = @Dept",
    Result = Value.NativeQuery(Source, Query, [Dept = "Sales"])
in
    Result
// If UserFilter = "1=1; DROP TABLE Users; --", injection still succeeds
```

### Anti-Pattern 3: Disabling Security Features

```m
// NEVER disable these safeguards
// In Power BI Desktop: File > Options > Security
// Keep enabled:
// - [✓] Require user approval for new native database queries 
// - [✓] Use Data Privacy Firewall
// - [✓] Validate certificate revocation
```

## Security Features in Power BI

### Native Database Query Approval 

Power BI includes a critical security setting that requires user approval before running native queries:

```
File > Options > Security > Native Database Queries
[✓] Require user approval for new native database queries (Recommended)
```

**What it protects against:**
- TMDL scripts containing malicious SQL 
- Unintended query execution when opening reports
- SQL injection attempts embedded in .pbix files

**Best Practice:** Keep this setting enabled in production environments.

### Data Privacy Firewall 

The Data Privacy Firewall prevents data leakage between sources:

- Blocks passing values from one source to another without privacy level checks
- Prevents using data from a private source as parameters to query a public source
- Disabling ("Fast Combine") is **not recommended** for security 

## Complete Security Pattern

### Production-Ready Parameterized Query

```m
let
    // Step 1: Source connection (configured with least privilege)
    Source = Sql.Database(
        "production-server",
        "SalesDB",
        [CreateNavigationProperties = false]
    ),
    
    // Step 2: Validate inputs (if from external source)
    ValidateInput = (input as text, allowedPattern as text) as text =>
        let
            IsValid = Text.Contains(allowedPattern, input) or
                      Text.Length(input) <= 50  // Reasonable length limit
        in
            if IsValid then input else error "Invalid input",
    
    // Step 3: Define parameters with validation
    UserCategory = ValidateInput(
        "Electronics",  // From slicer/parameter
        "Electronics,Clothing,Home"
    ),
    MinPrice = 100,
    MaxPrice = 1000,
    
    // Step 4: Execute parameterized query
    Result = Value.NativeQuery(
        Source,
        "SELECT ProductID, Name, ListPrice, Category
         FROM Products
         WHERE Category = @Category
           AND ListPrice BETWEEN @MinPrice AND @MaxPrice
         ORDER BY ListPrice",
        [
            Category = UserCategory,
            MinPrice = MinPrice,
            MaxPrice = MaxPrice
        ],
        [
            EnableFolding = true,      // Maintains query folding 
            PreserveTypes = true        // Keeps data types 
        ]
    ),
    
    // Step 5: Additional transformations (still foldable)
    FilteredResult = Table.SelectRows(
        Result,
        each [ListPrice] > 500  // This will fold to source 
    )
in
    FilteredResult
```

## Audit and Monitoring

### Checking for Vulnerable Code

```m
// Diagnostic query to review all Value.NativeQuery calls
let
    // This would need to be run in a development environment
    // to review existing queries for security issues
    
    // Pattern to check for string concatenation in queries
    VulnerablePatterns = {
        "&",
        "Text.Combine",
        "& """,
        "' & "
    }
    
    // Manual review required - no automated scanning in Power BI
in
    "Review all Value.NativeQuery calls for string concatenation"
```

### Query Diagnostics 

Use Query Diagnostics to monitor actual queries sent to the database:

1. **Enable diagnostics**: Tools > Options > Diagnostics > Enable query diagnostics
2. **Run your query**
3. **Review logs** for unexpected SQL patterns
4. **Verify** that parameters are properly parameterized (not inlined)

## Best Practices Summary

### Do's
✓ **Always use parameters** - Never concatenate user input into query strings
✓ **Validate all inputs** - Even with parameters, validate length and format
✓ **Use least privilege accounts** - Read-only accounts for query operations
✓ **Keep security settings enabled** - Native query approval, privacy firewall
✓ **Review all native queries** - Especially when importing from untrusted sources 
✓ **Test with malicious inputs** - Verify parameterization works

### Don'ts
✗ **Never use string concatenation** - Even for "simple" inputs
✗ **Don't disable native query approval** - Critical security safeguard
✗ **Avoid dynamic object names** - If necessary, use strict whitelists
✗ **Don't ignore privacy levels** - Fast combine bypasses security 
✗ **Never embed credentials** - Use parameters or data source settings

## Security Checklist

```
□ All Value.NativeQuery calls use parameters (not concatenation)
□ Database accounts have least privilege (read-only where possible)
□ "Require user approval for native queries" enabled
□ Privacy levels configured appropriately
□ Input validation implemented for all external parameters
□ Query diagnostics reviewed periodically
□ TMDL scripts inspected before execution 
□ No disabled certificate revocation checks
□ Data gateway connections secured
□ Regular security reviews of M code
```

## Summary

**Key Points for SQL Injection Prevention with Value.NativeQuery:**

1. **Parameters are mandatory** - Use `?` (list) or `@name` (record) placeholders
2. **Never concatenate** - String concatenation defeats parameterization
3. **Database objects require whitelists** - Use allowed lists for dynamic tables/columns
4. **Security settings protect you** - Keep native query approval enabled
5. **Validate all inputs** - Even with parameters, check length and format
6. **Review untrusted code** - TMDL scripts and shared code may contain injection 
7. **Monitor actual queries** - Use Query Diagnostics to verify behavior

**The Golden Rule:**
> Parameters are for data, not code. If you're concatenating anything that came from outside your control, you're doing it wrong. `Value.NativeQuery` with proper parameters is safe; string concatenation with user input is never safe.
