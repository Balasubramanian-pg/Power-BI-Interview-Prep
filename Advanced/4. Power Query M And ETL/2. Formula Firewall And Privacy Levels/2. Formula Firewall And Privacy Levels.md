# Formula Firewall and Privacy Levels in Power Query M

## Overview

The Formula Firewall is Power Query's security mechanism that protects data privacy by isolating data sources based on their configured privacy levels. It prevents unintended data leakage between sources with different sensitivity classifications.

```
Source A → Privacy Level → Firewall → Privacy Level → Source B
(Public)        ↓                            ↓        (Private)
            Can data flow?              Can data flow?
```

## Privacy Levels Explained

Each data source can be assigned one of three privacy levels:

| Level | Description | Example Sources |
|-------|-------------|-----------------|
| **Public** | Data can be freely shared with any other source | Public APIs, open government data |
| **Organizational** | Data limited to a trusted organizational boundary | Internal databases, SharePoint, corporate file shares |
| **Private** | Most sensitive data; cannot be combined with other sources | PII data, HR records, financial systems |

### How Privacy Levels Restrict Data Flow

The firewall enforces these rules:

```
Source A Level    Source B Level    Can Data Flow?
-----------------------------------------------
Public            Public            YES
Public            Organizational    YES
Public            Private           NO
Organizational    Organizational    YES  
Organizational    Private           NO
Private           Private           NO
```

**Key rule:** Data from a more restricted source cannot flow to a less restricted source.

## The Formula Firewall in Action

### When the Firewall Triggers

The firewall activates when a query combines data from multiple sources and Power Query cannot guarantee privacy boundaries are respected .

```m
// THIS TRIGGERS THE FIREWALL
let
    // Source A - Configured as Public
    PublicData = Csv.Document(File.Contents("C:\public\data.csv")),
    
    // Source B - Configured as Private  
    PrivateData = Sql.Database("server", "HR_Database"),
    
    // Combining sources - Firewall evaluates
    Combined = Table.Combine({PublicData, PrivateData})
in
    Combined
```

### The Error Message

When the firewall blocks a query, you'll see:

> **"Formula.Firewall: Query 'Query1' (step 'Combined') references other queries or steps, so it may not directly access a data source. Please rebuild this data combination."**

Or for direct source mixing:

> **"Formula.Firewall: The query references multiple data sources and may produce a result where data from one source is combined with data from another source. To combine data, create a new query, then reference both queries."**

## Solving Firewall Issues

### Solution 1: Merge in a Separate Query

The most common fix is separating the combination into its own query :

```m
// Query1 - Get Public Data
let
    Source = Csv.Document(File.Contents("C:\public\data.csv")),
    #"Promoted Headers" = Table.PromoteHeaders(Source)
in
    #"Promoted Headers"

// Query2 - Get Private Data
let
    Source = Sql.Database("server", "HR_Database"),
    Employees = Source{[Schema="dbo",Item="Employees"]}[Data]
in
    Employees

// Query3 - Combine (references Query1 and Query2)
let
    PublicData = Query1,
    PrivateData = Query2,
    Combined = Table.Combine({PublicData, PrivateData})
in
    Combined
```

### Solution 2: Buffer to Break Evaluation

`Table.Buffer` forces immediate evaluation, which can resolve some firewall issues :

```m
let
    Source1 = Sql.Database("server", "Database1"),
    Data1 = Source1{[Schema="dbo",Item="Table1"]}[Data],
    
    Source2 = Sql.Database("server", "Database2"),
    Data2 = Source2{[Schema="dbo",Item="Table2"]}[Data],
    
    // Buffer forces evaluation of Data1
    Buffered1 = Table.Buffer(Data1),
    
    // Now combine
    Combined = Table.Join(Buffered1, "ID", Data2, "ID")
in
    Combined
```

### Solution 3: Adjust Privacy Levels

If data is appropriately classified, adjust levels (with caution) :

1. **In Power BI Desktop**: File > Options > Privacy
2. **Select**: "Ignore the Privacy Levels and potentially improve performance"
3. **Or set each source**: Data source settings > Edit Permissions > Privacy Level

**Warning:** Ignoring privacy levels can expose sensitive data. Only use when you fully control all sources and understand the data flow .

### Solution 4: Use Dataflows

Move data combination to Power BI Dataflows, which have different privacy handling:

1. Load individual sources into dataflow entities
2. Combine in dataflow transformations
3. Reference combined entity in dataset

## Native Query Folding and the Firewall

### When Folding Prevents Firewall Errors

If all data sources fold to the same database, the firewall doesn't trigger :

```m
// These both fold to SQL Server - Firewall not triggered
let
    Source = Sql.Database("server", "SalesDB"),
    Customers = Source{[Schema="dbo",Item="Customers"]}[Data],
    Orders = Source{[Schema="dbo",Item="Orders"]}[Data],
    Joined = Table.Join(Customers, "CustomerID", Orders, "CustomerID")
in
    Joined
```

Since both tables come from the same physical source with consistent privacy, Power Query can push the join to the database without moving data between privacy zones .

### Breaking Folding Triggers Firewall

When you break query folding, data materializes in Power Query, potentially triggering the firewall :

```m
let
    Source = Sql.Database("server", "SalesDB"),
    Customers = Source{[Schema="dbo",Item="Customers"]}[Data],
    
    // Buffer breaks folding - data now in Power Query
    Buffered = Table.Buffer(Customers),
    
    // Adding index - local operation
    WithIndex = Table.AddIndexColumn(Buffered, "RowID", 1, 1)
in
    WithIndex
```

## Advanced Patterns

### Pattern 1: Privacy-Aware Query Design

```m
// BAD - Mixes sources in one query
let
    LocalFile = Excel.Workbook(File.Contents("C:\data.xlsx"), null, true),
    SQLData = Sql.Database("server", "DB"),
    Combined = Table.Join(LocalFile, "ID", SQLData, "ID")
in
    Combined

// GOOD - Separate concerns
// Query1: GetLocalData
let
    Source = Excel.Workbook(File.Contents("C:\data.xlsx"), null, true)
in
    Source

// Query2: GetSQLData
let
    Source = Sql.Database("server", "DB"),
    Data = Source{[Schema="dbo",Item="Table"]}[Data]
in
    Data

// Query3: CombineData (references Query1, Query2)
let
    Local = GetLocalData,
    SQL = GetSQLData,
    Combined = Table.Join(Local, "ID", SQL, "ID")
in
    Combined
```

### Pattern 2: Conditional Privacy Handling

```m
let
    // Determine if sources are the same
    Source1 = Sql.Database("server1", "DB"),
    Source2 = Sql.Database("server2", "DB"),
    
    // Check if same server (simplified)
    SameSource = false,  // Would need actual comparison
    
    Result = if SameSource then
        // Safe to combine directly
        Table.Join(Source1, "ID", Source2, "ID")
    else
        // Need to buffer or separate
        let
            Buffered1 = Table.Buffer(Source1),
            Joined = Table.Join(Buffered1, "ID", Source2, "ID")
        in
            Joined
in
    Result
```

## Performance Implications

### Fast Combine vs. Privacy Enforcement

The "Fast Combine" option (ignoring privacy levels) can improve performance but at security cost :

| Setting | Performance | Security |
|---------|-------------|----------|
| **Privacy levels enforced** | Slower - Firewall evaluates each step | Secure - No data leakage |
| **Fast Combine enabled** | Faster - No firewall checks | Risky - Potential data exposure |

**When Fast Combine might be acceptable:**
- All data sources are from trusted, internal systems
- You've manually verified no sensitive data flow issues
- Development/testing environments with synthetic data

**Never use Fast Combine when:**
- Mixing public APIs with internal data
- Handling PII or regulated data
- Data sources have different owners/trust boundaries

### Firewall Evaluation Overhead

The firewall adds overhead by:
- Tracking data lineage through each step
- Validating privacy boundaries at transformation boundaries
- Potentially blocking or buffering operations

For complex queries with many steps, this can significantly impact performance.

## Best Practices Summary

### Do's
✓ **Separate source queries** - Keep each data source in its own query
✓ **Combine in dedicated query** - Reference source queries in a final combination query 
✓ **Use consistent sources** - Prefer joining data that folds to same source
✓ **Document privacy levels** - Know what level each source has
✓ **Test with real data** - Verify firewall behavior before production

### Don'ts
✗ **Don't ignore privacy warnings** - They indicate potential data exposure
✗ **Don't blindly use Fast Combine** - Only when you fully control all sources
✗ **Don't mix private and public without isolation** - Use separate queries
✗ **Don't assume all sources have same privacy** - Check each source
✗ **Don't disable firewall in production** - Security risk

## Common Scenarios and Solutions

| Scenario | Problem | Solution |
|----------|---------|----------|
| **Combine local Excel with SQL Server** | Firewall triggered | Create separate queries, combine in third |
| **Join two SQL tables from same server** | No issue | Folds to SQL - safe |
| **Join SQL table with SQL table from different server** | Firewall may trigger | Buffer one source or use separate queries |
| **API data with internal database** | Privacy mismatch | Buffer API data or create separate queries |
| **Multiple transformations after combine** | Complex lineage tracking | Buffer after each major transformation |

## Summary

**Key Points for Formula Firewall and Privacy Levels:**

1. **Three privacy levels**: Public, Organizational, Private
2. **Firewall prevents**: Data flow from restricted to less restricted sources
3. **Common fix**: Separate source queries from combination queries
4. **Fast Combine**: Disables privacy checks (use cautiously)
5. **Query folding**: Same-source operations avoid firewall
6. **Buffer helps**: Forces evaluation, can resolve some issues

**The Golden Rule:**
> The Formula Firewall protects your data. Work with it, not against it—design queries with clear separation between data sources, and combine them only in dedicated queries where you control the privacy boundaries.
