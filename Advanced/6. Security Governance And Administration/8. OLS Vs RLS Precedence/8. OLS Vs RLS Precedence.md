# 8. OLS Vs RLS Precedence

## Overview

Understanding the precedence between Object Level Security (OLS) and Row Level Security (RLS) is crucial in Power BI, as it directly affects data access and security. Misunderstanding this concept can lead to unintended data exposure or restrictions. The flow of security application can be visualized as follows:
```
Query → Authentication → Authorization → Data Access → Result
           ↓             ↓            ↓
      OLS Evaluation  RLS Evaluation  Data Filtering
```

## The Security Evaluation Process

### Object Level Security (OLS)

Object Level Security (OLS) is responsible for controlling access to specific objects within a Power BI model, such as tables, measures, or columns. OLS is applied at the object level, determining which users or roles can view or interact with these objects.

**Characteristics:**
- Applied at the object level (tables, measures, columns)
- Controls access to specific objects within a model
- Evaluates user roles and permissions

### Row Level Security (RLS)

Row Level Security (RLS) filters data based on user identity, ensuring that users can only see the data they are authorized to access. RLS is applied at the row level, dynamically filtering data based on the user's role or identity.

**Characteristics:**
- Applied at the row level
- Filters data based on user identity and role
- Dynamically adjusts data visibility

### Security Precedence

Security precedence refers to the order in which OLS and RLS are evaluated. Understanding this precedence is essential to ensure that data access is correctly controlled.

**Characteristics:**
- OLS is evaluated first, controlling access to objects
- RLS is evaluated second, filtering data based on user identity

## The Security Evaluation Mechanism

The security evaluation mechanism in Power BI involves a series of steps, from authentication to data access. The flow of this mechanism can be represented as follows:
```
Authentication → Authorization → OLS Evaluation → RLS Evaluation → Data Access
    ↓             ↓             ↓             ↓
User Identity  User Roles  Object Access  Row-Level Filtering
```

### How Security is Evaluated

When a user queries a Power BI model, the following steps occur:
1. **Authentication**: The user is authenticated, and their identity is verified.
2. **Authorization**: The user's roles and permissions are evaluated, determining which objects they can access.
3. **OLS Evaluation**: OLS rules are applied, controlling access to specific objects within the model.
4. **RLS Evaluation**: RLS rules are applied, filtering data based on the user's identity and role.

## Internal Query Language

Power BI uses DAX (Data Analysis Expressions) as its internal query language. DAX is used to define measures, calculate columns, and create data models.

### What is DAX?

DAX is a collection of functions, operators, and constants that can be used to create calculations and data models in Power BI.

```dax
-- Example DAX query
EVALUATE
SUMMARIZE(
    'Sales',
    'Sales'[Region],
    "Total Sales", SUM('Sales'[Sales Amount])
)
```

### Example: Using DAX to Apply RLS

```dax
-- User-facing DAX query
EVALUATE
FILTER(
    'Sales',
    'Sales'[Region] = USERNAME()
)

-- Generated internal representation
-- This would be translated into a query that filters data based on the user's identity
```

## When OLS and RLS are Efficient

### Scenario 1: Simple Security Model

```dax
-- Example DAX query with simple security model
EVALUATE
FILTER(
    'Sales',
    'Sales'[Region] = USERNAME()
)
```

**Result:**
- Data is filtered based on the user's region
- Only authorized users can access the data
- The security model is easy to maintain and update

### Scenario 2: Complex Security Model

```dax
-- Example DAX query with complex security model
EVALUATE
FILTER(
    'Sales',
    'Sales'[Region] = USERNAME() && 'Sales'[Country] = USERNAME()
)
```

**Process:**
1. The user's region and country are evaluated
2. The data is filtered based on the user's region and country
3. The security model is more complex, but still efficient

## When OLS and RLS Become Problematic

### Scenario 1: Overly Complex Security Model

```dax
-- Example DAX query with overly complex security model
EVALUATE
FILTER(
    'Sales',
    'Sales'[Region] = USERNAME() && 'Sales'[Country] = USERNAME() && 'Sales'[City] = USERNAME()
)
```

**Problem:**
- The security model is too complex, leading to performance issues
- The data is filtered based on too many conditions, causing errors
- The security model is difficult to maintain and update

### Scenario 2: Incorrect Security Configuration

```dax
-- Example DAX query with incorrect security configuration
EVALUATE
FILTER(
    'Sales',
    'Sales'[Region] = "Incorrect Region"
)
```

**Problem:**
- The security configuration is incorrect, leading to unauthorized data access
- The data is not filtered correctly, causing errors
- The security model is not effective

## Optimization Strategies

### Strategy 1: Simplify Security Model

```dax
-- BEFORE: Complex security model
EVALUATE
FILTER(
    'Sales',
    'Sales'[Region] = USERNAME() && 'Sales'[Country] = USERNAME() && 'Sales'[City] = USERNAME()
)

-- AFTER: Simplified security model
EVALUATE
FILTER(
    'Sales',
    'Sales'[Region] = USERNAME() && 'Sales'[Country] = USERNAME()
)
```

**Key Function:** Simplifying the security model improves performance and reduces errors.

### Strategy 2: Use Efficient Data Retrieval

```dax
-- BEFORE: Inefficient data retrieval
EVALUATE
SUMMARIZE(
    'Sales',
    'Sales'[Region],
    "Total Sales", SUM('Sales'[Sales Amount])
)

-- AFTER: Efficient data retrieval
EVALUATE
SUMMARIZE(
    FILTER(
        'Sales',
        'Sales'[Region] = USERNAME()
    ),
    'Sales'[Region],
    "Total Sales", SUM('Sales'[Sales Amount])
)
```

**Key Function:** Using efficient data retrieval improves performance and reduces errors.

## Best Practices Summary

### Do's
✓ **Simplify Security Model** - Improves performance and reduces errors
✓ **Use Efficient Data Retrieval** - Improves performance and reduces errors
✓ **Monitor Performance** - Identifies issues and optimizes performance
✓ **Test Security** - Ensures security rules are correct and effective
✓ **Document Security** - Ensures security rules are well-documented and maintainable
✓ **Review Security Regularly** - Ensures security rules are up-to-date and effective

### Don'ts
✗ **Overcomplicate Security Model** - Leads to performance issues and errors
✗ **Use Inefficient Data Retrieval** - Leads to performance issues and errors
✗ **Ignore Performance** - Leads to slow and unresponsive queries
✗ **Neglect Security Testing** - Leads to security vulnerabilities and data breaches
✗ **Fail to Document Security** - Leads to confusion and maintenance issues
✗ **Ignore Security Reviews** - Leads to outdated and ineffective security rules
