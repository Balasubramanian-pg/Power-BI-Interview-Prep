# 11. External Sharing Risk And Hard Revoke

## Overview
External sharing in Power BI allows users to share reports and dashboards with others outside their organization, but it also introduces security risks if not managed properly. Misunderstanding these risks can lead to unintended data exposure, compromising sensitive information. Effective management of external sharing and the ability to hard revoke access when necessary are crucial for maintaining data security in Power BI. 
```
Query → [Authentication] → [Authorization] → [Data Access] → Result
           ↓             ↓            ↓
      [User Identity]    [Permissions]   [Data Encryption]
```

## The Core Security Components

### Authentication
Authentication in the context of external sharing refers to the process of verifying the identity of users who are attempting to access shared Power BI content. This is crucial for ensuring that only authorized individuals can view or interact with the shared data.

**Characteristics:**
- **Multi-Factor Authentication (MFA)**: Adds an extra layer of security by requiring users to provide a second form of verification, such as a code sent to their phone, in addition to their username and password.
- **Azure Active Directory (AAD) Integration**: Leverages AAD for identity management, allowing for centralized control over user identities and access permissions.
- **Guest Access**: Enables the sharing of Power BI content with users outside the organization, with the ability to control their access levels.

### Authorization
Authorization determines what actions a user can perform on the shared content once they have been authenticated. This involves setting permissions that define the level of access each user has to the shared reports, dashboards, or datasets.

**Characteristics:**
- **Role-Based Access Control (RBAC)**: Assigns users to roles that define their permissions and access levels, simplifying the management of user privileges.
- **Dataset-Level Security**: Allows for the application of security filters and roles directly to datasets, ensuring that users can only see the data they are authorized to access.
- **Report and Dashboard Permissions**: Controls who can view, edit, or manage specific reports and dashboards.

### Data Encryption
Data encryption is the process of converting plaintext data into unreadable ciphertext to protect it from unauthorized access. In the context of external sharing, encryption ensures that even if data is intercepted or accessed by unauthorized parties, it will be indecipherable without the decryption key.

**Characteristics:**
- **At-Rest Encryption**: Encrypts data when it is stored, protecting it from unauthorized access by those who might gain physical access to the storage devices.
- **In-Transit Encryption**: Encrypts data as it is transmitted over networks, protecting it from interception and eavesdropping.
- **Key Management**: Involves the secure generation, distribution, and management of encryption keys to ensure that only authorized parties can decrypt and access the data.

## The Sharing and Revocation Process

### How External Sharing Works
The process of externally sharing Power BI content involves several steps, from preparing the content for sharing to managing access and revoking it when necessary.
```
[Content Preparation] → [Sharing] → [Access Management] → [Revocation] → [Audit]
    ↓            ↓            ↓            ↓
[Data Classification]    [User Invitation]    [Permission Assignment]     [Access Removal]
```

### When Access Needs to Be Revoked
Access revocation is triggered by various events, such as a change in user roles, termination of employment, or a security breach. The steps to revoke access include:
1. **Identify the Need for Revocation**: Determine that access needs to be removed, based on organizational policies or security requirements.
2. **Locate the Shared Content**: Find all Power BI content that has been shared with the user or group whose access is being revoked.
3. **Remove User Permissions**: Update the permissions for the shared content to remove the user's access.
4. **Notify Stakeholders**: Inform relevant parties, such as the user whose access is being revoked and the content owner, about the change in access permissions.

## When External Sharing is Efficient

### Scenario 1: Controlled Guest Access
```dax
-- Example of controlled guest access
// Create a new security group for guests
New-SecurityGroup -Name "GuestAccess"

// Add guests to the security group
Add-AzureADGroupMember -ObjectId (Get-AzureADGroup -Filter "DisplayName eq 'GuestAccess'").ObjectId -Members (Get-AzureADUser -Filter "UserPrincipalName eq 'guest@example.com'")
```

**Result:**
- Guests can access shared content without needing a full organizational account.
- Access is controlled and can be easily revoked.
- Guests are restricted to viewing only the shared content and cannot access other organizational resources.

### Scenario 2: Role-Based Access Control
```dax
-- Example of role-based access control
// Define roles for the workspace
New-Role -Name "Viewer" -Description "Can view reports"
New-Role -Name "Editor" -Description "Can edit reports"

// Assign roles to users
Add-RoleMember -Role "Viewer" -User "user1@example.com"
Add-RoleMember -Role "Editor" -User "user2@example.com"
```

**Process:**
1. Define roles that align with job functions or responsibilities.
2. Assign users to these roles based on their needs.
3. Manage access to shared content through these roles.

## Common External Sharing Patterns

### Pattern 1: Overly Permissive Roles
```dax
-- Problematic code: overly permissive role
// Role has too many permissions
New-Role -Name "Admin" -Description "Can do everything"
```

**Problem:** The role has too many permissions, potentially leading to unauthorized access.

**Better:**
```dax
-- Fixed code: more restrictive role
// Define roles with least privilege
New-Role -Name "Viewer" -Description "Can view reports"
New-Role -Name "Editor" -Description "Can edit reports"
```

### Pattern 2: Unrestricted Guest Access
```dax
-- Problematic code: unrestricted guest access
// Guests have unrestricted access
New-SecurityGroup -Name "GuestAccess" -Description "Unrestricted access for guests"
```

**Problem:** Guests have too much access, potentially leading to security breaches.

**Better:**
```dax
-- Fixed code: restricted guest access
// Restrict guest access to specific content
New-SecurityGroup -Name "GuestAccess" -Description "Restricted access for guests"
Add-AzureADGroupMember -ObjectId (Get-AzureADGroup -Filter "DisplayName eq 'GuestAccess'").ObjectId -Members (Get-AzureADUser -Filter "UserPrincipalName eq 'guest@example.com'")
```

## Optimization Strategies

### Strategy 1: Implement Least Privilege
```dax
-- BEFORE: overly permissive role
New-Role -Name "Admin" -Description "Can do everything"

-- AFTER: more restrictive role
New-Role -Name "Viewer" -Description "Can view reports"
New-Role -Name "Editor" -Description "Can edit reports"
```

Implementing least privilege ensures that users and roles have only the necessary permissions to perform their tasks, reducing the risk of unauthorized access.

### Strategy 2: Use Azure AD Groups for Access Control
```dax
-- BEFORE: individual user assignments
Add-AzureADGroupMember -ObjectId (Get-AzureADGroup -Filter "DisplayName eq 'AccessGroup'").ObjectId -Members (Get-AzureADUser -Filter "UserPrincipalName eq 'user1@example.com'")

-- AFTER: using Azure AD groups
New-SecurityGroup -Name "AccessGroup" -Description "Access for specific users"
Add-AzureADGroupMember -ObjectId (Get-AzureADGroup -Filter "DisplayName eq 'AccessGroup'").ObjectId -Members (Get-AzureADUser -Filter "UserPrincipalName eq 'user1@example.com'")
```

Using Azure AD groups simplifies access control by allowing you to manage permissions at the group level rather than individually.

### Strategy 3: Regularly Review and Update Permissions
```dax
-- BEFORE: outdated permissions
Get-AzureADGroupMember -ObjectId (Get-AzureADGroup -Filter "DisplayName eq 'AccessGroup'").ObjectId

-- AFTER: updated permissions
Remove-AzureADGroupMember -ObjectId (Get-AzureADGroup -Filter "DisplayName eq 'AccessGroup'").ObjectId -Members (Get-AzureADUser -Filter "UserPrincipalName eq 'formeruser@example.com'")
```

Regularly reviewing and updating permissions ensures that access is always aligned with current needs and reduces the risk of unauthorized access due to outdated permissions.
