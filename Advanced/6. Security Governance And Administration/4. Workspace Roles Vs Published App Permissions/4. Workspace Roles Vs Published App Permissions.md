# 4. Workspace Roles Vs Published App Permissions

## Overview

Understanding the distinction between workspace roles and published app permissions is crucial in Power BI, as it directly impacts the security and accessibility of reports and dashboards. Misunderstanding these concepts can lead to unintended data exposure or restricted access, hindering the effectiveness of business intelligence solutions. The flow of permissions from workspace to published app can be visualized as follows:
```
Workspace → [Workspace Roles] → [Published App] → [App Permissions] → End User
           ↓             ↓            ↓
      [Role Assignment]    [Permission Setup]   [User Access]
```


## The Permission Framework

The permission framework in Power BI consists of three major components: workspace roles, published app permissions, and user assignments.

### Workspace Roles

Workspace roles define the level of access users have within a workspace, determining what actions they can perform, such as editing reports or managing the workspace. 
**Characteristics:**
- **Admin**: Full control over the workspace, including managing roles and permissions.
- **Member**: Ability to edit and publish reports, but not manage the workspace or roles.
- **Contributor**: Can edit reports but not publish them or manage the workspace.

### Published App Permissions

Published app permissions control what features and data are accessible to users when they interact with a published app, ensuring that sensitive data is protected and that users can only perform intended actions.
**Characteristics:**
- **Read**: Users can view reports and dashboards but cannot interact with them beyond filtering and slicing.
- **Edit**: Users can modify reports and dashboards, but changes are not saved to the original published app.
- **Build**: Users can create new content within the app, such as new reports or dashboards.

### User Assignments

User assignments tie workspace roles and published app permissions together, determining the actual access level a user has to a published app based on their role in the workspace and the permissions set for the app.
**Characteristics:**
- **Direct Assignment**: Users are directly assigned a role in the workspace or permissions in the published app.
- **Group Membership**: Users inherit roles or permissions through their membership in security groups.


## The Permission Flow

The permission flow from workspace to published app is a critical process that determines the final access level a user has to a published app. This flow can be represented as:
```
Workspace Role → Published App Permission → User Assignment → Final Access Level
    ↓            ↓            ↓            ↓
[Role Definition] [Permission Setup] [User or Group] [Effective Permissions]
```

### How Permissions Flow

When a user attempts to access a published app, Power BI evaluates their workspace role and the published app permissions to determine their final access level. This evaluation considers direct assignments, group memberships, and the specific permissions set for the app.

### When Role Changes Occur

When changes are made to a user's workspace role or to the permissions of a published app, the following steps occur:
1. **Role Update**: The user's new role is updated in the workspace.
2. **Permission Recalculation**: Power BI recalculates the user's effective permissions for the published app based on the new role.
3. **Access Level Update**: The user's access level to the published app is updated to reflect the new permissions.
4. **Notification**: Depending on the change, users may be notified of updates to their access levels or permissions.


## Internal Permission Mechanism

Power BI utilizes an internal permission mechanism to manage access to workspaces and published apps. This mechanism is based on a complex set of rules and permissions that are evaluated in real-time.

### What is the Internal Permission Mechanism?

The internal permission mechanism is a set of algorithms and data structures that Power BI uses to evaluate user permissions and determine access levels. This mechanism considers all relevant factors, including workspace roles, published app permissions, and user assignments.

## When Permissions Are Efficient

### Scenario 1: Well-Defined Roles

```dax
-- Example of well-defined roles
VAR AdminRole = 'Admin'
VAR MemberRole = 'Member'
RETURN
    SWITCH(
        TRUE,
        USERPRINCIPALNAME() = "AdminUser", AdminRole,
        USERPRINCIPALNAME() = "MemberUser", MemberRole
    )
```

**Result:**
- **Clear Access Levels**: Users have well-defined access levels based on their roles.
- **Easy Management**: Roles are easy to manage and update.
- **Improved Security**: Access to sensitive data is restricted to authorized users.

### Scenario 2: Granular Permissions

```dax
-- Example of granular permissions
VAR ReadPermission = 'Read'
VAR EditPermission = 'Edit'
RETURN
    SWITCH(
        TRUE,
        USERPRINCIPALNAME() = "ReadUser", ReadPermission,
        USERPRINCIPALNAME() = "EditUser", EditPermission
    )
```

**Process:**
1. **Define Permissions**: Granular permissions are defined for each user or group.
2. **Assign Permissions**: Permissions are assigned to users or groups based on their roles or needs.


## When Permissions Become Problematic

### Scenario 1: Overly Permissive Roles

```dax
-- Example of overly permissive roles
VAR AdminRole = 'Admin'
RETURN
    IF(
        USERPRINCIPALNAME() = "AnyUser", 
        AdminRole
    )
```

**Problem:**
- **Unintended Access**: Users may have unintended access to sensitive data or features.
- **Security Risks**: Overly permissive roles can pose significant security risks.
- **Difficulty in Management**: Roles are difficult to manage and update.

### Scenario 2: Insufficient Permissions

```dax
-- Example of insufficient permissions
VAR NoPermission = 'None'
RETURN
    IF(
        USERPRINCIPALNAME() = "AnyUser", 
        NoPermission
    )
```

**Problem:**
- **Restricted Access**: Users may not have the necessary permissions to perform their tasks.
- **Productivity Issues**: Insufficient permissions can lead to productivity issues and frustration.
- **Management Challenges**: Permissions are challenging to manage and update.
