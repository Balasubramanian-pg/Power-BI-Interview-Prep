# 3. Object Level Security When And How

## Overview

Object level security in Power BI is crucial for controlling access to sensitive data, ensuring that users can only view the information they are authorized to see. Misunderstanding or misconfiguring object level security can lead to data breaches or unauthorized access, compromising the integrity of the data and the trust of stakeholders. The process of implementing object level security involves several stages, which can be represented as follows:

```
Data Model → Row-Level Security (RLS) → Object-Level Security → User Access → Result
           ↓             ↓            ↓
      Data Preparation  RLS Rules   Object Permissions  User Roles
```


## The Security Framework

The security framework in Power BI consists of several key components that work together to provide a robust and flexible security system.

### Row-Level Security (RLS)

Row-Level Security (RLS) is a critical component of the security framework, responsible for controlling access to specific rows of data based on user identity. The characteristics of RLS include:

**Characteristics:**
- **Dynamic**: RLS rules can be based on user attributes, such as username or role.
- **Flexible**: RLS rules can be applied to multiple tables and models.
- **Scalable**: RLS can handle large datasets and complex security scenarios.

### Object-Level Security

Object-Level Security refers to the control of access to specific objects, such as tables, measures, or visualizations, within a Power BI model. The characteristics of Object-Level Security include:

**Characteristics:**
- **Granular**: Object-Level Security can be applied to individual objects or groups of objects.
- **Role-Based**: Access to objects is based on user roles or identities.
- **Hierarchical**: Object-Level Security can be applied to a hierarchy of objects, such as a table and its related measures.

### User Roles and Identities

User Roles and Identities are essential components of the security framework, as they define the access levels and permissions for each user. The characteristics of User Roles and Identities include:

**Characteristics:**
- **Unique**: Each user has a unique identity and role.
- **Dynamic**: User roles and identities can be updated or changed as needed.
- **Integrated**: User roles and identities can be integrated with external systems, such as Active Directory.


## The Security Process

The security process in Power BI involves several stages, from data preparation to user access. The flow of data and security can be represented as follows:

```
Data Source → Data Model → RLS → Object-Level Security → User Access → Result
    ↓            ↓             ↓            ↓            ↓
Data Preparation  Data Modeling  RLS Rules   Object Permissions  User Roles
```

### How Data Flows

When a user accesses a Power BI report, the data flows through several stages, including data preparation, data modeling, RLS, and object-level security. The process can be described as follows:

1. **Data Preparation**: The data is prepared and loaded into the Power BI model.
2. **Data Modeling**: The data is modeled and transformed into a format suitable for analysis.
3. **RLS**: The RLS rules are applied to the data, controlling access to specific rows.
4. **Object-Level Security**: The object-level security rules are applied, controlling access to specific objects.
5. **User Access**: The user accesses the report, and the security rules are enforced.


## DAX and Security

DAX (Data Analysis Expressions) is a powerful language used in Power BI to create calculations and data models. When it comes to security, DAX can be used to create dynamic security rules and permissions.

### What is DAX Security?

DAX security refers to the use of DAX expressions to create dynamic security rules and permissions. For example:

```sql
-- Create a dynamic security rule based on user identity
EVALUATE
FILTER(
    'Table',
    'Table'[User] = USERNAME()
)
```

### Example: Dynamic Security Rule

The following example shows how to create a dynamic security rule using DAX:

```dax
-- Create a dynamic security rule based on user role
EVALUATE
FILTER(
    'Table',
    'Table'[Role] = USERNAME()
)
```

Then, the generated internal representation would be:

```sql
-- Generated internal representation
SELECT *
FROM 'Table'
WHERE 'Table'[Role] = USERNAME()
```

## When Object-Level Security Becomes Problematic

### Scenario 1: Overly Complex Security Rules

```dax
-- Create an overly complex security rule
EVALUATE
FILTER(
    'Table',
    'Table'[Attribute1] = USERNAME() && 'Table'[Attribute2] = USERNAME()
)
```

**Problem:**
- **Performance Issues**: Overly complex security rules can lead to performance issues.
- **Difficulty in Management**: Complex security rules can be difficult to manage and update.
- **Security Risks**: Overly complex security rules can lead to security risks if not properly implemented.

### Scenario 2: Inconsistent Security Rules

```dax
-- Create inconsistent security rules
EVALUATE
FILTER(
    'Table',
    'Table'[Attribute1] = USERNAME() || 'Table'[Attribute2] = USERNAME()
)
```

**Problem:**
- **Inconsistent Access**: Inconsistent security rules can lead to inconsistent access to data.
- **Security Risks**: Inconsistent security rules can lead to security risks if not properly implemented.
- **Difficulty in Management**: Inconsistent security rules can be difficult to manage and update.
