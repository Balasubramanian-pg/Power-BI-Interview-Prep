# 13. Service Principal RLS And User Identity

## Overview

Service Principal Row-Level Security (RLS) and User Identity are crucial concepts in Power BI, as they enable secure data access and filtering based on user roles and identities. Misunderstanding these concepts can lead to data exposure or restricted access, making it essential to grasp their core principles. The flow of Service Principal RLS and User Identity can be represented as follows:
```
Authentication → Authorization → Data Filtering → Result
           ↓             ↓            ↓
      User Identity  Service Principal  RLS Rules
```


## The Core Components of Service Principal RLS

### Service Principals

Service Principals are Azure Active Directory (AAD) applications that represent Power BI services, enabling secure authentication and authorization. They are responsible for managing access to Power BI resources, such as workspaces and reports.

**Characteristics:**
- Unique client ID and client secret
- Registered in Azure Active Directory (AAD)
- Used for authentication and authorization
- Can be used to implement RLS

### User Identity

User Identity refers to the unique characteristics of a Power BI user, such as their username, email, and role. This information is used to determine the user's access level and filter data accordingly.

**Characteristics:**
- Unique username and email
- Assigned roles and permissions
- Used for authentication and authorization
- Can be used to implement dynamic data masking

### Row-Level Security (RLS)

Row-Level Security (RLS) is a feature in Power BI that enables data filtering based on user roles and identities. RLS rules are defined in the data model and applied at query time, ensuring that users only see the data they are authorized to access.

**Characteristics:**
- Defined in the data model
- Applied at query time
- Uses user identity and service principal information
- Can be used to implement dynamic data masking


## The Authentication and Authorization Process

The authentication and authorization process involves several stages, including user authentication, service principal authentication, and data filtering. The flow of this process can be represented as follows:
```
User Authentication → Service Principal Authentication → Data Filtering → Result
    ↓            ↓            ↓
User Identity  Service Principal  RLS Rules
```

### How Data Flows

When a user accesses a Power BI report, the following process occurs:
1. **User Authentication**: The user is authenticated using their username and password.
2. **Service Principal Authentication**: The Power BI service authenticates with the service principal using the client ID and client secret.
3. **Data Filtering**: The RLS rules are applied to the data, filtering out rows that the user is not authorized to access.
4. **Result**: The filtered data is returned to the user.


## Internal Query Language

Power BI uses DAX (Data Analysis Expressions) as its internal query language. DAX is used to define data models, create measures, and implement RLS rules.

### What is DAX?

DAX is a collection of functions, operators, and constants that can be used to create calculations and data models in Power BI.

```sql
-- Example DAX query
EVALUATE
FILTER(
    'Sales',
    'Sales'[Region] = "North"
)
```

### Example: Implementing RLS using DAX

To implement RLS using DAX, you can create a measure that filters data based on the user's role.
```dax
-- User-facing code
Measure = 
CALCULATE(
    SUM('Sales'[Sales Amount]),
    FILTER(
        'Sales',
        'Sales'[Region] = USERNAME()
    )
)

-- Generated internal representation
-- This is not directly visible, but it would be similar to the following
EVALUATE
FILTER(
    'Sales',
    'Sales'[Region] = USERNAME()
)
```

```dax

    )
```
