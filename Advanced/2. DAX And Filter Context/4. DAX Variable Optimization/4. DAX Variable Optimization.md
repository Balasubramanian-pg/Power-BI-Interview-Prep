# DAX Variables (VAR) — Performance, Semantics, and a Concrete Example

Short answer up front: **VAR is not just syntactic sugar.**
A variable expression is evaluated **once per evaluation context**, its value is reused wherever the variable is referenced, and that reuse typically avoids re-running expensive subexpressions. This is documented behavior and the standard best-practice for both reliability and performance. ([Microsoft Learn][1])


## 1. What the engine guarantees (concise)

* A `VAR <name> = <expression>` stores the *result* of `<expression>` and that stored result is reused for the rest of the expression evaluation. This is true for scalar and table variables: once evaluated in that evaluation context, the value does not change. ([Microsoft Learn][1])
* Using variables improves readability and often reduces repeated work because the engine does not re-evaluate the variable expression each time the variable name is referenced. ([Microsoft Learn][2])


## 2. Why that matters for performance (practical points)

* **Avoid duplicated work.** If you have an expensive `CALCULATE(...)`, `FILTER(...)`, or `SUMX(...)` used multiple times inside a single measure, putting it in a VAR ensures the expensive expression is computed once and reused. ([SQLBI][3])
* **Shorter query plans.** Reusing a cached intermediate result simplifies the query plan the engine generates, which generally lowers CPU and I/O. ([Microsoft Learn][2])
* **Deterministic behavior.** Variables also prevent subtle bugs caused by re-evaluation ordering because the variable value is fixed once evaluated. ([SQLBI][4])


## 3. Important nuance — evaluation timing and context

* Variables are evaluated *when the DAX engine reaches the point of their evaluation within the expression* and are evaluated once per evaluation context (for example: per row in a row context or per cell in a visual). The details about **when** exactly (before or after certain CALCULATE parameter processing) can affect results if you rely on side-effects of filter manipulation — so ordering matters. ([SQLBI][4])


## 4. Concrete example — variable prevents multiple evaluations

### Problem (inefficient / repeated work)

You want a YoY percentage where the current-year sales and prior-year sales are each calculated using the same expensive filter; without VAR you might write:

```DAX
YoY% (naive) =
DIVIDE(
    CALCULATE( SUM(Sales[Amount]), <complex filter A> ) 
  - CALCULATE( SUM(Sales[Amount]), SAMEPERIODLASTYEAR('Date'[Date]) , <complex filter A> ),
    CALCULATE( SUM(Sales[Amount]), SAMEPERIODLASTYEAR('Date'[Date]) , <complex filter A> )
)
```

Here `<complex filter A>` is applied three times; the engine would need to evaluate that same complex filtering logic repeatedly.

### Optimized (use VAR)

```DAX
YoY% (optimized) =
VAR SalesNow =
    CALCULATE( SUM(Sales[Amount]), <complex filter A> )
VAR SalesPY =
    CALCULATE( SUM(Sales[Amount]), SAMEPERIODLASTYEAR('Date'[Date]) , <complex filter A> )
RETURN
DIVIDE( SalesNow - SalesPY, SalesPY )
```

Why this is better:

* `SalesNow` and `SalesPY` each evaluate their expensive `CALCULATE(...)` once per evaluation context; the results are reused in the `RETURN`.
* The expensive filter `<complex filter A>` is not re-computed multiple times for the same visual cell. This commonly reduces CPU and I/O and shortens query execution time. ([SQLBI][3])


## 5. Table variables vs. scalar variables — practical difference

* A **scalar** VAR holds a single value; reuse is simple and cheap.
* A **table** VAR holds a table result. The engine does not re-execute the table expression every time you reference the variable; the result is computed once in that evaluation context and the same table is reused. However, materialization and memory behavior can differ depending on engine optimizations, so very large table variables may have memory implications. For exact internal memory/materialization details you should test on your model (see Verification note). ([SQLBI][4])


## 6. Example where VAR prevented repeated evaluation of a function

Imagine this costly expression:

```DAX
BigFilter =
FILTER(
    ALL(BigTable),
    [some complex boolean using LOOKUPVALUE, CONCATENATEX, or nested filters]
)
```

Used twice inside a measure:

* Once to compute a numerator `COUNTROWS(BigFilter)`
* Once to compute a denominator `SUMX(BigFilter, BigTable[Value])`

Without VAR the engine would need to run that `FILTER(...)` twice. With VAR you write:

```DAX
VAR BF = FILTER( ALL(BigTable), <complex predicate> )
VAR Num = COUNTROWS(BF)
VAR Den = SUMX(BF, BigTable[Value])
RETURN DIVIDE(Num, Den)
```

Result: the complex predicate and filter run once per evaluation context; `BF` is reused for both `Num` and `Den`. This often yields measurable performance gains. ([SQLBI][3])


## 7. When variables may *not* help

* If the repeated subexpression depends on changing filter context created later in the measure (ordering issues), putting it in VAR may capture an earlier snapshot and change semantics. Care with evaluation order is required. ([SQLBI][4])
* Very small trivial expressions (e.g., `1 + 1`) won’t benefit. The cost of evaluation is already negligible.

## 10. Practical checklist to apply VAR for optimization

* Identify repeated, expensive subexpressions (filters, CALCULATEs, iterators).
* Put each repeated expression into a clearly named `VAR`.
* Ensure evaluation order/semantics are preserved (test results before/after).
* Use DAX Studio + Performance Analyzer to measure improvement. ([SQLBI][3])
