# Filter Propagation in Power BI

## What Happens Across a Chain A → B → C

Filter propagation is governed entirely by **relationship directionality** and **cardinality**. Nothing magical happens. The model follows strict rules.

Let’s assume:

* Table A → Table B → Table C
* All relationships are active
* Default direction is single

We’ll walk through exactly what happens.


# 1. Default Case: Single Direction Relationships

Assume:

A (Dimension) → B (Fact)
B (Dimension) → C (Fact)

Single direction means:

Filters flow from the “one” side to the “many” side.

So in typical star schema:

Dimension → Fact

Never the reverse.


## Case 1: Filtering Table A

If you filter A:

* Filter propagates to B
* Then propagates to C

Because filters move downstream along relationship direction.

So A filters B, and B filters C.

This works naturally.


## Case 2: Filtering Table C

If you filter C:

Does A get filtered?

Answer: No. Not by default.

Why?

Because single-direction relationships only allow filters to move one way.

C cannot push filters upstream to B.
B cannot push filters upstream to A.

So A remains unfiltered.


# 2. Understanding Directionality Deeply

In Power BI:

Each relationship has:

* Cardinality (1:* or *:*)
* Cross-filter direction (Single or Both)

Filter flow follows:

One side → Many side (if single)

If bidirectional:
Many ↔ One

So filter movement depends entirely on how you configure the relationship.


# 3. Visual Example

Imagine:

A = Product
B = Sales
C = Promotions

Relationships:

Product → Sales
Sales → Promotions

If you filter Promotions table:

* Sales does not get filtered unless relationship is bidirectional.
* Product definitely does not get filtered.

Because filters do not climb upstream by default.


# 4. When Would Table A Get Filtered?

Only in these scenarios:

## Scenario 1: Bidirectional Relationships

If you set:

B ↔ C
A ↔ B

Then:

Filtering C
→ filters B
→ filters A

Because filters can now flow both ways.

But this is dangerous in complex models.


## Scenario 2: Using DAX to Force It

Even without changing relationship direction, you can force filter propagation using DAX.


# 5. Forcing Filter Propagation

## Method 1: CROSSFILTER

Inside CALCULATE:

```DAX
CALCULATE(
    [Measure],
    CROSSFILTER(B[Key], C[Key], BOTH)
)
```

Temporarily changes relationship direction for that calculation only.

This is safer than model-level bidirectional relationships.


## Method 2: TREATAS

You can manually inject filters:

```DAX
CALCULATE(
    [Measure],
    TREATAS(VALUES(C[Key]), B[Key])
)
```

This takes filter values from C
and applies them to B as if relationship existed.

You can chain it:

```DAX
CALCULATE(
    [Measure],
    TREATAS(VALUES(C[Key]), B[Key]),
    TREATAS(VALUES(B[Key]), A[Key])
)
```

Now filtering C impacts A.


## Method 3: Bridge Table

In many-to-many or complex chains:

Introduce a bridge table that connects dimensions.

This allows clean filter propagation without forcing bidirectional chaos.


# 6. Why Bidirectional Relationships Are Dangerous

They can cause:

* Ambiguous filter paths
* Circular dependencies
* Unexpected results
* Performance degradation

Especially in snowflake models.

Enterprise best practice:

Keep relationships single direction unless absolutely required.

Use DAX to control direction at measure level.


# 7. Special Case: Many-to-Many Relationships

In many-to-many:

Filters may behave differently because Power BI uses expanded tables.

But propagation still respects direction.

It does not magically propagate upstream.


# 8. Real Bug I Encountered

Model:

Customer → Sales → RegionMapping

RegionMapping was connected to Sales.

User filtered RegionMapping.
Customer visuals were not filtering.

Developer solution attempt:
Turn on bidirectional everywhere.

Result:
Totals doubled due to ambiguous paths.

Correct fix:
Use TREATAS in measure instead of changing relationship direction.

This preserved clean model design.


# 9. Key Rules to Remember

1. Filters only flow in relationship direction.
2. Single direction = One → Many only.
3. Filtering the “many” side does not affect the “one” side.
4. Bidirectional allows upstream propagation.
5. Prefer DAX-level forcing over model-level bidirectional.


# 10. Interview-Ready Summary

"In a chain A → B → C with single-direction relationships, filtering A will propagate to B and then C. However, filtering C will not propagate back to B or A because filters only move in the defined relationship direction. To force upstream propagation, I would either use CROSSFILTER inside CALCULATE to temporarily change direction or use TREATAS to manually apply filter values across tables. I avoid model-level bidirectional relationships unless absolutely necessary to prevent ambiguity and performance issues."
