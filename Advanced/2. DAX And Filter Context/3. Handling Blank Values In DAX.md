# Handling `BLANK()` in DAX

## How It Behaves in Arithmetic vs Logical Operations

`BLANK()` in DAX is not just “empty.”
It is a special semantic value that behaves differently depending on context.

If you misunderstand it, ratios inflate, averages lie, and visuals quietly mislead.

Let’s break this down precisely.


# 1. How `BLANK()` Behaves in Arithmetic Operations

In arithmetic, `BLANK()` often behaves like zero.

## 1.1 Addition and Subtraction

```DAX
BLANK() + 5 = 5  
5 + BLANK() = 5  
BLANK() - 5 = -5  
```

DAX implicitly converts `BLANK()` to 0 in numeric arithmetic.

So:

`BLANK() + BLANK()` = 0

This can be dangerous if you expected the result to stay blank.


## 1.2 Multiplication

```DAX
BLANK() * 5 = BLANK()
```

Multiplication preserves blank.

Reason:
If any operand is blank, result is blank.


## 1.3 Division

```DAX
5 / BLANK()
```

Returns BLANK, not error.

And:

```DAX
BLANK() / 5
```

Returns BLANK.

This is different from SQL, where division by NULL propagates NULL strictly.


# 2. How `BLANK()` Behaves in Logical Functions

Logical operations treat `BLANK()` differently.

## 2.1 Equality Comparison

```DAX
BLANK() = 0
```

Returns TRUE.

This surprises many people.

Because DAX treats blank as zero in numeric comparisons.

But:

```DAX
BLANK() = ""
```

Also TRUE in text comparison context.

So blank is “context-sensitive.”


## 2.2 Boolean Context

In IF statements:

```DAX
IF(BLANK(), "Yes", "No")
```

Returns "No".

Blank behaves like FALSE.


## 2.3 ISBLANK()

This is the only reliable way to detect blank:

```DAX
ISBLANK([Measure])
```

Do not rely on = 0 or = "" checks.


# 3. Why BLANK Skews Averages and Ratios

The problem is subtle.

## 3.1 Average Behavior

Example:

You calculate:

```DAX
AVERAGE(Sales[Amount])
```

If some rows are BLANK:

* AVERAGE ignores blank rows.
* But if blanks were converted to 0 earlier, they are counted.

This changes denominator.

So:

True blank → excluded
Zero → included

Big difference.


## 3.2 Ratio Problem

Example:

```DAX
Conversion Rate :=
DIVIDE([Conversions], [Visitors])
```

If [Visitors] is blank, DIVIDE returns blank.

But if Visitors = 0:

You may get 0 or blank depending on alternate result.

If upstream logic converted blank to 0,
denominator changes silently.

Ratios inflate or deflate incorrectly.


# 4. Real Bug I Encountered

We had:

```DAX
Avg Revenue :=
SUM(Sales[Revenue]) / COUNTROWS(Sales)
```

Problem:
Revenue column had blanks for non-billed transactions.

SUM ignored blanks correctly.

But COUNTROWS counted all rows, including blank revenue rows.

Result:
Average revenue was artificially low.

Fix:

```DAX
Avg Revenue :=
AVERAGEX(
    FILTER(Sales, NOT ISBLANK(Sales[Revenue])),
    Sales[Revenue]
)
```

Now:
Only valid revenue rows included.


# 5. Ensuring Blanks Don’t Skew Results

## 5.1 Use DIVIDE Instead of /

Always use:

```DAX
DIVIDE(numerator, denominator)
```

Because it:

* Handles blank safely
* Avoids divide-by-zero errors
* Returns BLANK instead of Infinity


## 5.2 Control Denominator Explicitly

Instead of:

```DAX
SUM(Sales[Revenue]) / COUNTROWS(Sales)
```

Use:

```DAX
DIVIDE(
    SUM(Sales[Revenue]),
    COUNTROWS(
        FILTER(Sales, NOT ISBLANK(Sales[Revenue]))
    )
)
```

Now blanks do not distort denominator.


## 5.3 Use COALESCE Carefully

```DAX
COALESCE([Measure], 0)
```

Converts blank to zero.

But only do this at presentation layer.

If done inside core measure, it may:

* Change averages
* Affect filter logic
* Break conditional formatting


## 5.4 Avoid Converting Blank Too Early

Do NOT do:

```DAX
SUM(Sales[Revenue]) + 0
```

This forces blank to zero.

Let blanks remain blank until final display step.


# 6. Visual Impact of Blank vs Zero

Blank:

* Visual hides the data point.
* Does not show bar/line.
* Often excluded from averages.

Zero:

* Shows as real value.
* Included in aggregation.
* Changes totals.

This is not cosmetic.
It changes meaning.


# 7. Advanced Insight

DAX engine treats BLANK as:

* 0 in numeric arithmetic
* FALSE in Boolean context
* Empty string in text context

But:

Aggregation functions treat blank differently.

SUM ignores blank.
COUNT does not.
COUNTBLANK counts only blanks.
AVERAGE ignores blanks.

So behavior depends on function category.


# 8. Best-Practice Pattern for Ratios

Safe pattern:

```DAX
Safe Ratio :=
VAR Num = [Numerator]
VAR Den = [Denominator]
RETURN
IF(
    ISBLANK(Den) || Den = 0,
    BLANK(),
    DIVIDE(Num, Den)
)
```

This ensures:

* No artificial zeros
* No inflated percentages
* No misleading totals


# 9. Interview-Ready Summary

"`BLANK()` in DAX behaves differently depending on context. In arithmetic addition and subtraction it behaves like zero, but multiplication and division propagate blank. In logical comparisons, blank equals zero in numeric contexts and behaves like FALSE in Boolean expressions. This can skew averages and ratios if blanks are unintentionally converted to zero. To prevent distortion, I use ISBLANK checks, AVERAGEX with filtered rows, and the DIVIDE function instead of the division operator. I avoid converting blanks to zero until the presentation layer."
