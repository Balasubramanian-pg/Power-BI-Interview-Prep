# Context Transition in DAX

## What Actually Happens When You Use CALCULATE on a Measure

Context Transition is one of those DAX behaviors that feels invisible until it breaks your logic.

Let’s break it down precisely and technically.


# 1. First: The Two Contexts in DAX

## Row Context

Row context exists when DAX evaluates one row at a time.

Examples:

* Calculated columns
* Iterators like SUMX, FILTER, ADDCOLUMNS

Row context means:
You can read column values from the “current row.”

It does NOT filter the model.


## Filter Context

Filter context is the set of filters applied to a calculation.

Examples:

* Report slicers
* Visual filters
* CALCULATE filters

Filter context determines what data is visible to aggregation functions.


# 2. What Context Transition Actually Is

Context transition happens when:

CALCULATE is executed inside a row context.

What it does:

It converts the current row context into an equivalent filter context.

Technically:

For each column in the current row context,
CALCULATE generates an implicit filter:

Table[Column] = CurrentRowValue

Then it evaluates the expression under that filter context.


# 3. Step-by-Step Example

Assume a calculated column in Sales table:

```DAX
Customer Total :=
CALCULATE(SUM(Sales[Amount]))
```

What happens internally for each row:

Step 1
Row context exists:
Sales[CustomerID] = 101
Sales[OrderID] = 5001
Sales[Amount] = 200

Step 2
CALCULATE triggers context transition.

Step 3
Row context converts to filter context:

Filter applied:
Sales[CustomerID] = 101
Sales[OrderID] = 5001
Sales[Amount] = 200

Step 4
SUM(Sales[Amount]) now runs under this filter context.

Result:
It sums only that single row.

So the column just returns the same Amount value.

This surprises many people.


# 4. Important Clarification

If you call a measure inside a row context:

```DAX
SUMX(Sales, [Total Sales])
```

Even if you do not write CALCULATE,
measures are automatically wrapped in CALCULATE.

So context transition still happens.

This is a critical point.


# 5. Exact Internal Sequence

When CALCULATE is evaluated:

1. Evaluate filter arguments.
2. Perform context transition if row context exists.
3. Apply explicit filters.
4. Evaluate expression in modified filter context.

Order matters.


# 6. Real Bug I Encountered

## Scenario

We were calculating customer-level contribution inside a calculated column.

Model:

* Sales table
* Customer table (related via CustomerID)

Requirement:
Calculate each sale’s percentage of total customer sales.

Initial attempt:

```DAX
Customer % :=
DIVIDE(
    Sales[Amount],
    CALCULATE(SUM(Sales[Amount]), ALLEXCEPT(Sales, Sales[CustomerID]))
)
```

We expected:

Numerator = current row amount
Denominator = total for that customer

But result was always 100%.


# 7. Why It Broke

The calculated column runs in row context.

Inside CALCULATE:

Context transition occurred.

Filter context became:

Sales[CustomerID] = current value
Sales[OrderID] = current value
Sales[Amount] = current value

Then ALLEXCEPT removed filters except CustomerID.

But the context transition happened BEFORE ALLEXCEPT.

So:

* Row filters were applied
* ALLEXCEPT removed some filters
* But behavior was inconsistent because of evaluation order

Effectively, the denominator was still evaluating only the current row.


# 8. Root Cause

Unintended context transition caused:

Each row to filter itself before aggregation.

We thought we were calculating a group total.
We were actually calculating a row total.


# 9. How We Fixed It

Solution:

Move logic to a measure instead of calculated column.

Correct version:

```DAX
Customer % :=
DIVIDE(
    SUM(Sales[Amount]),
    CALCULATE(
        SUM(Sales[Amount]),
        ALLEXCEPT(Customer, Customer[CustomerID])
    )
)
```

Now:

* No row context
* Only filter context
* CALCULATE modifies filter context correctly
* No unintended context transition

Result was accurate.


# 10. Another Common Bug Pattern

Using SUMX over a dimension:

```DAX
SUMX(Customer, [Total Sales])
```

If [Total Sales] already uses CALCULATE,
you get repeated context transition.

Sometimes this:

* Multiplies totals
* Produces inflated results

Fix:
Avoid unnecessary iterators.
Or isolate logic in variables outside row context.


# 11. Mental Model for Context Transition

Think of row context as:

"I am standing on one row."

Think of filter context as:

"Only rows matching this condition are visible."

Context transition says:

"Take the row I am standing on, and restrict the entire table to match it."

It turns position into restriction.


# 12. Advanced Insight

Context transition:

* Only occurs inside CALCULATE or CALCULATETABLE.
* Happens automatically when calling a measure inside row context.
* Converts ALL columns in current row context, not just key columns.
* Can dramatically change performance if row context contains many columns.


# 13. Interview-Ready Summary

"Context transition occurs when CALCULATE is executed inside a row context. It converts the current row’s column values into equivalent filters, effectively turning row context into filter context before evaluating the expression. I encountered a bug where a calculated column intended to compute customer-level percentages always returned 100%. The issue was unintended context transition causing the denominator to evaluate only the current row. The fix was moving the logic to a measure, eliminating row context and ensuring the filter context behaved as expected."
