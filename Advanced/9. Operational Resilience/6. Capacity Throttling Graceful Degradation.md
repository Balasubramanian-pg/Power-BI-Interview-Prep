# Capacity Throttling and Graceful Degradation

## Overview

Power BI Premium and Microsoft Fabric employ capacity throttling as a self-protection mechanism that preserves system stability when compute consumption exceeds provisioned resources. This approach allows the platform to maintain partial functionality during resource constraints rather than failing catastrophically. Understanding throttling behavior enables capacity administrators to design resilient workloads that degrade gracefully under pressure.

## Throttling Mechanics

Throttling occurs when operations consume more compute unit seconds than the capacity SKU permits within defined time windows [[1]]. The platform applies throttling at the capacity level, meaning one overloaded capacity does not affect others running in the same tenant. This isolation prevents cascading failures across unrelated workspaces.

Fabric employs two complementary mechanisms that work before throttling becomes necessary. Bursting allows operations to temporarily exceed provisioned compute, delivering fast performance without requiring oversized capacities [[1]]. Smoothing distributes compute consumption across future timepoints, with interactive operations smoothed over five to sixty-four minutes and background operations smoothed across twenty-four hours [[1]].

## Throttle Stages

The throttling system progresses through defined stages as capacity utilization increases. Overage protection permits ten minutes of future capacity consumption before any throttling begins [[1]]. This buffer absorbs temporary spikes without impacting user experience.

When utilization exceeds the ten-minute threshold, interactive operations receive twenty-second delays at submission. Background operations continue unaffected during this phase. If consumption reaches one hour of future capacity, new interactive operations are rejected entirely while background operations remain permitted [[1]]. The final stage occurs when twenty-four hours of future capacity are consumed, at which point all new requests face rejection until carryforward compute is paid down through idle capacity.

## Identifying Throttling Events

Users encounter specific error messages when throttling occurs. The status code `CapacityLimitExceeded` appears alongside messages indicating the organization's compute capacity has exceeded limits [[1]]. Refresh operations log distinct messages noting that too many semantic models are being processed concurrently.

Capacity administrators can confirm throttling through the Fabric Capacity Metrics app. The Compute page displays system events showing throttling history, while throttling charts illustrate when smoothed usage exceeds defined limits [[1]]. The Overages tab provides visual history of capacity overutilization including carryforward, cumulative, and burndown data.

## Graceful Degradation Strategies

Designing for graceful degradation requires accepting that throttling will occur and planning workload behavior accordingly. Several approaches reduce throttling impact while maintaining essential functionality.

**Schedule Optimization**

Refresh operations should occur during non-peak periods when overall capacity usage remains low [[9]]. The schedule view in the metrics app reveals whether refresh events cluster together unnecessarily. Spreading refreshes across available windows prevents concurrent operations from triggering throttle stages prematurely.

**Semantic Model Scale-Out**

Enabling semantic model scale-out creates read-only replicas that isolate refresh operations from interactive queries [[9]]. The read-write replica handles refresh while users query the read-only copy. This separation prevents refresh operations from consuming compute needed for user-facing workloads.

**Incremental Refresh**

Large semantic models benefit from incremental refresh policies that partition tables and refresh only recent data [[9]]. This approach reduces refresh duration and compute consumption, creating room for additional operations within the same capacity window.

**Model Complexity Reduction**

Calculated tables and columns computed during refresh increase memory pressure and extend processing time [[9]]. Moving these calculations to data sources or ETL processes reduces the refresh burden. Simplified models complete faster and consume fewer resources during each refresh cycle.

**Retry Logic**

Custom refresh operations using XMLA or REST APIs should implement retry logic with exponential backoff [[9]]. The built-in Power BI scheduling facility performs automatic retries when throttling occurs, but custom implementations require explicit handling. Retries allow operations to succeed once capacity becomes available.

## Compound Throttling Protection

Fabric includes protection against compound throttling when operations trigger chains of dependent requests [[1]]. A report visual querying a semantic model that reads from OneLake forms such a chain. Without protection, each link in the chain could face independent throttling decisions.

Compound throttling protection ensures requests are throttled once per capacity in the chain rather than at each step. Workloads supporting this protection include semantic models using DirectQuery to other semantic models and DAX queries from paginated reports [[1]]. This reduces the likelihood of cascading throttling across dependent operations.

## Capacity Management Actions

When throttling occurs frequently, administrators have several options to restore normal operations.

**SKU Adjustment**

F SKU capacities can be resized manually at any time through admin settings [[1]]. Increasing the SKU provides more compute per timepoint, accelerating burndown of carryforward usage. This temporary measure resolves immediate throttling while longer-term capacity planning occurs.

**Pause and Resume**

Pausing a capacity triggers billing for accumulated future capacity usage [[1]]. When resumed, the capacity starts with zero future consumption, accepting new operations immediately. This approach provides rapid relief from throttling states.

**Autoscale Enablement**

P SKU capacities can enable autoscale billing, which adds resources automatically during high-demand periods [[11]]. This prevents throttling by expanding capacity rather than restricting operations. Autoscale requires appropriate licensing and budget approval.

## Monitoring and Alerting

Proactive monitoring prevents throttling from impacting users unexpectedly. Capacity administrators should configure email alerts notifying when capacity reaches one hundred percent of provisioned resources [[1]]. These alerts provide early warning before throttling stages activate.

Regular review of the Capacity Metrics app reveals utilization patterns that predict future throttling. Trends showing consistent overages indicate the need for capacity expansion or workload redistribution across multiple capacities [[1]].

## Workload-Specific Considerations

Certain Fabric workloads exhibit different throttling behavior based on their operational requirements. Eventstreams run continuously and cannot be throttled conventionally. Instead, compute allocated to keeping streams open is reduced until capacity returns to acceptable levels [[1]].

Real-Time Intelligence skips the initial delay stage of throttling to maintain real-time performance characteristics [[1]]. Throttling begins at the rejection phase rather than applying twenty-second delays. This ensures time-sensitive operations maintain responsiveness during capacity constraints.

Warehouse operations classify almost all requests as background operations, benefiting from twenty-four-hour smoothing [[1]]. This prevents peaks from triggering throttling quickly, allowing flexible usage patterns for data warehousing workloads.

## Operational Resilience Principles

Capacity throttling represents one component of broader operational resilience. The system maintains in-flight operations even after throttling begins, allowing submitted work to complete [[1]]. This assurance prevents data loss or partial execution during capacity surges.

Graceful degradation accepts reduced functionality rather than complete failure. Interactive users may experience delays while background operations continue. Background operations may queue while critical system functions remain available. This tiered approach preserves essential capabilities during resource constraints.

Designing resilient Power BI deployments requires understanding throttling triggers, monitoring capacity health, and implementing workload strategies that accommodate temporary resource limitations. The platform provides mechanisms to manage throttling, but administrators must apply these tools appropriately for their specific usage patterns and business requirements.
