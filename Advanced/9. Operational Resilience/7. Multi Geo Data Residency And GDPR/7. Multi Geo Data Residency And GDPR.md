# Multi-Geo Data Residency and GDPR Compliance

## Overview

Multi-Geo capabilities in Power BI Premium and Microsoft Fabric enable organizations to deploy content across data centers in regions separate from the tenant home region [[21]]. This functionality addresses regional regulatory requirements, industry-specific compliance mandates, and organizational data sovereignty policies. Understanding the architectural implications of Multi-Geo deployments ensures proper implementation while maintaining GDPR compliance and operational continuity.

## Multi-Geo Architecture

A geography contains multiple Azure regions within defined boundaries. The United States geography includes West Central US and South Central US as distinct regions [[21]]. Organizations select specific regions during capacity creation, determining where compute and storage resources physically reside.

Once a capacity is created in a selected region, all workspaces assigned to that capacity store their content in the corresponding data center [[21]]. This regional assignment persists throughout the capacity lifecycle. Capacity locations cannot be changed after creation, requiring new capacity provisioning for region migration [[19]].

Sovereign clouds support Multi-Geo across regions within that cloud environment [[21]]. China North does not support Multi-Geo capabilities. Power BI Premium Per-User licenses lack Multi-Geo support, requiring Premium capacity or Fabric capacity for regional deployment flexibility [[21]].

## GDPR Compliance Requirements

The General Data Protection Regulation governs personal data processing for European Union residents regardless of organizational location [[9]]. Power BI users handling EU citizen data must implement safeguards addressing data sovereignty, transfer restrictions, and subject rights. Government access laws such as FISA 702 create additional complexity for cross-border data transfers requiring careful legal review [[9]].

GDPR compliance extends beyond technical controls to encompass organizational processes, documentation, and accountability measures. Power BI provides built-in features supporting compliance obligations, but administrators must configure and operate these features appropriately for their specific regulatory context [[9]].

## Data Residency Configuration

Multi-Geo enablement occurs during capacity creation through region selection in the Admin portal [[21]]. The region dropdown menu displays available geographies, with the home tenant region as the default selection. Choosing a non-default region triggers confirmation prompts ensuring administrators understand the regional assignment implications [[19]].

Capacity settings display region information for all provisioned capacities. The Admin portal Capacity settings page shows Power BI Embedded capacities alongside Premium capacities, each indicating their geographic location [[19]]. This visibility supports compliance auditing and capacity management activities.

## Workspace Migration Between Regions

Moving workspaces between capacities in different regions requires creating new capacity in the target region, then reassigning workspaces [[19]]. During migration, certain operations may fail including semantic model publishing and scheduled refresh activities. Neither source nor destination capacities should be deleted or paused during migration to prevent content loss [[21]].

Workspace migration follows these steps: open workspace settings, select Workspace type from the sidebar, then choose the target capacity from the License capacity dropdown menu [[21]]. Free users experience no downtime during migration if the tenant maintains spare v-cores. Temporary movement to shared capacity avoids v-core requirements but introduces user downtime [[21]].

Large-storage format semantic models cannot be moved between regions after creation [[21]]. Reports referencing relocated large-format models return Cannot load model errors. Organizations must either maintain these models in their original region or redeploy as new models in the target region before removing the original [[21]].

## Data Location and Metadata Distribution

Compute and storage resources including OneLake and experience-specific storage reside in the Multi-Geo region [[21]]. Certain tenant metadata remains in the home region regardless of capacity location. This distributed metadata architecture includes push datasets, dashboard and report metadata, service buses for gateway queries, permissions, semantic model credentials, and metadata linked to Purview Data Map [[21]].

Cached data and queries stored in remote regions remain at rest in that region. Data at rest replicates to another region within the same Azure geography for disaster recovery when the geography contains multiple regions [[21]]. Data in transit may traverse multiple geographies depending on routing configurations.

Query text and query result traffic for imported and DirectQuery models does not transit through the home region [[21]]. Report metadata originates from the home region, and DNS routing states may direct certain traffic outside the region. Features including screenshots and data alerts process data in the home region regardless of capacity location [[21]].

Enhanced metadata scanning caches detailed semantic model metadata in the home tenant even when scanning remote region semantic models [[21]]. This architectural decision affects organizations with strict metadata residency requirements requiring careful assessment during deployment planning.

## Data Retention During Region Transitions

Source data may remain in the original region for up to 30 days following region migration [[21]]. End users cannot access this data during the retention period. The system removes and destroys data from the source region throughout this 30-day window [[21]].

Organizations must confirm that region movements comply with corporate and government requirements before initiating data transfer [[21]]. This validation prevents inadvertent regulatory violations during capacity reconfiguration activities.

## Encryption and Key Management

Power BI employs advanced encryption protocols protecting data storage throughout the data lifecycle [[9]]. Data in transit requires TLS 1.2 or higher enforcement [[24]]. Microsoft-managed keys typically handle encryption, though organizations may implement bring-your-own-key strategies for additional control [[9]].

Azure Key Vault integration enables organizations to manage encryption keys separately from Power BI service [[9]]. Administrators configure access policies allowing Power BI to use these keys during data processing and storage operations. BYOK strategies require PowerShell commands for implementation and apply only to data uploaded from PBIX files [[9]].

Key revocation renders associated data unreadable within approximately 30 minutes [[9]]. This capability provides emergency response options for compromised keys or termination scenarios requiring immediate data access termination.

## Data Loss Prevention Policies

Data Loss Prevention policies align with Microsoft 365 policies, enabling organizations to enforce rules and trigger remediation actions when sensitive data detection occurs [[9]]. DLP policies apply at the dataset level ensuring data compliance and privacy throughout the analytics lifecycle.

Microsoft Purview portal serves as the configuration interface for DLP policies [[9]]. Custom policies must be created for Fabric environments since predefined templates remain unavailable. Policies monitor various item types including semantic models, datasets, and lakehouses [[9]].

Policy tips attach to items when sensitive content detection occurs, informing users about sensitive information nature and guiding appropriate actions [[9]]. Access restrictions limit visibility to data owners or authorized users based on policy configuration settings.

## Row-Level Security Implementation

Row-Level Security restricts data access at the row level, ensuring users view only data relevant to their responsibilities [[9]]. RLS implementation requires defining roles and rules in Power BI Desktop before publishing reports to the Power BI service [[9]].

Dynamic row-level security employs DAX functions including USERNAME() and USERPRINCIPALNAME() enabling flexible access control based on authenticated user identity [[9]]. Role-based access control complements RLS by restricting user permissions according to organizational roles [[9]].

Geographic boundaries can be incorporated into RLS rules, restricting data access based on user location or regional assignments [[1]]. This approach supports data residency compliance by preventing cross-border data access even when technical infrastructure permits connectivity.

## Azure Active Directory Integration

Azure Active Directory provides centralized user authentication and access management for Power BI environments [[9]]. OAuth authentication enables user-specific content viewing based on assigned permissions. Multi-factor authentication strengthens security by requiring additional verification steps during authentication [[9]].

Conditional access policies apply security measures based on user conditions including location, device compliance, and risk assessment [[9]]. Azure AD integration with Office 365 applications simplifies user login processes while maintaining security controls [[9]].

Activity monitoring and detailed logging support security incident detection and response [[9]]. Audit trails track access and modifications to data and reports, ensuring transparency and accountability for compliance demonstrations [[9]].

## Data Subject Rights Support

GDPR grants data subjects rights including access, rectification, erasure, processing restriction, processing objection, and data portability [[9]]. Power BI supports these rights through features facilitating data export and deletion, enabling organizations to respond efficiently to subject requests [[9]].

Transparent processes for handling data subject requests must operate in timely manners with clear communication to requestors [[9]]. Power BI compliance features including sensitivity labels and DLP policies support personal data safeguarding during request fulfillment [[9]].

Microsoft Purview integration streamlines data subject request management, allowing organizations to track, process, and document each request under GDPR requirements [[9]]. Comprehensive records of requests and responses demonstrate compliance during regulatory audits [[9]].

## Operational Considerations

Dataflows Gen1 require Azure Data Lake Storage Gen2 configuration for Multi-Geo usage [[21]]. Metrics in Power BI lack Multi-Geo support, limiting deployment options for organizations relying on this feature [[21]]. Power BI Q&A Settings remain unsupported due to detailed semantic model metadata residing in the home tenant [[21]].

Workspaces containing non-Power BI Fabric items cannot move between regions [[21]]. All non-Power BI Fabric items must be deleted before workspace migration. Non-Power BI items can be recreated after migration completes, typically within 30 minutes [[21]].

## Monitoring and Compliance Verification

Regular security audits verify alignment with GDPR standards and organizational policies [[9]]. Annual audits accommodate evolving regulations and adapt practices accordingly [[9]]. Audit evaluations assess data management policy effectiveness, role definitions, and responsibility assignments [[9]].

Microsoft Defender for Cloud Apps facilitates real-time monitoring and alerts for user activities involving sensitive data [[9]]. Integration with Power BI enables effective anomaly detection tracking data access patterns and user behaviors [[9]]. Proactive alert systems enable quicker responses to potential security breaches before escalation [[9]].

Capacity Metrics apps and admin portals provide visibility into regional capacity utilization and compliance status [[1]]. Regular review reveals utilization patterns predicting future compliance challenges. Trends indicating consistent cross-region data flows may require architectural adjustments or policy updates [[1]].

## Deployment Patterns

Organizations typically adopt one of several deployment patterns when implementing Multi-Geo capabilities. Single-region deployments maintain all capacities in the home region, simplifying management while potentially violating residency requirements for international operations [[23]].

Multi-region deployments provision capacities in multiple geographies, distributing workloads according to data residency requirements [[23]]. This approach increases operational complexity but provides compliance assurance for organizations operating across regulatory boundaries.

Hybrid patterns combine regional capacities with centralized governance, balancing compliance requirements against management overhead [[23]]. Governance teams must ensure capacity settings align with residency requirements while monitoring cross-region data flows [[28]].

## Limitations and Constraints

Power BI Embedded Multi-Geo limitations mirror Power BI Premium Multi-Geo considerations [[19]]. Capacity location changes require new capacity creation rather than modification of existing resources [[19]]. Content deletion without reassignment moves workspace content to shared capacity in the home region [[19]].

API support enables Multi-Geo management through Get Capacities, Assign To Capacity, and Azure Resource Manager operations [[19]]. The Get Capacities API returns capacity lists including region properties specifying location. Assign To Capacity operations require admin permissions on workspaces and admin or assign permissions on target capacities [[19]].

Mobile applications present additional considerations for geographic data residency. Android and iOS platforms do not support geographic data residency requirements for notifications when certain features are enabled [[6]]. Organizations with strict notification residency requirements must evaluate mobile deployment implications separately.

## Compliance Documentation

Organizations must document data residency compliance for each deployment region [[1]]. This documentation supports regulatory audits and demonstrates due diligence in data protection efforts. Records should include capacity locations, workspace assignments, data flow diagrams, and access control configurations [[1]].

Compliance documentation extends to vendor assessments, data processing agreements, and cross-border transfer mechanisms [[9]]. Organizations processing EU citizen data must maintain records of processing activities as required by GDPR Article 30 [[9]].

Regular training programs educate users about GDPR requirements and data security responsibilities [[9]]. Training reduces human error risks, a significant contributor to data breaches [[9]]. Well-informed teams maintain compliance and ensure data protection remains organizational priority [[9]].
