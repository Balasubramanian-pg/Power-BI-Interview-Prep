# Incremental Refresh and Hard Deletes in Power BI

## Overview

Incremental Refresh is designed to improve refresh performance by processing only **new or changed data partitions** instead of reloading entire datasets. While highly efficient, it introduces a critical operational challenge known as the **Hard Delete Problem**.

Hard deletes occur when records are physically removed from the source system. Incremental refresh mechanisms typically detect **new or updated rows**, but not rows that disappear.

This creates a paradox where refresh succeeds technically while historical accuracy slowly diverges.

```id="k5p2rn"
Source System Delete
        ↓
Row Removed From Source
        ↓
Incremental Refresh Runs
        ↓
Old Partition Not Reprocessed
        ↓
Deleted Data Persists in Model
```

The dataset becomes historically incorrect without producing errors.


## How Incremental Refresh Works

Incremental refresh partitions data using time ranges.

Typical configuration:

* Historical period stored
* Recent period refreshed
* Future partitions created automatically

Example:

```id="p3v8dy"
Historical Data   Recent Window
(Archive)         (Refresh Zone)
Jan–Dec 2023 | Last 7 Days
```

During refresh:

1. Old partitions remain untouched.
2. Only defined refresh window reloads data.
3. New partitions are appended.

Efficiency comes from avoiding full reloads.


## What Are Hard Deletes

A hard delete means a record is permanently removed from the source table.

Example:

| OrderID | Status    |
| ------- | --------- |
| 1001    | Cancelled |

Later removed entirely from source.

Unlike updates, no trace remains for incremental logic to detect.


## Why Incremental Refresh Misses Deletes

Incremental refresh relies on **filter predicates**, usually:

```id="f6n1qt"
RangeStart <= Date < RangeEnd
```

The engine reloads rows matching the date range.

If a deleted row belongs to an **older partition**, that partition is never refreshed again.

Result:

```id="x8d4sw"
Source = Correct
Power BI = Contains Deleted Rows
```

Data drift accumulates over time.


## The Operational Paradox

Expected assumption:

```id="r7c3hm"
Incremental Refresh → Accurate + Fast
```

Operational reality:

```id="z2m9ke"
Incremental Refresh
        +
Hard Deletes
        ↓
Performance Maintained
Accuracy Slowly Degrades
```

Efficiency trades off against historical correctness.


## Real Enterprise Scenario

### Transaction System

Orders older than 90 days are purged from operational database.

Incremental policy:

* Store 5 years
* Refresh last 7 days

Outcome:

* Deleted historical orders remain indefinitely in Power BI.
* Finance reconciliation begins diverging.

Refresh logs show success.


## Types of Delete Behaviors

### 1. True Hard Deletes

Rows physically removed.

Highest risk scenario.


### 2. Soft Deletes (Preferred)

Rows marked inactive.

Example:

```id="w1p6az"
IsDeleted = TRUE
```

Incremental refresh detects updates.


### 3. Archive Migration

Old data moved to another table or storage layer.

Dataset stops receiving corrections.


## Detecting Hard Delete Issues

Warning indicators:

* Dataset row count continuously grows
* Source totals smaller than Power BI totals
* Historical reconciliation mismatches
* Audit discrepancies

Comparison checks often reveal drift months later.


## Mitigation Strategies

## Strategy 1: Sliding Refresh Window

Refresh a broader historical period.

Example:

```id="n5x2vb"
Refresh Last 90 Days
```

Older partitions periodically corrected.

Tradeoff:
Higher refresh cost.


## Strategy 2: Periodic Full Refresh

Schedule occasional full dataset refresh.

Example pattern:

```id="d4q7mt"
Daily Incremental
Weekly Full Refresh
```

Resynchronizes history.


## Strategy 3: Detect Changes Pattern

Use change tracking columns.

Example:

```id="u9k1ex"
LastModifiedDate
```

Deletes trigger updates indirectly.

Works only if source supports tracking.


## Strategy 4: Soft Delete Modeling (Best Practice)

Modify source or ETL logic.

Instead of deleting:

```id="y3v8rc"
Record Flagged Deleted
```

Power BI filters inactive rows.

Incremental refresh remains accurate.


## Strategy 5: Hybrid Partition Maintenance via XMLA

Advanced enterprise solution.

Process:

```id="c6m4pw"
Detect Delete Range
        ↓
Reprocess Specific Partition
```

XMLA automation selectively rebuilds partitions affected by deletes.

Maintains performance and accuracy.


## Strategy 6: Validation Reconciliation Checks

Create monitoring measures:

```id="b2z9ld"
Source Count vs Model Count
```

Detect divergence early.


## Partition Design Considerations

Smaller partitions reduce delete risk impact.

Example:

| Partition Size | Delete Impact |
| -------------- | ------------- |
| Yearly         | High          |
| Monthly        | Moderate      |
| Daily          | Low           |

Granularity improves correction flexibility.


## Incremental Refresh With Large Models

Large enterprise models often combine:

* Incremental refresh
* Aggregation tables
* XMLA partition processing

Hard delete handling becomes an operational responsibility rather than automatic engine behavior.


## Engine Perspective

VertiPaq treats partitions as immutable historical snapshots unless explicitly reprocessed.

```id="e7t2rq"
No Reprocessing
      ↓
No Delete Awareness
```

The engine assumes historical stability.


## Best Practices Summary

### Do's

✓ Prefer soft delete patterns
✓ Refresh wider historical windows periodically
✓ Schedule occasional full refreshes
✓ Monitor source vs model reconciliation
✓ Design smaller partitions when possible

### Don'ts

✗ Assume incremental refresh detects deletes
✗ Freeze historical partitions permanently
✗ Ignore reconciliation checks
✗ Depend solely on refresh success status
✗ Use incremental refresh on volatile history tables


## Summary

**Key Insight:**

Incremental refresh optimizes performance by avoiding historical reprocessing, but this optimization removes visibility into hard deletes.

Without corrective strategies, datasets gradually diverge from source truth while refresh operations continue to succeed.

Operationally resilient Power BI architectures combine incremental refresh with delete-aware design patterns, ensuring both scalability and long-term data accuracy.
