# Composite Model Consistency Debugging in Power BI

## Overview

Composite models introduce flexibility by combining:

* Import tables (VertiPaq)
* DirectQuery tables
* Dual storage dimensions
* Aggregation tables

However, this flexibility creates one of the most difficult challenges in advanced Power BI architecture: **result consistency debugging**.

The **Composite Model Consistency Problem** occurs when identical visuals, measures, or filters produce different results depending on execution path, cache state, or storage mode selection.

```id="n4x2qf"
User Query
     ↓
Formula Engine Plan
     ↓
Import Path     DirectQuery Path
     ↓                ↓
Different Execution Logic
     ↓
Result Inconsistency
```

The difficulty lies in the fact that the model appears structurally correct while results vary subtly.


## Why Consistency Problems Exist

Composite models operate across **multiple execution engines**.

Each engine has different characteristics:

| Engine             | Behavior               |
| ------------------ | ---------------------- |
| VertiPaq (Import)  | In-memory evaluation   |
| DirectQuery Source | External SQL execution |
| Dual Mode          | Dynamic switching      |
| Aggregation Tables | Query redirection      |

The Formula Engine dynamically decides where queries execute.

Consistency issues arise when execution decisions change.


## The Execution Switching Problem

### Dynamic Storage Resolution

Power BI evaluates queries using rules such as:

```id="a7u5zm"
Can aggregation answer query?
        ↓
Yes → Import table used
No  → DirectQuery executed
```

Two visuals using the same measure may follow different paths.

Result differences can emerge from:

* Data latency
* NULL handling
* Relationship evaluation
* Source precision differences


## Common Consistency Conflict Areas

### 1. Import vs DirectQuery Data Latency

Import tables refresh periodically.
DirectQuery tables read live data.

```id="y3k1pb"
Import Data → Yesterday
DirectQuery → Current Data
```

Totals no longer reconcile.

This is the most common composite model issue.


### 2. Aggregation Table Redirection

Aggregation tables intercept queries.

```id="s2m8kt"
Query
  ↓
Aggregation Hit?
  ↓
Yes → Import Aggregation
No  → DirectQuery Detail
```

If aggregation granularity differs slightly, results diverge.


### 3. Dual Table Mode Switching

Dual tables behave differently depending on context.

```id="d5v7qn"
Small Filter → Import Cache
Large Scan → DirectQuery
```

Filter propagation logic changes between executions.


### 4. NULL vs BLANK Semantics

Import mode:

```id="f8z3rc"
BLANK = Comparable member
```

DirectQuery mode:

```id="r6k2ae"
NULL ≠ NULL
```

Join results may differ silently.


### 5. Source Calculation Differences

External systems may handle:

* Rounding
* Data types
* Aggregation precision

differently than VertiPaq.

Example:

```id="x0j6up"
Decimal precision mismatch
```

Small discrepancies accumulate in totals.


## Debugging Methodology

Consistency debugging requires isolating execution paths.

### Step 1: Identify Storage Mode Used

Use Performance Analyzer.

Look for:

* DirectQuery events
* VertiPaq scans
* Aggregation hits

Goal:
Determine where computation occurred.


### Step 2: Force Execution Isolation

Temporarily test measures.

Import-only test:

```dax id="u9n4tc"
Test Import :=
CALCULATE ( [Measure], REMOVEFILTERS() )
```

DirectQuery-only testing may involve disabling aggregation tables.

Compare outputs.


### Step 3: Validate Aggregation Mapping

Check:

* GroupBy columns
* Summarization behavior
* Relationship alignment

Even minor mismatches break consistency.


### Step 4: Validate Relationship Paths

Composite models often introduce hidden ambiguity.

Check for:

* Multiple filter paths
* Many-to-many relationships
* Nullable keys

Relationship differences may appear only under DirectQuery execution.


### Step 5: Compare Row-Level Results

Create validation visuals:

```id="e2p1lm"
Detail Table View
```

Compare Import vs DirectQuery outputs at lowest grain.

Totals hide inconsistencies.


## Diagnostic Patterns

### Pattern 1: Aggregation Drift

Symptoms:

* Detail correct
* Total incorrect

Cause:
Aggregation table mismatch.


### Pattern 2: Mode Switching Drift

Symptoms:

* Result changes after slicer interaction.

Cause:
Dual table switching storage mode.


### Pattern 3: Latency Drift

Symptoms:

* Numbers differ after refresh.

Cause:
Import snapshot vs live DirectQuery data.


### Pattern 4: Join Drift

Symptoms:

* Missing rows or unexpected blanks.

Cause:
NULL equality differences.


## Recommended Debug Toolkit

### Tools

* Performance Analyzer
* DAX Studio Server Timings
* Query Diagnostics
* VertiPaq Analyzer
* Native Query inspection

Key debugging question:

```id="t4h6xr"
Where did this query actually run?
```

Not:

Is the DAX correct?


## Prevention Strategies

### Strategy 1: Define Source Authority

Decide which storage mode owns truth.

Example:

```id="k3p8ow"
Financial Metrics → Import
Operational Metrics → DirectQuery
```

Avoid mixed ownership.


### Strategy 2: Align Refresh Windows

Reduce timing gaps between Import and live sources.


### Strategy 3: Standardize NULL Handling

Replace NULL keys upstream with explicit unknown members.


### Strategy 4: Carefully Design Aggregations

Aggregation tables must exactly match:

* Dimensions
* Granularity
* Relationships


### Strategy 5: Limit Dual Tables

Dual mode improves performance but increases unpredictability.

Use selectively.


## Engine Paradox Explained

| Design Goal              | Side Effect                 |
| ------------------------ | --------------------------- |
| Flexibility              | Multiple execution paths    |
| Performance optimization | Dynamic routing             |
| Hybrid storage           | Semantic inconsistency risk |

Composite models optimize execution dynamically, but consistency assumes deterministic execution.


## Mental Model

Composite models operate like:

```id="z1q5dc"
Adaptive Query System
```

The engine continuously changes strategy, and debugging requires tracking those decisions.


## Best Practices Summary

### Do's

✓ Verify execution location first
✓ Align aggregation granularity precisely
✓ Monitor Dual table behavior
✓ Standardize NULL handling
✓ Compare detail-level outputs

### Don'ts

✗ Assume identical execution paths
✗ Mix live and snapshot data casually
✗ Ignore aggregation redirection
✗ Debug only at total level
✗ Treat composite models like Import models


## Summary

**Key Insight:**

Composite model issues are rarely caused by incorrect DAX. They arise from **execution path variability** across storage engines.

Consistency debugging focuses on identifying where calculations occur, validating aggregation routing, and ensuring semantic alignment between Import and DirectQuery behavior.

Reliable composite architectures minimize ambiguity, clearly define data ownership, and reduce situations where the engine must dynamically choose between competing execution strategies.
