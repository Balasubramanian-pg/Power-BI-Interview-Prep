# The Memory Cliff: Low Cardinality Columns in Power BI

## Overview

Low cardinality columns are usually considered ideal for VertiPaq compression. Fewer unique values typically mean smaller memory usage and faster scans.

However, under certain modeling conditions, Power BI hits what is known as the **Memory Cliff**. A column that appears harmless suddenly causes a disproportionate increase in memory consumption and query cost.

This paradox occurs because VertiPaq optimization depends not only on **cardinality**, but also on **distribution, relationships, and filter interaction**.

```id="7qv2ae"
Low Cardinality Column
        ↓
Dictionary Encoding Efficient
        ↓
Model Growth / Relationships Added
        ↓
Encoding Strategy Changes
        ↓
Sudden Memory Expansion (Memory Cliff)
```


## Understanding Cardinality

### What Is Cardinality?

Cardinality represents the number of unique values in a column.

Example:

| Column     | Rows | Unique Values | Cardinality |
| ---------- | ---- | ------------- | ----------- |
| Gender     | 10M  | 2             | Very Low    |
| Country    | 10M  | 120           | Low         |
| CustomerID | 10M  | 10M           | High        |

VertiPaq normally compresses **low cardinality columns extremely well**.

Expected assumption:

Low cardinality → Less memory

The Memory Cliff breaks this assumption.


## How VertiPaq Stores Columns

VertiPaq uses two main structures:

### 1. Dictionary

Stores unique values once.

```id="bb2r9s"
Dictionary
-----------
0 → Active
1 → Inactive
```

### 2. Data Encoding (Value IDs)

Fact rows store references instead of text values.

```id="zly8xq"
Row Storage
-----------
0,0,1,0,1,1,0
```

Compression efficiency depends on encoding strategy.


## Encoding Strategies

VertiPaq dynamically selects encoding.

### Value Encoding (Best Case)

Stores small integer references.

Characteristics:

* Extremely compact
* Fast scans
* Minimal memory footprint

Used when:

* Column distribution is stable
* Numeric mapping is efficient


### Hash Encoding

Used when value encoding becomes inefficient.

Characteristics:

* Larger memory usage
* Additional lookup cost
* Reduced compression efficiency

The Memory Cliff typically appears when VertiPaq switches encoding strategy.


## What Creates the Memory Cliff

### Trigger 1: Column Used in Relationships

A low cardinality column added as:

* Relationship key
* Bridge table column
* Many-to-many join participant

causes additional structures:

```id="u9tqef"
Column
 + Relationship Index
 + Join Mapping
 + Filter Bitmap
```

Memory usage increases non-linearly.


### Trigger 2: Extremely Large Fact Tables

Example:

| Column | Unique Values |
| ------ | ------------- |
| Status | 5             |

Seems harmless.

But across:

100 million rows

Filter bitmaps must be maintained across segments, increasing memory overhead significantly.


### Trigger 3: Skewed Value Distribution

Example:

| Status | Distribution |
| ------ | ------------ |
| Active | 99.5%        |
| Closed | 0.5%         |

VertiPaq may lose compression efficiency because segment optimization weakens.

Result:

* Larger internal structures
* Poor segment elimination


### Trigger 4: Auto-Exist Expansion

When multiple low-cardinality columns interact:

```id="k1p0cu"
Region × Status × Channel
```

The engine evaluates combinations.

Even with small individual cardinality, combinations explode logically.

This increases:

* Temporary memory
* FE materialization
* Query spill risk


## The Engine Paradox

Expected logic:

```id="n0o8mf"
Lower Cardinality → Smaller Model
```

Actual behavior near the cliff:

```id="k6rk5r"
Low Cardinality
      +
Large Row Count
      +
Relationships / Filters
      ↓
Bitmap Explosion
      ↓
Memory Spike
```

Compression stops scaling linearly.


## Real Example Scenario

### Model Design

Fact table:

* 250M rows
* Column: `OrderStatus`
* Values: Pending, Shipped, Cancelled, Returned

Initially:

Memory usage minimal.

Later changes:

* Added slicers
* Added relationship usage
* Used heavily in CALCULATE filters

Outcome:

Model memory increased significantly despite unchanged cardinality.

Cause:

Additional filter propagation structures.


## Detecting the Memory Cliff

### Using VertiPaq Analyzer

Watch for:

| Indicator                    | Warning Sign                        |
| ---------------------------- | ----------------------------------- |
| Column size disproportionate | Low distinct count but large memory |
| High hierarchy size          | Unexpected growth                   |
| Relationship size increase   | Index overhead                      |
| Segment memory imbalance     | Encoding shift                      |


### DMV Inspection

Key DMV:

```sql id="bkh0ty"
DISCOVER_STORAGE_TABLE_COLUMNS
```

Look for:

* Encoding type changes
* Large column data size
* Unexpected dictionary growth


## Query Symptoms

Models near the memory cliff show:

* Sudden dataset size growth
* Slower refresh times
* Increased FE materialization
* Memory pressure during visuals
* Capacity eviction (Premium environments)


## Optimization Strategies

### Strategy 1: Avoid Relationship Keys on Low-Value Columns

Bad practice:

```id="k6goe3"
Status → Relationship Key
```

Better:

Use surrogate numeric keys instead.


### Strategy 2: Convert to Dimension Tables

Instead of storing repeatedly:

```id="h2y31c"
Fact[Status]
```

Create:

```id="ck4vl6"
DimStatus
```

Benefits:

* Better compression reuse
* Reduced bitmap duplication


### Strategy 3: Reduce Filter Usage Frequency

Avoid heavy filtering on extremely large fact columns.

Prefer filtering via dimensions.


### Strategy 4: Remove Unnecessary Columns

Even small columns multiply cost across billions of rows.

Rule:

Unused low-cardinality columns still consume segment memory.


### Strategy 5: Partition Large Tables

Partitioning limits bitmap scope.

Benefits:

* Smaller segment scans
* Reduced memory spikes
* Improved refresh scalability


## When Low Cardinality Is Actually Ideal

Low cardinality columns perform best when:

* Used as attributes, not keys
* Stored in dimension tables
* Not heavily involved in many-to-many logic
* Distribution reasonably balanced


## Best Practices Summary

### Do's

✓ Keep low-cardinality attributes in dimensions
✓ Monitor encoding type changes
✓ Use VertiPaq Analyzer regularly
✓ Partition very large fact tables
✓ Minimize relationship complexity

### Don'ts

✗ Use status-like columns as relationship keys
✗ Assume low distinct count equals low memory
✗ Ignore distribution skew
✗ Overcombine slicer attributes
✗ Leave unused columns in large facts


## Summary

**Key Insight:**

VertiPaq compression depends on more than cardinality.

When large row counts, relationships, and filter propagation intersect, low-cardinality columns can cross a tipping point where internal indexing and bitmap structures dominate memory usage.

This sudden transition is known as the **Memory Cliff**.

Understanding this behavior helps prevent unexpected dataset growth and ensures scalable Power BI semantic model architecture.
