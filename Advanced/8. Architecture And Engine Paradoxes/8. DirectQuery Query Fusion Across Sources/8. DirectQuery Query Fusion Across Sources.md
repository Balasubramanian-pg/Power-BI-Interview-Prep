# DirectQuery Query Fusion Across Sources in Power BI

## Overview

DirectQuery enables Power BI to query external systems in real time instead of importing data into VertiPaq. When multiple DirectQuery sources exist inside a single model, the engine attempts an advanced optimization known as **Query Fusion**.

Query Fusion tries to combine operations into fewer source queries to reduce latency and data transfer. However, when queries span **multiple data sources**, fusion becomes partially or completely impossible, creating an architectural paradox.

```id="q4m9xt"
Visual Query
      ↓
Formula Engine Plan
      ↓
Can Queries Be Fused?
      ↓
Yes → Single Source Execution
No  → Split Queries Across Sources
      ↓
Local Join + Materialization
```

Performance degradation often occurs even when each individual source performs well.


## What Is Query Fusion

Query Fusion is the process where Power BI merges multiple logical operations into a single native query sent to the data source.

Instead of:

```id="f1r8dp"
Query 1 → Source
Query 2 → Source
Query 3 → Source
```

The engine produces:

```id="g7t3kl"
Single Optimized Query → Source
```

Benefits include:

* Reduced network round trips
* Source-side aggregation
* Smaller data transfer
* Faster visual execution

Fusion works best within a **single DirectQuery source**.


## Single-Source Fusion (Ideal Scenario)

Example model:

```id="m2z6ax"
SQL Server
 ├─ Sales
 ├─ Product
 └─ Customer
```

DAX query:

```dax id="r8e1nw"
SUMMARIZECOLUMNS(
    Product[Category],
    "Sales", SUM(Sales[Amount])
)
```

Power BI generates one SQL query containing:

* Joins
* Filters
* Aggregations

Execution occurs entirely in the database engine.

Result:
Minimal Power BI processing.


## Cross-Source DirectQuery Architecture

Composite enterprise models often include:

```id="k9u4br"
SQL Server      Snowflake
    |               |
 Sales           Marketing
        \       /
         Dimension Tables
```

Now a visual references tables from both systems.

Fusion complexity increases immediately.


## Why Cross-Source Fusion Breaks

Databases cannot execute joins across independent systems without a federation layer.

Power BI therefore must:

```id="v6p1os"
Query Source A
Query Source B
        ↓
Transfer Results
        ↓
Join Inside Formula Engine
```

This process is called **query decomposition**.

The optimization reverses:

Source computation → Power BI computation.


## Engine Execution Flow

### Cross-Source Query Plan

```id="u3c5jn"
DAX Query
   ↓
Formula Engine
   ↓
Native Query A → Source A
Native Query B → Source B
   ↓
Intermediate Tables Returned
   ↓
FE Local Join
   ↓
Final Aggregation
```

Large intermediate datasets may be materialized in memory.


## The Fusion Paradox

Expected assumption:

```id="p1e7yx"
DirectQuery avoids memory usage
```

Actual behavior:

```id="z8h4tw"
Multiple Sources
      +
Cross Filtering
      ↓
Data Pulled Into Power BI
      ↓
Temporary In-Memory Processing
```

DirectQuery can behave partially like Import during execution.


## Performance Symptoms

Models affected by failed fusion show:

* Slow visuals despite powerful databases
* High network transfer time
* Long "Query End" gaps in Performance Analyzer
* High Formula Engine duration
* Sudden latency spikes

Typical pattern:

| Metric            | Observation |
| ----------------- | ----------- |
| Source Query Time | Moderate    |
| FE Time           | High        |
| Rows Returned     | Very Large  |


## Example Scenario

### Model

* Sales stored in SQL Server
* Campaign data stored in Snowflake
* Shared Customer dimension

Visual:

```id="x5j2dc"
Customer Segment vs Sales vs Campaign Spend
```

Execution:

1. Query SQL Server for sales.
2. Query Snowflake for campaign metrics.
3. Transfer datasets.
4. Perform join locally.

If each source returns millions of rows, performance collapses.


## Folding Boundary Concept

Query Fusion depends on **query folding boundaries**.

```id="a6v0ke"
DAX Logic
   ↓
Foldable Operations
   ↓
Source Execution
```

Cross-source joins create a folding boundary that cannot be crossed.

Anything beyond that boundary executes inside Power BI.


## Dual Tables and Partial Fusion

Dual storage dimensions sometimes help.

```id="y3l9fs"
Dual Dimension
   ↓
Cached Locally
```

If dimension filters resolve locally:

* Smaller queries sent to sources
* Reduced transferred rows

However, fact-to-fact joins across sources still prevent fusion.


## Cardinality Amplification Risk

When fusion fails:

```id="n4k2qm"
Large Source A Result
        ×
Large Source B Result
        ↓
Expensive Local Join
```

Intermediate table size grows rapidly.

This becomes a major scalability risk.


## Optimization Strategies

### Strategy 1: Source Consolidation (Best Solution)

Place frequently joined datasets in the same system whenever possible.

```id="h8t5zn"
Single Analytical Database
```

Enables full query fusion.


### Strategy 2: Aggregation Tables

Import aggregated summaries locally.

Benefits:

* Smaller DirectQuery requests
* Reduced cross-source joins
* Faster visuals


### Strategy 3: Dimension Replication

Replicate shared dimensions across sources.

Goal:
Allow filtering before cross-source interaction.


### Strategy 4: Reduce Cross-Source Measures

Avoid visuals combining metrics from unrelated systems unless necessary.

Design reports around source domains.


### Strategy 5: Use Data Virtualization Layer

Technologies such as:

* Fabric Warehouse
* Synapse
* Federated SQL engines

can perform joins before Power BI execution.


## Detecting Fusion Failure

### Performance Analyzer Indicators

Warning signs:

* Multiple DirectQuery events per visual
* Large row counts returned
* Long Formula Engine processing


### Native Query Inspection

If multiple native queries appear instead of one unified query, fusion did not occur.


## Engine Paradox Explained

| Scenario                  | Engine Behavior    |
| ------------------------- | ------------------ |
| Single Source DirectQuery | Query Fusion Works |
| Multiple Sources          | Fusion Breaks      |
| Fusion Breaks             | FE Performs Join   |
| FE Join                   | Performance Drops  |

DirectQuery scalability depends more on **data location alignment** than dataset size.


## Mental Model

Think of Query Fusion as:

```id="c1z7rb"
Push Work to the Database
```

Cross-source models force Power BI to pull work back instead.


## Best Practices Summary

### Do's

✓ Keep frequently joined data in one source
✓ Use aggregation tables strategically
✓ Replicate shared dimensions when needed
✓ Monitor folding boundaries
✓ Test visuals combining multiple systems

### Don'ts

✗ Assume DirectQuery scales automatically
✗ Join large facts across sources
✗ Ignore intermediate row counts
✗ Overuse cross-source visuals
✗ Depend on Power BI for distributed joins


## Summary

**Key Insight:**

DirectQuery Query Fusion works only within a single execution environment. When queries span multiple sources, Power BI cannot push computation downward and must instead materialize intermediate datasets locally.

The paradox is that DirectQuery models designed for scale can lose scalability when data is distributed across systems.

Effective architecture minimizes cross-source joins and aligns analytical workloads with the physical location of data.
