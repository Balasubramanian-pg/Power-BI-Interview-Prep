# The Null Equality Conflict in Composite Models in Power BI

## Overview

Composite Models combine multiple storage modes inside a single semantic model, typically:

* Import (VertiPaq)
* DirectQuery
* Dual tables

While this architecture enables flexibility and scale, it introduces a subtle engine inconsistency known as the **Null Equality Conflict**.

The paradox occurs because **Import models and DirectQuery sources treat NULL values differently during joins and filtering**, leading to inconsistent results depending on where execution occurs.

```id="p7k4z1"
Composite Model Query
        ↓
Formula Engine Decision
        ↓
Import Path        DirectQuery Path
(VertiPaq)         (Source System)
        ↓                 ↓
NULL Handling Differs
        ↓
Aggregation Mismatch
```

A report may return different results without any visible modeling error.


The Formula Engine dynamically decides execution location.


## Understanding NULL Semantics

### SQL NULL Behavior (DirectQuery)

In relational databases:

```sql id="a4n1tt"
NULL = NULL → UNKNOWN (not TRUE)
```

Meaning:

* NULL values do not match each other.
* Joins exclude NULL keys.
* Filters behave using three-valued logic.

Example:

```sql id="z6xw2e"
WHERE CustomerID = NULL
```

Returns no rows.

Correct SQL syntax:

```sql id="o8dy51"
WHERE CustomerID IS NULL
```


### VertiPaq BLANK Behavior (Import Mode)

Power BI Import models use **BLANK()**, not SQL NULL.

VertiPaq behavior:

```id="v2l5sp"
BLANK = BLANK → TRUE (engine equivalent member)
```

Implication:

* Blank values can match.
* Relationships may aggregate blanks together.
* Filters treat blanks as valid comparable members.


## Where the Conflict Appears

### Cross Storage Joins

Example composite model:

```id="t5q9rc"
Customer (Import)
Sales (DirectQuery)
Relationship → CustomerID
```

Scenario:

* Some CustomerID values are NULL in source.
* Import table contains BLANK rows.

Execution differences emerge depending on query folding.


### Engine Decision Path

```id="h1r3ku"
Query Requested
      ↓
Can Fold to Source?
      ↓
Yes → SQL NULL Logic
No  → VertiPaq BLANK Logic
```

Same visual can produce different row counts.


## Real Example Scenario

### Data Situation

Sales table (DirectQuery):

| CustomerID | Sales |
| ---------- | ----- |
| 101        | 200   |
| NULL       | 150   |

Customer table (Import):

| CustomerID |
| ---------- |
| 101        |
| BLANK      |


### Result Variation

If executed in SQL source:

NULL rows do not join.

Result:
Sales = 200

If executed in VertiPaq:

BLANK matches BLANK.

Result:
Sales = 350

Both results follow valid engine rules.

This creates analytical inconsistency.


## Why Composite Models Trigger This

Composite models introduce:

* Multiple evaluation engines
* Dynamic query routing
* Partial query folding
* Hybrid aggregations

NULL interpretation depends on execution location rather than model definition.


## Additional Conflict Areas

### Aggregations Tables

Aggregation tables stored in Import mode may treat blanks differently than DirectQuery detail tables.

Result:
Totals mismatch between aggregated and detailed views.


### Dual Storage Dimensions

Dual tables switch modes dynamically.

```id="s3g7wf"
Dual Table
   ↓
Import Mode (cache hit)
DirectQuery Mode (cache miss)
```

NULL equality rules change silently.


### Many-to-Many Relationships

NULL keys inside bridge tables increase inconsistency risk because matching behavior differs across engines.


## Symptoms of the Null Equality Conflict

Common indicators:

* Totals changing after slicer interaction
* Import vs DirectQuery visuals disagreeing
* Aggregation tables not reconciling
* Random appearing BLANK categories
* Drillthrough returning different counts

DAX logic appears correct while results vary.


## Detecting the Issue

### Validation Test

Create comparison measures:

```dax id="e6c1my"
Import Count :=
COUNTROWS ( Customer )
```

```dax id="d9v2qb"
DirectQuery Count :=
COUNTROWS ( Sales )
```

Unexpected mismatches involving blank members indicate possible NULL conflicts.


### Query Folding Check

Use Performance Analyzer:

* If query folds → SQL semantics apply.
* If not → VertiPaq semantics apply.

Execution location reveals behavior source.


## Optimization Strategies

### Strategy 1: Eliminate NULL Keys (Best Practice)

Handle NULL values upstream.

Example ETL fix:

```id="y2m8ka"
NULL → -1 (Unknown Member Key)
```

Create explicit unknown dimension member.

This standardizes behavior across engines.


### Strategy 2: Use Surrogate Keys

Avoid nullable relationship columns.

Preferred structure:

```id="k4w0ns"
CustomerKey (NOT NULL)
```

Prevents join ambiguity.


### Strategy 3: Normalize Blank Handling in Power Query

Example transformation:

```id="w7j1ax"
Replace null with "Unknown"
```

Ensures consistent equality evaluation.


### Strategy 4: Avoid Nullable Relationship Columns

Relationships involving nullable keys are primary risk zones in composite models.


### Strategy 5: Align Aggregation Storage Modes

Ensure aggregation tables and detail tables follow consistent blank-handling logic.

## Best Practices Summary

### Do's

✓ Replace NULL keys during ETL
✓ Create explicit unknown members
✓ Use surrogate non-null keys
✓ Test Import vs DirectQuery totals
✓ Monitor query folding behavior

### Don'ts

✗ Allow nullable relationship columns
✗ Assume BLANK equals SQL NULL
✗ Mix aggregation modes carelessly
✗ Ignore inconsistent totals
✗ Depend on engine execution location


## Summary

**Key Insight:**

Composite models unify multiple storage engines, but they do not unify NULL semantics.

The Null Equality Conflict occurs because VertiPaq treats BLANK values as comparable members while SQL engines treat NULL as non-equal values. When execution shifts between engines, identical DAX logic can produce different analytical outcomes.

Robust composite model design requires eliminating nullable relationship keys and enforcing consistent unknown-member handling across all storage modes.
