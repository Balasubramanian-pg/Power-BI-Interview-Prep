# DAX WINDOW Function: Partitioning and Blank Value Handling

## Overview

The `WINDOW` function returns a set of rows positioned within a specified interval. It's one of the window functions (along with `INDEX` and `OFFSET`) added to DAX for more sophisticated calculations.

```
WINDOW ( from[, from_type], to[, to_type][, <relation>][, <orderBy>][, <blanks>][, <partitionBy>][, <matchBy>] )
```

## Partitioning with PARTITIONBY

The `PARTITIONBY` clause divides the relation into logical groups. The window calculation restarts for each partition.

### Basic Partitioning Example

```dax
RunningSumByFiscalYear =
SUMX (
    WINDOW (
        1, ABS, 0, REL,
        ALLSELECTED (
            'Date'[Fiscal Year],
            'Date'[Month Number Of Year]
        ),
        PARTITIONBY ( 'Date'[Fiscal Year] )  // Restart for each fiscal year
    ),
    [Total Sales]
)
```

This calculates a running total that resets at the beginning of each fiscal year.

### Partitioning Behavior

- If `PARTITIONBY` is omitted, the entire relation is treated as a single partition
- All columns in `PARTITIONBY` must come from the relation or a related table
- Sorting (`ORDERBY`) is applied within each partition independently

## The Blanks Parameter

The `blanks` parameter controls how blank values are handled during sorting. As of recent updates, three options are supported:

| Value | Behavior |
|-------|----------|
| `DEFAULT` | Numeric: blanks between zero and negative values<br>String: blanks before all strings (including empty strings) |
| `FIRST` | Blanks always appear at the beginning, regardless of sort order |
| `LAST` | Blanks always appear at the end, regardless of sort order |

### How Blank Sorting Works

**Default behavior** (from Microsoft documentation):
- **Numeric values**: Blank values are ordered between zero and negative values
  - Order: ... -2, -1, (blank), 0, 1, 2 ...
- **String values**: Blank values are ordered before all strings, including empty strings
  - Order: (blank), "", "A", "B" ...

### Using the Blanks Parameter

```dax
SalesWithBlanksHandled = 
WINDOW(
    1, ABS, 10, ABS,
    ALL('Sales'),
    ORDERBY([SalesAmount]),
    LAST,  // Blanks appear at the end
    PARTITIONBY([ProductCategory])
)
```

### Priority Rules

When both the function-level `blanks` parameter and expression-level `blanks` in `ORDERBY()` are specified, the expression-level setting takes priority for that specific expression. Other expressions without explicit blank handling fall back to the function-level setting.

## Practical Example: Partitioning with Blank Handling

```dax
CategoryRanking = 
VAR CurrentCategory = SELECTEDVALUE('Product'[Category])
RETURN
CALCULATE(
    [Total Sales],
    WINDOW(
        -2, REL, 0, REL,
        ALL('Product'),
        ORDERBY([Total Sales], DESC, LAST),  // Blanks last within this sort
        PARTITIONBY('Product'[Category])      // Per-category windows
    )
)
```

## Key Points

1. **Partitions are independent** - Windows reset at partition boundaries
2. **Blank positioning matters** - Controls whether rows with missing data appear first or last
3. **DEFAULT is current default** - Though `FIRST` and `LAST` are now available for explicit control
4. **Performance consideration** - Partitioning reduces sort scope, can improve calculation speed

## Common Gotchas

- If `PARTITIONBY` columns can't uniquely identify rows with `ORDERBY`, DAX tries to add columns automatically - if it can't, you get an error
- Empty table returned if the corresponding outer value of a `PARTITIONBY` column doesn't exist in the relation
- The `blanks` parameter only affects sorting order, not calculation logic
