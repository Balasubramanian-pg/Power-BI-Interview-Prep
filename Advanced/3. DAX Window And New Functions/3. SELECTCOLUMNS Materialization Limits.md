# SELECTCOLUMNS Materialization Limits in Power BI

## Overview

`SELECTCOLUMNS` is a table function that returns a table with selected columns from an existing table plus new columns defined by DAX expressions . It's an iterator function that evaluates expressions in a row context .

```
SELECTCOLUMNS(<Table>, <Name>, <Expression>[, <Name>, <Expression>]…)
```

The function returns a table with the same number of rows as the input table, containing one column for each name-expression pair .

## Materialization Behavior

### What Gets Materialized

When `SELECTCOLUMNS` executes, it:
- Iterates through every row of the input table
- Evaluates each expression in a row context
- Returns only the explicitly defined columns (not all original columns) 

This makes it fundamentally different from `ADDCOLUMNS`, which returns all original columns plus new ones .

### Performance Implications

The key materialization consideration is **what data actually gets stored in memory during execution**.

**Test results from a 600,000-row, 15-column table** :
- `SELECTCOLUMNS` returning 2 columns: **3,563 ms**
- `ADDCOLUMNS` returning 15+1 columns: **167,730 ms**

The performance difference becomes dramatic when the source has many columns. `SELECTCOLUMNS` only materializes the columns you explicitly define, while `ADDCOLUMNS` materializes the entire source table plus new columns .

## DirectQuery Limitations

`SELECTCOLUMNS` in DirectQuery mode has a significant limitation:

> **Not supported when used in calculated columns or Row-Level Security (RLS) rules** under DirectQuery mode 

## RLS Performance Issues

Using `SELECTCOLUMNS` in RLS filters can cause severe performance degradation because:

```dax
// BAD: Filter evaluated for every row in Projects table
Projects table :=
[Key_Project] IN
SELECTCOLUMNS(
    FILTER(
        'Engagement Role',
        'Engagement Role'[UserName] = USERNAME()
    ),
    "Key_project",
    'Engagement Role'[Key_project]
)
```

The `SELECTCOLUMNS` expression is evaluated for every row of the filtered table, completely bypassing the columnar storage advantages .

**Better approach**: Filter the table that contains the matching column and let relationships propagate the filter .

## Row Context and Context Transition

`SELECTCOLUMNS` creates a row context for expression evaluation. When you need to perform aggregations inside `SELECTCOLUMNS`, you must explicitly trigger context transition with `CALCULATE` :

```dax
// Wrong - SUM doesn't work in row context
SELECTCOLUMNS(
    Sales,
    "Customer", Sales[Customer],
    "Total", SUM(Sales[Amount])  // Won't work
)

// Correct - CALCULATE triggers context transition
SELECTCOLUMNS(
    Sales,
    "Customer", Sales[Customer],
    "Total", CALCULATE(SUM(Sales[Amount]))
)
```

## Data Lineage

`SELECTCOLUMNS` preserves data lineage for simple column references :
- **Simple column references**: `Sales[Customer]` → lineage preserved
- **Any expression**: `[Customer] & " - " & [Region]` → lineage broken

This affects whether the resulting column can automatically filter related tables.

## Practical Materialization Limits

### Memory Considerations
- Each column added to `SELECTCOLUMNS` increases the materialized table size
- The function always returns a fully materialized table (not a reference)
- For very large tables, materializing too many columns can exhaust memory

### Row Count Impact
Unlike `SUMMARIZE`, `SELECTCOLUMNS` never reduces row count through grouping - it always returns the same number of rows as the input table . This means:
- If your input has 10M rows, the result has 10M rows
- Filtering must be done before `SELECTCOLUMNS` if row reduction is needed

## Optimization Patterns

### Pattern 1: Filter First, Then Select
```dax
// More efficient - reduces rows before materialization
SELECTCOLUMNS(
    FILTER(Sales, Sales[Date] >= DATE(2024,1,1)),
    "OrderID", Sales[OrderID],
    "Amount", Sales[Amount]
)
```

### Pattern 2: Minimal Column Selection
```dax
// Good - only materializes needed columns
VAR RequiredData = 
    SELECTCOLUMNS(
        LargeTable,
        "ID", [RowID],
        "Value", [Metric]
    )
```

### Pattern 3: Avoid in RLS When Possible
```dax
// Instead of SELECTCOLUMNS in RLS
// Use relationship-based filtering
[EmployeeID] IN VALUES('Employee'[EmployeeID])
```

## Summary

**Materialization limits of SELECTCOLUMNS**:

1. **Memory footprint** scales with number of columns × number of rows
2. **Performance degrades** when source has many columns but you only need few
3. **Row context** requires `CALCULATE` for aggregations
4. **DirectQuery** has restrictions (no calculated columns/RLS)
5. **RLS usage** can kill performance if misapplied
6. **Lineage preservation** only for simple column references

**When to use**: When you need a subset of columns from a large table and want to minimize materialized data. Avoid when you need grouping/aggregation (use `SUMMARIZE` or `GROUPBY` instead) .
