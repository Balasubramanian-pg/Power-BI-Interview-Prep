# NATURALINNERJOIN Memory Spikes in Power BI

## Overview

`NATURALINNERJOIN` is a DAX function that performs an inner join between two tables based on columns with the same names .

```
NATURALINNERJOIN(<LeftTable>, <RightTable>)
```

Returns a table with:
- Common columns from the left table
- Other columns from both tables
- Only rows where values exist in both tables 

## Why NATURALINNERJOIN Causes Memory Spikes

### 1. Full Table Materialization

`NATURALINNERJOIN` materializes **both input tables completely in memory** before performing the join . Unlike relationships that leverage the VertiPaq engine's optimized storage, this function forces the formula engine to create physical copies of both tables.

### 2. Column Explosion

When applied to wide tables, the result set contains **all columns from both tables** . A 300-column table joined with another 300-column table produces up to 600 columns in the result .

### 3. Row Multiplication

Inner joins can produce Cartesian products if the join columns aren't unique . A 3-billion row table joined to another large table can generate intermediate results exceeding available memory.

## DirectQuery Limitations

In DirectQuery mode, `NATURALINNERJOIN` has a hard limit: **result sets cannot exceed 1 million rows** . This limit exists because:

- The formula engine must materialize results locally
- Network transfer of massive result sets would be impractical
- The storage engine pushes what it can to the source, but the join itself happens in Power BI 

> "The resultset of a query to external data source has exceeded the maximum allowed size of 1000000 rows" 

## Measured Performance Impact

According to benchmarks (July 2025) :

| Metric | INTERSECT | NATURALINNERJOIN |
|--------|-----------|-------------------|
| Execution Time (10M rows) | 2.3s | 0.7s (when optimized) |
| Memory Usage | 156MB | 98MB (37% reduction) |
| Query Plan Operations | 15 | 8 (47% simpler) |

The memory spike occurs when **unoptimized**—joining full tables instead of selected columns.

## Root Causes of Memory Spikes

### Cause 1: Joining Entire Tables

```dax
// DANGEROUS - Materializes ALL columns
NATURALINNERJOIN(
    FactSales,        // 50 columns, 100M rows
    DimProduct        // 30 columns, 100K rows
)
```

### Cause 2: Missing Relationships

If tables lack relationships, `NATURALINNERJOIN` must scan both tables to find matches, dramatically increasing memory pressure .

### Cause 3: Data Type Mismatches

Even with same-named columns, strict comparison semantics mean type mismatches force in-memory conversion :

> "Strict comparison semantics are used during join. There is no type coercion; for example, 1 does not equal 1.0." 

### Cause 4: Data Lineage Conflicts

Columns must have matching data lineage. Derived columns (from `SUMMARIZE`, `ADDCOLUMNS`) break lineage, causing join failures or inefficient fallback mechanisms .

## Optimization Patterns

### Pattern 1: SELECTCOLUMNS Before Joining

```dax
// OPTIMIZED - Only materialize needed columns
OptimizedJoin =
VAR LeftReduced =
    SELECTCOLUMNS(
        FactSales,
        "ProductKey", FactSales[ProductKey],
        "SalesAmount", FactSales[Amount],
        "Quantity", FactSales[Qty]
    )
VAR RightReduced =
    SELECTCOLUMNS(
        DimProduct,
        "ProductKey", DimProduct[ProductKey],
        "ProductName", DimProduct[Name],
        "Category", DimProduct[Category]
    )
RETURN
    NATURALINNERJOIN(LeftReduced, RightReduced)
```

This reduces memory from 80 columns to 5 .

### Pattern 2: Use Relationships Instead

```dax
// BETTER - Leverage existing relationships
SalesWithProduct =
CALCULATETABLE(
    SUMMARIZE(
        FactSales,
        FactSales[ProductKey],
        "SalesAmount", SUM(FactSales[Amount])
    ),
    DimProduct[Category] = "Electronics"
)
// No explicit JOIN needed - relationship handles it
```

### Pattern 3: TREATAS for Virtual Relationships

```dax
// When no relationship exists, use TREATAS
FilteredSales =
CALCULATE(
    [Total Sales],
    TREATAS(
        VALUES('FilterTable'[ProductID]),
        'Product'[ProductID]
    )
)
```

This performs filtering without materializing a joined table .

### Pattern 4: Force Data Type Alignment

```dax
// Fix type mismatches with CONVERT
AlignedJoin =
VAR LeftTable =
    SELECTCOLUMNS(
        TableA,
        "JoinKey", CONVERT(TableA[KeyColumn], STRING),
        "ValueA", TableA[Value]
    )
VAR RightTable =
    SELECTCOLUMNS(
        TableB,
        "JoinKey", CONVERT(TableB[KeyColumn], STRING),
        "ValueB", TableB[Value]
    )
RETURN
    NATURALINNERJOIN(LeftTable, RightTable)
```

This resolved a reported performance issue where identical-looking columns had different internal data types .

## Memory Spike Symptoms

When a memory spike occurs, you'll see:

1. **Power BI Desktop**: Query takes 20-30+ minutes 
2. **Power BI Service**: "Resources exceeded" errors 
3. **DirectQuery**: "Exceeded maximum allowed size of 1000000 rows" 
4. **DAX Studio**: High memory usage in Query Plan > 500MB+

## Diagnostic Query

```dax
// Check for potential memory issues
EVALUATE
SUMMARIZE(
    NATURALINNERJOIN(
        FactSales,
        DimProduct
    ),
    "RowCount", COUNTROWS(FactSales),
    "EstimatedJoinRows", 
        COUNTROWS(FactSales) * 
        COUNTROWS(DimProduct) / 
        DISTINCTCOUNT(FactSales[ProductKey])
)
```

If `EstimatedJoinRows` exceeds 10 million, expect memory pressure.

## Decision Framework

```
Need to combine two tables?
        ↓
Can you use existing relationships?
    ↓                       ↓
   Yes                      No
    ↓                       ↓
Use CALCULATE,          Must join in DAX?
SUMMARIZE with               ↓
filter propagation     Use SELECTCOLUMNS first
    ↓                       ↓
  Fast                 Reduce to minimum columns
    ↓                       ↓
  Done                 Consider TREATAS instead
```

## Best Practices Summary

### Do's
✓ **Reduce columns first** - Use `SELECTCOLUMNS` to limit input 
✓ **Filter before join** - Apply filters to reduce rows
✓ **Check data types** - Ensure join columns match exactly 
✓ **Use relationships** - Let the engine optimize joins 
✓ **Test with subsets** - Validate with 1% of data first

### Don'ts
✗ **Don't join wide tables** - Materializes all columns
✗ **Don't use in DirectQuery without filters** - 1M row limit 
✗ **Don't ignore lineage** - Derived columns break joins 
✗ **Don't assume uniqueness** - Check join column cardinality 
✗ **Don't use in RLS** - Not supported in DirectQuery RLS 

## When NATURALINNERJOIN Is Appropriate

Use `NATURALINNERJOIN` when:

1. **Creating calculated tables** during import (not at query time)
2. **Working with small lookup tables** (< 10K rows)
3. **Building static dimension tables** in Power Query (better there)
4. **Prototyping** before optimizing with relationships

Otherwise, prefer relationship-based approaches or `TREATAS` for dynamic filtering .
