# DETAILROWS and Dynamic Security in Power BI

## Overview

`DETAILROWS` is a DAX function that returns a table of detail rows for a given measure. When combined with dynamic security patterns, it can be used to control access to detailed data while maintaining aggregate visibility.

```
DETAILROWS ( [<Measure>] )
```

- Returns the detail rows that contribute to a measure's value
- When no measure is specified, returns details for the current context
- Useful for drill-through scenarios and granular access control

## How DETAILROWS Works with Security Context

The critical behavior of `DETAILROWS` is that it **respects all security filters** that are active in the current context. This means:

- Row-Level Security (RLS) rules are applied
- User context from `USERPRINCIPALNAME()` is honored
- Table relationships and cross-filtering directions affect the result

### Basic Example

```dax
Total Sales = SUM(Sales[Amount])

Sales Details = 
DETAILROWS([Total Sales])
```

When a user with restricted access views the `Sales Details` measure, they only see rows they're permitted to access based on active RLS rules.

## Dynamic Security Pattern with DETAILROWS

### Scenario: Regional Sales Managers

A common pattern involves giving managers access to aggregated data but restricting access to detailed transaction records.

```dax
// Regional Sales Measure - always visible
Regional Sales = 
CALCULATE(
    SUM(Sales[Amount]),
    'Sales Territory'[Region] = SELECTEDVALUE('User Region'[Region])
)

// Drill-through Details - security applied
Regional Sales Details = 
VAR HasAccess = 
    CALCULATE(
        COUNTROWS('User Region'),
        'User Region'[UserPrincipalName] = USERPRINCIPALNAME(),
        'User Region'[Region] = SELECTEDVALUE('Sales Territory'[Region])
    ) > 0
RETURN
    IF(
        HasAccess,
        DETAILROWS([Regional Sales]),
        ROW("No Access", BLANK())
    )
```

### Security Table Structure

```
User Region Table:
| UserPrincipalName       | Region        | AccessLevel |
|-------------------------|---------------|-------------|
| manager1@company.com    | North America | Full        |
| manager1@company.com    | Europe        | Aggregate   |
| manager2@company.com    | Asia          | Full        |

Sales Territory Table:
| Region        | Territory | SalesRep     |
|---------------|-----------|--------------|
| North America | USA       | rep1@co.com  |
| North America | Canada    | rep2@co.com  |
| Europe        | UK        | rep3@co.com  |
```

## Controlling DETAILROWS Access Based on User

### Pattern 1: Two-Level Security

```dax
// Role definition for Sales Managers
[UserRegion] = USERPRINCIPALNAME()
```

Combined with bi-directional cross-filtering between User Region and Sales Territory tables .

### Pattern 2: Conditional Detail Access

```dax
Measure with Conditional Details =
VAR CurrentUser = USERPRINCIPALNAME()
VAR UserAccessLevel =
    CALCULATE(
        VALUES('User Access'[AccessLevel]),
        'User Access'[UserPrincipalName] = CurrentUser
    )
VAR SummaryValue = [Total Sales]
RETURN
    IF(
        UserAccessLevel = "Full",
        SummaryValue,  // Users see aggregate
        IF(
            ISDRILLDOWNFILTERED(),  // Check if drill-down is active
            "Details Restricted",     // Block drill-through
            SummaryValue              // Allow aggregate view
        )
    )
```

## Performance Considerations

### Materialization Impact

`DETAILROWS` can materialize large tables when invoked. Combined with dynamic security:

- Security filters are evaluated **for each detail row**
- Complex security logic can slow down drill-through operations
- Consider using `SUMMARIZE` or `TOPN` to limit returned rows

```dax
Optimized Details =
VAR HasAccess = [CheckUserAccess]  // Pre-calculated security check
RETURN
    IF(
        HasAccess,
        TOPN(1000, DETAILROWS([Measure]), [TransactionDate], DESC),  // Limit results
        ROW("Message", "Contact administrator for access")
    )
```

### Relationship Design for Performance

For optimal performance with `DETAILROWS` and dynamic security:

1. **Use bi-directional filters** sparingly - they increase cardinality
2. **Pre-calculate user mappings** in Power Query rather than DAX
3. **Star schema design** - keep security tables as dimensions 

```dax
// Efficient - pre-joined in Power Query
User Access Combined = 
    SELECTCOLUMNS(
        'User Region',
        "User", [UserPrincipalName],
        "Region", [Region],
        "HasFullAccess", [AccessLevel] = "Full"
    )
```

## Common Patterns and Anti-Patterns

### Good Pattern: Bridge Table Security

```
User Table → UserRegion Bridge → Region Table → Sales Table
   ↑              ↑                  ↑
[USERNAME()]  [RLS Filter]    [Bi-directional?]
```

Set RLS on User Table only, let relationships propagate .

### Bad Pattern: Complex DAX in DETAILROWS

```dax
// AVOID - Security logic in DETAILROWS
PoorPerformance =
DETAILROWS(
    CALCULATE(
        [Sales],
        FILTER(
            ALL('User Region'),
            'User Region'[User] = USERPRINCIPALNAME() &&
            'User Region'[Region] = SELECTEDVALUE('Sales'[Region])
        )
    )
)
```

This evaluates security for every row in DETAILROWS.

### Good Pattern: Pre-Filtered Details

```dax
// PREFERRED - Security via relationships
EfficientDetails =
VAR AllowedRegions =
    CALCULATETABLE(
        VALUES('Region'[Region]),
        'User Region'[User] = USERPRINCIPALNAME()
    )
RETURN
    CALCULATE(
        DETAILROWS([Sales]),
        'Region'[Region] IN AllowedRegions
    )
```

## Testing DETAILROWS Security

### Desktop Testing

```dax
// Test measure to verify security
Test User Access = 
VAR CurrentUser = USERPRINCIPALNAME()
VAR AllowedRows = COUNTROWS(DETAILROWS([Sales]))
VAR RestrictedRows = 
    COUNTROWS(
        CALCULATETABLE(
            DETAILROWS([Sales]),
            REMOVEFILTERS('User Region')
        )
    )
RETURN
    "User: " & CurrentUser & 
    " Allowed: " & AllowedRows & 
    " Total: " & RestrictedRows
```

### Service Validation
- Use "View as" role testing in Power BI Desktop 
- Verify drill-through behavior for different user personas
- Test with and without RLS roles applied

## DirectQuery Considerations

In DirectQuery mode, `DETAILROWS` with dynamic security has additional limitations:

- `USERPRINCIPALNAME()` is supported
- Complex DAX in RLS may not work - use relationship-based security instead 
- Bi-directional cross-filtering becomes critical for security propagation

```dax
// DirectQuery-safe pattern
DirectQuery Details =
VAR CurrentUser = USERPRINCIPALNAME()
RETURN
    CALCULATE(
        DETAILROWS([Sales]),
        'User Region'[Email] = CurrentUser,
        CROSSFILTER('User Region'[Region], 'Sales'[Region], BOTH)
    )
```

## Summary

**Key Points for DETAILROWS Dynamic Security:**

1. **Respects existing RLS** - No need to re-implement security in DETAILROWS calls
2. **Performance matters** - Security evaluation multiplies by detail row count
3. **Relationships over DAX** - Use bi-directional filters instead of complex FILTER expressions
4. **Test thoroughly** - Verify both aggregate and detail access levels
5. **Limit result sets** - Use TOPN or filters to prevent large materialization

**Best Practice Checklist:**
- [ ] Security table connected with proper relationship direction
- [ ] RLS applied at highest level (User table, not fact table)
- [ ] DETAILROWS calls include row limits for large tables
- [ ] Tested with multiple user personas
- [ ] Performance validated under load
- [ ] Drill-through behavior verified
