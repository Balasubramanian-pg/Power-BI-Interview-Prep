# DAX OFFSET Function and Order Stability

## Overview

The `OFFSET` function is a window function that retrieves a value from a row at a specified number of steps (delta) from the current row within a table or partition.

```
OFFSET ( <delta> [, <relation> [, <orderBy> [, <blanks> [, <partitionBy> [, <matchBy> ] ] ] ] ] )
```

- **`delta`**: Number of rows to move (positive = forward, negative = backward)
- **`relation`**: Table to operate on (usually `ALL()` or `ALLSELECTED()`)
- **`orderBy`**: Defines the sort order - **required for deterministic results**

## The Order Stability Problem

Tables in DAX (like SQL tables) are unordered sets. Without explicit ordering, there's no guarantee which row is "+1" or "-1".

### Unstable Order Example

```dax
Sales Previous Row = 
CALCULATE(
    SUM(Sales[Amount]),
    OFFSET(
        -1,
        ALL(Sales[Date], Sales[Product]),
        ORDERBY([Date])  // Problem: Duplicate dates exist!
    )
)
```

With this data:
| Date | Product | Amount |
|------|---------|--------|
| 2024-01-01 | A | 100 |
| 2024-01-01 | B | 150 | ← Tie on date
| 2024-01-02 | A | 200 |

For row `2024-01-01, B`, the previous row could be:
- `2024-01-01, A` (if A sorts before B)
- Blank (if B is considered first within the date)
- Result depends on physical storage order

## Achieving Stable Order

Add columns to break ties until the combination is unique:

```dax
Sales Previous Row Stable = 
CALCULATE(
    SUM(Sales[Amount]),
    OFFSET(
        -1,
        ALL(Sales[Date], Sales[Product]),
        ORDERBY([Date], [Product])  // Product breaks the tie
    )
)
```

**Best Practice:** Always ensure the `ORDERBY` columns uniquely identify each row. Include a unique index/ID as the last sort column if necessary.

## Key Parameters

### ORDERBY()
Defines sort order. Multiple columns are evaluated left to right.

### Relation
Typically uses `ALL()` or `ALLSELECTED()` to remove filter context and allow access to rows outside current filters.

### Blanks
Controls blank handling in sort:
- `BLANKS` (default)
- `FIRST`
- `LAST`

## Common Patterns

### Previous Row (Time Intelligence Alternative)
```dax
Sales Previous Month = 
CALCULATE(
    [Total Sales],
    OFFSET(
        -1,
        ALL('Date'[Date]),
        ORDERBY([Date])
    )
)
```

### Moving Average (3-period)
```dax
Sales MA3 = 
VAR CurrentSales = [Total Sales]
VAR Prev1 = CALCULATE([Total Sales], OFFSET(-1, ALL('Date'[Date]), ORDERBY([Date])))
VAR Prev2 = CALCULATE([Total Sales], OFFSET(-2, ALL('Date'[Date]), ORDERBY([Date])))
RETURN
    AVERAGEX({CurrentSales, Prev1, Prev2}, [Value])
```

## Performance Notes

- Requires sort operation behind the scenes
- Performance degrades with very large tables
- High cardinality columns in `ORDERBY` generally perform better for sorting
- Consider using with proper data model design

## The Golden Rule

>[!Note]
> If the combination of columns in `ORDERBY` doesn't uniquely identify each row, the `OFFSET` result is non-deterministic.

Always include enough columns to break ties. When in doubt, add a unique identifier as the last sort column.
