# COMBINEVALUES and Hash Collision Risk in Power BI

## Overview

`COMBINEVALUES` is a DAX function that concatenates two or more strings using a specified delimiter. It's primarily designed for creating relationship columns in composite models and DirectQuery scenarios.

```
COMBINEVALUES(<delimiter>, <expression>, <expression>[, <expression>]…)
```

- Returns a concatenated string of all expressions separated by the delimiter
- **Handles BLANK values** - automatically filters them out
- **Used internally by Power BI** for relationships in composite models

## How COMBINEVALUES Works

### Basic Usage

```dax
Customer Key = COMBINEVALUES("-", Customer[CustomerID], Customer[Country])
```

Results:
| CustomerID | Country | Customer Key |
|------------|---------|--------------|
| 1001 | USA | 1001-USA |
| 1002 | | 1002 |  // BLANK country is omitted
| 1003 | Canada | 1003-Canada |

### Automatic BLANK Handling

Unlike simple concatenation with `&`, `COMBINEVALUES` automatically excludes BLANK values:

```dax
// With & operator - includes BLANK as "(Blank)"
Customer[CustomerID] & "-" & Customer[Country]
// Result: "1001-USA", "1002-", "1003-Canada"

// With COMBINEVALUES - filters out BLANK
COMBINEVALUES("-", Customer[CustomerID], Customer[Country])
// Result: "1001-USA", "1002", "1003-Canada"
```

## The Hash Collision Risk

### What Is a Hash Collision?

When `COMBINEVALUES` is used to create relationship keys in composite models, Power BI creates **hash values** of the combined strings for performance. A **hash collision** occurs when two different input strings produce the same hash value.

This is **rare** but **theoretically possible**.

### Why This Matters

In composite models (DirectQuery + Import), Power BI uses these hash values to:
- Match rows between tables
- Maintain relationships
- Ensure data integrity

A hash collision would cause:
- Wrong rows joined together
- Incorrect query results
- Data integrity issues

### Probability Assessment

Microsoft acknowledges the theoretical risk but states:
- Probability is **extremely low** with 64-bit hashing
- The risk is **comparable to cosmic rays causing bit flips** in memory 
- Most implementations are safe for normal business use

### High-Risk Scenarios

Risk increases when:

1. **Very large data volumes** - billions of rows increase probability
2. **Short, similar strings** - "A" vs "B" vs "C" have more collision potential
3. **Many columns combined** - more combinations, more hash values
4. **High cardinality relationships** - millions of unique keys

## Mitigation Strategies

### Strategy 1: Use Integer Keys When Possible

```dax
// Instead of combining multiple columns
Product Key Combined = COMBINEVALUES(
    "|", 
    Product[Category], 
    Product[SubCategory], 
    Product[ProductID]
)

// Use a single integer surrogate key
Product Key = Product[ProductKey]  // Integer from data source
```

### Strategy 2: Add a Unique High-Entropy Component

```dax
// Add a unique identifier to reduce collision probability
Safe Combined Key = COMBINEVALUES(
    "|",
    Product[Category],
    Product[SubCategory],
    Product[ProductID],  // Already unique
    FORMAT(Product[RowVersion], "X")  // Add entropy
)
```

### Strategy 3: Use COMBINEVALUES Only When Necessary

```dax
// Only combine if you must
VAR NeedCombinedKey = 
    ISCROSSFILTERED(Table1) || 
    ISCROSSFILTERED(Table2)

RETURN
IF(
    NeedCombinedKey,
    COMBINEVALUES("-", [Col1], [Col2]),
    [SingleColumn]
)
```

### Strategy 4: Document and Monitor

For critical systems, consider:
- Regular data validation checks
- Sample testing of joins
- Documentation of combined key strategy

## Composite Model Implications

### Storage Engine Behavior

In composite models:
- Import tables: full strings stored
- DirectQuery tables: hashes created for relationships
- Combined keys: hashed for cross-source joins

### Relationship Creation

When creating relationships between Import and DirectQuery tables:
1. Power BI creates hash of the combined key
2. Hash stored in memory for Import side
3. Hash calculated at query time for DirectQuery side
4. Join performed on hash values

This is where collision risk exists.

## Practical Examples

### Example 1: Safe Customer Key

```dax
// Safe approach - includes unique CustomerID
Customer Link Key = 
COMBINEVALUES(
    "||",
    Customer[CustomerID],      // Unique
    Customer[CompanyName],     // Descriptive
    Customer[Country]          // Additional context
)
```

### Example 2: Date Dimension Key

```dax
// Relatively safe - date strings have good distribution
Date Key = 
COMBINEVALUES(
    "-",
    'Date'[Year],
    FORMAT('Date'[Month], "00"),
    FORMAT('Date'[Day], "00")
)
// Results: "2024-01-15", "2024-01-16", etc.
```

### Example 3: Validation Measure

```dax
// Check for potential collisions
Collision Check = 
VAR KeyCount = 
    DISTINCTCOUNT(Table[CombinedKey])
VAR RowCount = 
    COUNTROWS(Table)
RETURN
IF(
    KeyCount < RowCount,
    "Potential collision detected!",
    "No collisions"
)
```

## Alternative Approaches

### Option 1: Concatenate with Delimiter (No Hashing)

```dax
// Simple concatenation - full string stored
Simple Key = 
Table[Col1] & "|" & Table[Col2] & "|" & Table[Col3]
```

**Pros**: No hash collision risk  
**Cons**: Higher storage, slower for large joins

### Option 2: Surrogate Key from Source

```dax
// Use existing unique identifier
Source Key = Table[NaturalKey]  // Already unique
```

**Pros**: No collision risk, efficient  
**Cons**: Requires key in source system

### Option 3: Calculated Hash with Algorithm

```dax
// Custom hashing with known algorithm
Custom Hash = 
VAR InputString = 
    Table[Col1] & "|" & Table[Col2]
RETURN
HASH64(InputString)  // Not actual DAX - conceptual
```

**Note**: DAX doesn't have built-in hash functions besides COMBINEVALUES

## Best Practices Summary

### Do's
✓ **Use with unique identifiers** - include a column with high uniqueness  
✓ **Test with representative data** - especially edge cases  
✓ **Consider business criticality** - financial systems need more caution  
✓ **Document the approach** - so others understand the trade-offs  
✓ **Validate results** - spot-check joins periodically  

### Don'ts
✗ **Don't combine only low-cardinality columns** - "Red|Large|Cotton" has limited uniqueness  
✗ **Don't ignore the theoretical risk** - acknowledge it in documentation  
✗ **Don't use for cryptographic purposes** - not designed for security  
✗ **Don't assume it's 100% collision-proof** - no hash function is  

## Microsoft's Official Stance

From Microsoft documentation:

> "The hash value collisions can potentially occur. Collisions are an unlikely event with the current 64-bit hash function. The risk is comparable to that of hardware malfunction due to cosmic rays hitting the chip and causing bit flips." 

And:

> "If you're working with composite models and you define a relationship based on multiple columns, Power BI creates an internal hash of the combined columns. This behavior ensures that the relationship can work across different storage modes." 

## Summary

**Key Points for COMBINEVALUES and Hash Collision Risk:**

1. **Theoretical risk exists** - any hash function can have collisions
2. **Probability is extremely low** - comparable to cosmic ray interference
3. **Risk increases with** - large data volumes, low-entropy inputs
4. **Mitigation** - include unique high-cardinality columns
5. **Alternatives exist** - integer keys, full string storage
6. **Validate critical data** - spot-check joins in important systems
