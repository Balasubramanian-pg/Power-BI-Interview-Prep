# The Scenario: Sales Transactions with a Shared Product-Promo Bridge

>[!Tip]
>Explain a specific scenario where you were forced to use a bidirectional cross-filter. What were the performance implications, and how did you mitigate them without reverting to a many-to-many relationship?

Imagine you have a retail data model with these tables:

**Sales** (fact) → linked to **Products** (dimension) via `ProductID`
**Promotions** (fact) → linked to **Products** (dimension) via `ProductID`

You need a report that shows, for a given product selection, **which promotions were active during periods when that product was sold** — essentially filtering Promotions *through* Products based on activity in Sales. Because both fact tables point *into* the dimension, a single-direction filter from Products won't propagate back out to Promotions. You're pushed toward enabling **bidirectional cross-filtering** on the Products ↔ Promotions relationship.

## Why Bidi Was Tempting Here

With bidi enabled on the Products relationship, selecting a product in a slicer would let the filter context flow from Sales → Products → Promotions, surfacing only promotions tied to actually-sold products. It "worked" immediately and required zero DAX.

## The Performance Implications

This is where things got painful at scale (~50M rows in Sales, ~5M in Promotions):

**Ambiguous filter propagation.** The engine had to resolve competing filter paths. With bidi active, every cross-table filter forced the VertiPaq engine to evaluate both directions simultaneously, which meant queries that previously ran in ~200ms were taking 2–4 seconds.

**DAX query plan bloat.** Storage engine queries multiplied. Where a unidirectional model would generate a clean, linear scan, bidi introduced CallbackDataID operations — these pull data back through the formula engine, bypassing VertiPaq's columnar compression advantage entirely.

**Report-level amplification.** Visuals that were previously independent now shared filter state unexpectedly. A slicer on Promotions would bleed into Sales aggregations in ways that confused end users and broke measure logic elsewhere on the page.

## The Mitigation: DAX-Driven Virtual Relationships

Rather than reverting to a many-to-many bridge table, the fix was to **remove bidi entirely and replicate its filtering behavior in DAX using `CROSSFILTER` and `TREATAS`**.

The key measure looked like this:

```dax
Promo Sales Overlap =
VAR SelectedProducts =
    VALUES( Products[ProductID] )
CALCULATE(
    [Total Promo Revenue],
    TREATAS( SelectedProducts, Promotions[ProductID] )
)
```

`TREATAS` virtually injects the product selection as a filter directly into the Promotions table, mimicking what bidi would have done at the relationship level — but *only when that specific measure is evaluated*, not globally across the entire model. The filter scope stays surgical.

For cases where I needed it query-wide rather than measure-by-measure, I wrapped it in a scoped `CALCULATE` with `CROSSFILTER( Products[ProductID], Promotions[ProductID], Both )` so the bidi behavior was opt-in per calculation rather than always-on.

## The Net Result

Query times dropped back to ~250ms. Filter bleed between unrelated visuals disappeared. And critically, the model remained a clean star schema — no bridge table, no many-to-many, and no hidden ambiguity for future developers to stumble on.

The broader lesson: **bidirectional cross-filtering solves a real problem, but it's a model-level sledgehammer**. When you can scope the same behavior to a specific DAX measure, you get the answer you need without paying the engine-wide performance tax.
