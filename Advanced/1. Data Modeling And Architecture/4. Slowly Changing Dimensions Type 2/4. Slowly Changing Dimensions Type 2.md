# Slowly Changing Dimensions (SCD Type 2) in Power BI Without a Data Warehouse

Implementing **Type 2 SCD** in Power BI without a formal data warehouse is absolutely possible. It simply means you move some of the warehouse logic into:

* Power Query (ETL layer inside Power BI)
* DAX (for effective date filtering)
* Proper data modeling

Below is a structured, real-world approach based on production-style implementation.


# 1. Objective of Type 2 SCD

Type 2 SCD preserves historical versions of dimension records.

If a customer changes region:

| Customer | Region | Start Date  | End Date    | Is Current |
| -------- | ------ | ----------- | ----------- | ---------- |
| C001     | West   | 01-Jan-2023 | 30-Jun-2024 | No         |
| C001     | North  | 01-Jul-2024 | 31-Dec-9999 | Yes        |

Fact records should join to the correct version based on transaction date.


# 2. Overall Architecture in Power BI (No Data Warehouse)

Without a warehouse, the architecture typically looks like this:

Source System → Power Query → SCD Logic → Dimension Table (with versions) → Fact Table → DAX Filtering

The key challenge:
Power BI relationships do not support date-range joins natively.

So we simulate it.


# 3. Step-by-Step Implementation

## Step 1: Prepare Source Data

Assume:

Fact Table:

* SalesID
* CustomerID
* TransactionDate
* Amount

Dimension Source:

* CustomerID
* Region
* Segment
* LastModifiedDate


## Step 2: Create Type 2 Structure in Power Query

In Power Query:

### A. Detect Changes

Group by CustomerID and sort by LastModifiedDate.

For each CustomerID:

* Sort ascending by LastModifiedDate
* Add Index column

Then:

* StartDate = Current row LastModifiedDate
* EndDate = Next row LastModifiedDate - 1 day
* If last row → EndDate = #date(9999,12,31)

Power Query logic conceptually:

1. Sort by CustomerID + LastModifiedDate
2. Add Index per Customer
3. Self-join on Index + 1
4. Compute EndDate

Resulting dimension:

| CustomerID | Region | StartDate | EndDate | SurrogateKey |
| ---------- | ------ | --------- | ------- | ------------ |

Generate a SurrogateKey using:

* Index column
* Or concatenate CustomerID & StartDate

This surrogate key is critical.


## Step 3: Handling the Relationship Problem

Power BI cannot do:

Fact.CustomerID = Dim.CustomerID
AND
Fact.TransactionDate BETWEEN Dim.StartDate AND Dim.EndDate

So we solve this using one of two approaches.


# Approach 1 (Recommended for Large Models): Pre-Resolve in Power Query

We push the date match into Power Query.

### Merge Fact with Dimension

In Power Query:

1. Merge Fact and Dimension on CustomerID.
2. Expand all matching dimension rows.
3. Filter rows where:

TransactionDate >= StartDate
AND
TransactionDate <= EndDate

4. Keep only matching rows.

Now each fact row is already mapped to correct surrogate key.

This makes the model simple.

Final Model:

Fact (with SurrogateKey)
→ Relationship to Dimension on SurrogateKey

Advantages:

* Best performance
* Clean DAX
* Works well for enterprise models


# Approach 2: Use DAX with Virtual Relationships (Smaller Models)

If you cannot pre-merge:

Create inactive relationship:
Fact[CustomerID] → Dim[CustomerID]

Then create measure like:

```DAX
Sales Amount :=
CALCULATE(
    SUM(Fact[Amount]),
    FILTER(
        DimCustomer,
        DimCustomer[CustomerID] = SELECTEDVALUE(Fact[CustomerID]) &&
        Fact[TransactionDate] >= DimCustomer[StartDate] &&
        Fact[TransactionDate] <= DimCustomer[EndDate]
    )
)
```

This works but:

* Slower
* Harder to maintain
* Not scalable


# 4. Managing Start and End Dates

## Rules Used

* StartDate = Effective date from source
* EndDate = Next StartDate - 1
* Last record EndDate = 31-Dec-9999
* IsCurrent flag = EndDate = 31-Dec-9999

## Why 9999-12-31?

It avoids NULL logic and simplifies filters.


# 5. Ensuring Historical Reporting Accuracy

Accuracy depends on three things:

## 1. No Overlapping Date Ranges

Validation rule in Power Query:

* Ensure EndDate of previous = StartDate of next - 1

## 2. One Active Version per Date

Test with DAX validation measure:

```DAX
Version Check :=
COUNTROWS(
    FILTER(
        DimCustomer,
        Fact[TransactionDate] >= DimCustomer[StartDate] &&
        Fact[TransactionDate] <= DimCustomer[EndDate]
    )
)
```

Should always return 1.

## 3. Surrogate Key Integrity

Fact table must always use SurrogateKey, not natural key.


# 6. Handling Incremental Refresh

Without warehouse:

* Store historical dimension table externally (Excel, SharePoint, Dataflow)
* On refresh:

  * Append new changes
  * Recompute only affected CustomerIDs

If using Power BI Dataflows:
SCD logic can be centralized and reused across reports.


# 7. Real-World Example Scenario

Customer moved from West to North in July.

If user selects:

Year = 2023 → Shows West
Year = 2024 Q3 → Shows North

This works because:

Fact is mapped to correct surrogate key version before entering the model.


# 8. Performance Considerations

Best Practices:

* Always pre-resolve SCD in Power Query
* Avoid DAX date-range filtering
* Use surrogate keys
* Keep dimension table small
* Avoid bi-directional relationships


# 9. When This Approach Fails

This approach becomes difficult when:

* Dimension changes daily for millions of records
* Fact table > 100M rows
* Frequent incremental refresh required

In such cases:
You need proper warehouse SCD implementation.


# 10. Interview-Level Summary Answer

If asked in interview:

"I implemented Type 2 SCD directly in Power Query by detecting changes using LastModifiedDate, generating StartDate and EndDate per version, and assigning surrogate keys. I then pre-joined the fact table to the correct dimension version based on transaction date using a date-range filter during merge. This ensured that each fact record was tied to the historically accurate dimension state. I used 31-Dec-9999 as the open-ended EndDate and validated that there were no overlapping ranges. This approach preserved historical reporting accuracy without requiring a data warehouse."


# Final Conceptual Understanding

Think of Type 2 SCD like version-controlled software:

Every change creates a new release.
Fact rows connect to the version that existed at the time of the transaction.
Power BI just needs help deciding which version that was.
