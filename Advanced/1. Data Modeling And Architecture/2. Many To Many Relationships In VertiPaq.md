## Scenario Overview

In one of my BI implementations for a pharma commercial analytics use case, I designed a Composite Model in Power BI combining:

* Import mode for historical transactional data
* DirectQuery mode for near real-time incremental data

The objective was to balance performance and freshness without overloading the source warehouse.

The dataset involved:

* 3+ years of historical sales data stored in a Snowflake warehouse
* Current month data refreshed every 15 minutes
* High-granularity fact table at HCP × Product × Date level
* Executive dashboards requiring sub-second performance

Importing everything would have caused long refresh times and large PBIX size. Pure DirectQuery would have slowed down executive dashboards. So Composite Model was the optimal trade-off.

---

## Model Architecture

### Import Layer

* Historical fact table aggregated to Month × Product × Territory
* Dimension tables fully imported
* Aggregation table stored in Import mode

### DirectQuery Layer

* Detailed fact table for current month
* Connected live to Snowflake
* Used for drill-through and near real-time visuals

### Storage Mode Strategy

* Dimensions set to Dual mode
* Aggregation table set to Import
* Detailed fact set to DirectQuery

This allowed Power BI to automatically redirect queries to Import when possible.

---

## Aggregation Table Design

Aggregation design was the most critical component.

### Step 1: Identify Query Patterns

I analyzed:

* DAX queries generated by visuals
* Drill paths used by users
* Most frequent grouping combinations

Common grouping levels were:

* Month × Product
* Month × Territory
* Quarter × Brand

No one was analyzing at raw transaction level except in drill-through pages.

---

### Step 2: Build Pre-Aggregated Table in Snowflake

Example aggregation logic:

```
SELECT 
    Month,
    Product_ID,
    Territory_ID,
    SUM(Sales) AS Sales,
    SUM(Units) AS Units
FROM Fact_Sales
WHERE Date < Current_Month
GROUP BY 1,2,3
```

This table reduced row count by more than 95 percent.

Then it was imported into Power BI and marked as an Aggregation table using:

* GroupBy columns mapped to dimension keys
* Summarized columns mapped to DirectQuery fact

---

### Step 3: Configure Manage Aggregations

Inside Power BI:

* Aggregated columns mapped as SUM to DirectQuery columns
* GroupBy columns mapped to their equivalent columns
* Detail table set to DirectQuery

Power BI then automatically hit Import when queries matched aggregation granularity.

## Consistency Challenges Between Import and DirectQuery

Composite models introduce subtle but serious consistency risks.

### 1. Time Boundary Drift

Problem:

* Import data refreshed nightly
* DirectQuery data was real-time

If users filtered across month boundary, totals could mismatch.

Solution:

* Hard boundary logic
* Historical data only up to last completed month in Import
* Current month exclusively in DirectQuery
* Created calculated flag:

```
Is_Current_Month = IF(Month = Current_Month, 1, 0)
```

Reports prevented mixing granularities accidentally.

---

### 2. DAX Behavior Differences

Import engine uses VertiPaq.
DirectQuery pushes SQL to source.

Certain DAX functions behaved differently:

* DISTINCTCOUNT performance degraded in DirectQuery
* Complex calculated columns not allowed in DirectQuery
* Time intelligence required explicit Date table

Solution:

* Moved heavy logic to Snowflake views
* Avoided calculated columns in DirectQuery fact
* Ensured Date dimension fully imported

---

### 3. Data Type and Rounding Differences

Import mode used decimal compression.
Snowflake returned numeric(38,10).

Minor rounding differences appeared in totals when users compared:

* Executive view (Import)
* Drill-through (DirectQuery)

Solution:

* Standardized numeric precision in Snowflake
* Explicit ROUND at aggregation level
* Used consistent data types in model

---

### 4. Relationship Behavior and Dual Mode Complexity

Dimensions were set to Dual.

If a visual accidentally included a non-aggregated column:

* Power BI fell back to DirectQuery
* Performance dropped dramatically

Solution:

* Strict modeling discipline
* Hide unnecessary columns
* Clear separation of “summary pages” and “detail pages”

---

### 5. Security Consistency

Row Level Security must work across both storage modes.

Challenge:

* RLS filters apply differently in DirectQuery

Solution:

* RLS applied only on dimension tables
* Avoided RLS on fact tables
* Ensured dimension filters propagated correctly

---

## Performance Outcome

After implementing aggregation table properly:

* 90 percent of dashboard queries hit Import
* Sub-second executive dashboards
* Drill-through pages executed DirectQuery in 2–5 seconds
* PBIX size reduced by more than 70 percent

---

## Key Lessons

### 1. Composite models are architectural decisions, not shortcuts

They require clear boundary definitions.

### 2. Aggregation tables must reflect real query patterns

Guessing aggregation levels leads to fallback to DirectQuery.

### 3. Consistency control is more important than performance

Users lose trust faster from number mismatches than from slow visuals.

### 4. Test with Performance Analyzer and DAX Studio

Confirm which storage mode each query hits.

---

## Short Summary of Reasoning

* Identified performance versus freshness trade-off
* Segmented data by time boundary
* Designed aggregation aligned with user grouping patterns
* Mapped aggregations carefully
* Addressed consistency risks: time, DAX behavior, precision, RLS

Composite models are like running two engines in one aircraft. If the throttle alignment is off, you feel turbulence. If tuned correctly, it flies smoothly and efficiently.
