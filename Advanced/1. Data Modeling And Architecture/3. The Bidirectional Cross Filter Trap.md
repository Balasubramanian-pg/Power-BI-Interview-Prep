### Topic: Role-Playing Dimensions in VertiPaq

## 1. Context and Scenario

Role-playing dimensions occur when a single dimension table, typically Date, connects to multiple columns in a fact table. This structure appears in transactional models requiring analysis by Order Date, Ship Date, or Due Date. This choice dictates model memory footprint and DAX complexity.

## 2. Baseline Behavior

VertiPaq requires a single active relationship between any two tables to ensure unambiguous filter propagation. The storage engine uses this active path to apply filters without DAX intervention. Query performance remains optimal because the relationship index is pre-calculated during data refresh.

## 3. What Changes in the Advanced Scenario

Introducing a second relationship forces one path to become inactive. The storage engine ignores inactive relationships during automatic filter propagation. DAX requires explicit activation of the inactive path via USERELATIONSHIP, or the dimension table must be duplicated to create separate physical relationships.

## 4. Internal Engine Handling

### 4.1 Relationship Structure
Active relationships build a persistent index for cross-filtering. Inactive relationships exist only as metadata until invoked by a function.

### 4.2 Filter Propagation
Filters flow automatically across active paths. Inactive paths require context modification within the measure definition to trigger propagation.

### 4.3 Query Execution Path
USERELATIONSHIP alters the query plan at execution time. The formula engine instructs the storage engine to swap the relationship index temporarily.

### 4.4 Compression or Encoding Impact
A single dimension table shares one dictionary across all columns. Duplicating the table creates separate dictionaries, even if the data values match.

## 5. Memory Implications

### Static Model Size
*   Inactive relationships add negligible metadata overhead.
*   Duplicating the dimension table doubles the dictionary size for that entity.
*   Columnar compression benefits are lost when data is split across multiple tables.

### Runtime Memory
*   USERELATIONSHIP requires minimal temporary memory for context switching.
*   Physical duplication does not increase runtime memory significantly beyond the static cost.
*   Large date tables magnify the storage penalty of duplication.

## 6. Performance Trade-offs

*   USERELATIONSHIP introduces minor CPU overhead during filter context modification.
*   Storage engine efficiency remains high with inactive relationships.
*   Duplicating tables reduces DAX complexity but increases refresh time and model size.
*   Query speed is generally comparable unless the model exceeds available RAM.

## 7. Risks and Mitigation

*   Restrict duplication to scenarios where model memory is unconstrained.
*   Mitigate DAX complexity by creating base measures that encapsulate USERELATIONSHIP logic.
*   Test model size after duplication to ensure dictionaries do not fragment.
*   Prefer inactive relationships for enterprise-scale deployments where memory is premium.

## 8. Short Executive Summary

Inactive relationships preserve memory and maintain a single source of truth. Duplication simplifies DAX for end users while inflating model size and refresh duration. Scalability generally demands the efficiency of inactive relationships. Developer convenience might justify duplication in smaller models where storage costs remain negligible.
