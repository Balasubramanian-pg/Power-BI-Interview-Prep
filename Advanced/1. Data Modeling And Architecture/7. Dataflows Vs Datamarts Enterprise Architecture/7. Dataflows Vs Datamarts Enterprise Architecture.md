# Dataflows vs Azure Data Factory vs SQL Views

## Enterprise Architecture Decision Framework

In enterprise BI architecture, the decision is not tool preference, it is workload placement.

Each tool belongs to a different architectural layer:

* **SQL Views** → Source-layer transformation
* **Azure Data Factory (ADF)** → Orchestration + heavy ETL
* **Power BI Dataflows** → Self-service semantic ETL inside Power BI ecosystem

The question is not which is better. The question is where transformation logic should live.


# 1. When I Recommend Power BI Dataflows

I recommend Dataflows when the transformation need is:

## 1.1 Reusable Semantic Preparation

* Standardizing dimensions like Customer, Product, Territory
* Creating conformed dimensions for multiple reports
* Centralizing business logic for Power BI only

If the consumers are primarily Power BI datasets, Dataflows provide a clean, reusable layer without needing data engineering involvement.


## 1.2 Self-Service but Governed

Dataflows work well when:

* BI team owns transformation logic
* Business logic changes frequently
* Central IT bandwidth is limited

They allow version-controlled Power Query logic stored in the service.


## 1.3 Medium Data Volume

Best suited for:

* < 50–100 million rows per entity
* Incremental refresh enabled
* Mostly append-based transformations

Beyond this, performance degrades significantly.


## 1.4 Power Platform Integration

If organization uses:

* Power Apps
* Dataverse
* Fabric Lakehouse (Gen2 Dataflows)

Dataflows integrate more naturally than SQL Views.


# 2. When I Recommend Azure Data Factory Instead

ADF is preferable when:

## 2.1 Heavy Data Engineering Workloads

* Complex joins across multiple systems
* Large data volumes
* CDC handling
* Slowly Changing Dimensions Type 2 at scale

ADF provides:

* Pipeline orchestration
* Error handling
* Monitoring
* Parallelization

Dataflows lack enterprise-grade orchestration.


## 2.2 Multi-Consumer Architecture

If data feeds:

* Power BI
* ML pipelines
* External APIs
* Data warehouse

Then transformation should not live in Power BI.

It should live upstream in ADF or SQL.


## 2.3 Advanced Scheduling & Dependencies

Dataflows:

* Limited dependency chaining
* No advanced retry logic
* Limited parameterization

ADF supports enterprise workflow management.


# 3. When SQL Views Are Better

SQL Views are preferable when:

## 3.1 Source Is a Strong SQL Engine

If source is:

* Azure SQL
* Snowflake
* Synapse
* SQL Server

Push transformations down.

SQL engines are optimized for joins and aggregations.

Query folding works best when logic remains inside SQL.


## 3.2 Performance Is Critical

Views:

* Execute close to data
* Avoid data movement
* Use indexes and statistics

Dataflows:

* Pull data out first
* Then transform in M engine
* May break folding


# 4. Real Enterprise Recommendation Matrix

| Scenario                        | Recommended |
| ------------------------------- | ----------- |
| Power BI-only semantic reuse    | Dataflows   |
| Enterprise data integration     | ADF         |
| High-performance SQL backend    | SQL Views   |
| Cross-platform data consumption | ADF or SQL  |
| Self-service BI team            | Dataflows   |
| Complex orchestration           | ADF         |


# 5. Specific Limitations Encountered with Dataflows (Query Folding)

This is where real-world issues appear.

## 5.1 Folding Breaks Easily

Query folding breaks when:

* Using custom columns with complex M logic
* Using certain text functions
* Using Index columns
* Using Table.Buffer()
* Merging with non-foldable sources
* Using Group By after non-folding steps

Once folding breaks, all subsequent steps execute in Power BI service memory.

Performance drops dramatically.


## 5.2 No Folding Visibility in Service (Gen1)

Unlike Power BI Desktop, Dataflows Gen1:

* Do not clearly show folding indicators
* Harder to debug performance issues

You must test queries in Desktop first.


## 5.3 Join Behavior Limitations

When joining:

* SQL source + Excel source
* API + SQL source

Folding stops immediately.

Entire dataset is pulled before join.

This becomes expensive for large tables.


## 5.4 Computed Entities Dependency Bottlenecks

Computed Entities:

* Require Enhanced Compute Engine
* Create refresh dependencies
* Can slow entire workspace refresh

Large dataflows chained together increase refresh times exponentially.


## 5.5 Incremental Refresh Limitations

Dataflows incremental refresh:

* Less flexible than dataset incremental refresh
* Partition management not transparent
* Limited control compared to ADF pipelines


## 5.6 No Advanced Indexing

Dataflows store data in:

* Azure Data Lake storage (CDM format)

You cannot:

* Create indexes
* Tune execution plans
* Optimize storage structures

Performance tuning is limited.


# 6. Real Limitation Case I Encountered

Scenario:

* 40M row fact table from Azure SQL
* Added Index column for surrogate key
* Performed Group By after

Result:

* Folding broke at Index step
* Entire 40M rows pulled to service
* Refresh time jumped from 6 minutes to 48 minutes

Resolution:

* Moved surrogate key generation to SQL
* Kept Dataflow only for light transformations

Refresh reduced back to 8 minutes.


# 7. Dataflows Gen2 (Fabric Context)

Gen2 improves:

* Direct Lake integration
* Better performance
* Lakehouse staging

But limitations still exist:

* Not full ETL replacement
* Not orchestration engine
* Not optimized like SQL for heavy transformations


# 8. Final Architectural Principle

Use Dataflows when transformation is:

* Semantic
* Power BI scoped
* Medium scale
* BI-team owned

Use ADF or SQL when transformation is:

* Enterprise-wide
* Heavy compute
* Multi-system
* Performance-critical


# 9. Interview-Ready Summary

"In enterprise architecture, I recommend Power BI Dataflows when transformations are Power BI-specific and require reusable semantic logic across datasets. However, for heavy ETL workloads, multi-system integration, or large volumes, I prefer Azure Data Factory or SQL views to keep processing closer to the source. One limitation I encountered with Dataflows is fragile query folding. Adding non-foldable steps like Index columns or complex M functions caused folding to break, leading to full data pulls and significant refresh time increases. To resolve this, I push heavy transformations upstream and use Dataflows for lighter, reusable shaping logic."
