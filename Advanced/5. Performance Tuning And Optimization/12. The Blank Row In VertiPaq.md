# The Blank Row in VertiPaq

## Overview

The blank row is a special row automatically added by VertiPaq to the one-side of a relationship when referential integrity is violated. It ensures that orphaned rows on the many-side remain visible in queries, preventing data from silently disappearing .

```
Fact Table (Many-side)          Dimension Table (One-side)
---------------------           -------------------------
ProductKey  Sales               ProductKey  ProductName
    1        100                    1        "TV"
    2        200                    2        "Phone"
    99       300                    ---- missing ProductKey 99 ----
                                    ═══════════════════════════════
                                    BLANK row (automatically added)

Result: Orphaned row (ProductKey=99) joins to BLANK row in dimension
```

## How the Blank Row Is Created

### Referential Integrity Violation

When a fact table contains values that don't exist in the related dimension table, VertiPaq automatically:

1. **Adds a blank row** to the dimension table with BLANK in every column 
2. **Links all orphaned rows** to this blank row via the relationship 
3. **Preserves data visibility** - orphaned rows still appear in queries

```dax
-- Example: Products removed from dimension, but sales remain
-- Sales for "Pink" products now join to blank row in Product table
```

### Visual Representation

```
Product Table (after RI violation):
| ProductKey | ProductName | Color    |
|------------|-------------|----------|
| 1          | TV          | Black    |
| 2          | Phone       | Silver   |
| 3          | Laptop      | Gray     |
| (BLANK)    | (BLANK)     | (BLANK)  | ← Auto-added blank row

Sales Table:
| SaleID | ProductKey | Amount |
|--------|------------|--------|
| 101    | 1          | 100    | → Joins to ProductKey=1
| 102    | 2          | 200    | → Joins to ProductKey=2
| 103    | 99         | 300    | → Joins to BLANK row
```

## Impact on DAX Functions

### VALUES vs. DISTINCT

The blank row creates a critical difference between two seemingly identical functions :

| Function | Blank Row Handling | Example Result |
|----------|-------------------|----------------|
| **VALUES** | Includes blank row as regular row | 16 colors (including blank) |
| **DISTINCT** | Excludes blank row entirely | 15 colors (only actual values) |

```dax
#Colors Distinct := COUNTROWS ( DISTINCT ( 'Product'[Color] ) )
#Colors Values := COUNTROWS ( VALUES ( 'Product'[Color] ) )

-- With RI violation: VALUES returns 16, DISTINCT returns 15
```

### ALL vs. ALLNOBLANKROW

The blank row also affects table and column references :

| Function | Blank Row Handling |
|----------|-------------------|
| **ALL** | Includes blank row |
| **ALLNOBLANKROW** | Excludes blank row |
| **Table reference** | Excludes blank row |

```dax
-- Problem: Inconsistent counting
Perc #Prods := 
DIVIDE ( 
    COUNTROWS ( 'Product' ),      -- Excludes blank row (2,433)
    COUNTROWS ( ALL ( 'Product' ) ) -- Includes blank row (2,434)
)
-- Result: Percentages don't sum to 100%

-- Solution 1: Use VALUES in numerator
Perc #Prods Values := 
DIVIDE ( 
    COUNTROWS ( VALUES ( 'Product' ) ),  -- Includes blank row
    COUNTROWS ( ALL ( 'Product' ) )       -- Includes blank row
)

-- Solution 2: Use ALLNOBLANKROW in denominator
Perc #Prods Distinct := 
DIVIDE ( 
    COUNTROWS ( DISTINCT ( 'Product' ) ),  -- Excludes blank row
    COUNTROWS ( ALLNOBLANKROW ( 'Product' ) ) -- Excludes blank row
)
```

## Performance Implications

### Minimal Direct Impact

According to Marco Russo, RI violations and the resulting blank row are generally **not a performance concern** :

> "This is not a problem. The presence of RI violations generates a blank row on the one-side of the relationship and in certain conditions, this could have an impact on performance. However, you should not be worried about that."

### When Performance Can Be Affected

The blank row can impact performance in specific scenarios:

1. **Large number of orphaned rows** - Many rows joining to a single blank row can create a hot spot
2. **Complex calculations** - Functions that need to handle the blank row specially may have slightly different query plans
3. **Model size** - The blank row itself adds one row to the dimension (negligible)

## Detecting and Monitoring Blank Row Issues

### Using VertiPaq Analyzer

VertiPaq Analyzer (in DAX Studio or Tabular Editor) provides key metrics :

| Metric | Description |
|--------|-------------|
| **RI Violations** | Total rows with referential integrity issues |
| **Invalid Rows** | Rows with non-blank values missing from dimension |
| **Blank Rows (Unrelated)** | Rows where relationship column is intentionally BLANK |

### Distinguishing Error Types

Recent discussions highlight the need to differentiate :

```dax
-- Two types of RI "errors":
-- 1. Missing Keys: Non-blank values not in dimension (actual data quality issue)
-- 2. Blank Rows: Intentionally blank values (business-valid)

-- Example: Job completion date intentionally blank for incomplete jobs
-- This should NOT be treated as an error
```

### Checking for Orphaned Rows

```dax
-- Find orphaned rows
EVALUATE
FILTER (
    Sales,
    NOT Sales[ProductKey] IN VALUES ( 'Product'[ProductKey] )
)

-- Count orphaned rows by type
EVALUATE
ROW (
    "Total Orphaned", 
        COUNTROWS (
            FILTER (
                Sales,
                NOT Sales[ProductKey] IN VALUES ( 'Product'[ProductKey] )
            )
        ),
    "Missing Keys",
        COUNTROWS (
            FILTER (
                Sales,
                NOT Sales[ProductKey] IN VALUES ( 'Product'[ProductKey] ) &&
                Sales[ProductKey] <> BLANK()
            )
        ),
    "Blank Values",
        COUNTROWS (
            FILTER (
                Sales,
                Sales[ProductKey] = BLANK()
            )
        )
)
```

## Visual Behavior

### Slicer and Filter Effects

The blank row appears in visuals only when no filters exclude it :

| Selection | Blank Row Visible? |
|-----------|-------------------|
| **Specific values** (e.g., 2022, 2023) | No |
| **Select All** | Yes |
| **No filter applied** | Yes |

**Explanation**: Selecting specific values filters the dimension to those values, excluding the blank row. "Select All" applies no filter, so the blank row becomes visible .

## Best Practices

### Do's
✓ **Validate relationships** - Ensure referential integrity where possible 
✓ **Use appropriate functions** - Choose VALUES vs DISTINCT based on blank row needs
✓ **Monitor RI violations** - Track in VertiPaq Analyzer 
✓ **Document intentional blanks** - Note when blanks are business-valid 
✓ **Test both scenarios** - Verify calculations with and without blank row

### Don'ts
✗ **Don't ignore RI violations** - They indicate data quality issues 
✗ **Don't blindly use table references** - They exclude blank row 
✗ **Don't assume VALUES and DISTINCT are interchangeable** - They differ with RI issues
✗ **Don't panic about performance** - Blank row itself rarely causes problems

## Summary

**Key Points About the Blank Row:**

| Aspect | Description |
|--------|-------------|
| **Purpose** | Preserves orphaned rows in queries |
| **Creation** | Auto-added when RI violated |
| **VALUES** | Includes blank row |
| **DISTINCT** | Excludes blank row |
| **ALL** | Includes blank row |
| **ALLNOBLANKROW** | Excludes blank row |
| **Performance** | Minimal direct impact |
| **Visuals** | Appears with "Select All" |

**The Golden Rule:**
> The blank row is VertiPaq's safety net for data integrity issues. Understand how your DAX functions handle it, validate your relationships, and choose the right function (VALUES vs DISTINCT, ALL vs ALLNOBLANKROW) based on whether you need to include or exclude the blank row in your calculations.
