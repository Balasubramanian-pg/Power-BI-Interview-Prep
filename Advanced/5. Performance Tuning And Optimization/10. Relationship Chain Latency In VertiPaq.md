# Relationship Chain Latency in VertiPaq

## Overview

Relationship chain latency refers to the cumulative performance cost incurred when filters propagate across multiple related tables in a VertiPaq model. Each relationship in a chain adds processing overhead, and understanding this cost is crucial for optimizing DAX query performance.

```
Filter on DimA → Relationship A-B → Relationship B-C → Result on Fact
       ↓                 ↓                 ↓                 ↓
    Source         Cost: Scan        Cost: Scan        Cumulative
    Table          + Translate       + Translate       Latency
```

## How Relationships Work in VertiPaq

### Physical Structure

Relationships in VertiPaq are data structures designed to speed up filter propagation from the one-side to the many-side of a relationship . They are carefully crafted to work with the internal columnar representation:

- **Regular relationships** (one-to-many, one-to-one) are the only truly optimized relationship type
- These create internal indexes mapping column values for fast lookup
- The physical structure remains the same regardless of filter direction settings

### Filter Propagation Cost

When a filter propagates through a relationship, VertiPaq performs a multi-step process :

```dax
// Example query requiring relationship traversal
EVALUATE
SUMMARIZECOLUMNS (
    'Product'[Brand],
    "Sales", [Sales Amount]
)
```

**Behind the scenes:**
1. Scan the filtered column in the dimension (Product[Brand])
2. Use the relationship to translate dimension row numbers into fact column values
3. Scan the fact column (Sales[ProductKey]) to count rows satisfying the filter

## Measuring Relationship Latency

### Baseline Without Relationship

```dax
// Scan Sales table directly - no relationship cost
EVALUATE
{
    COUNTROWS (
        FILTER ( Sales, Sales[ProductKey] >= 1 )
    )
}
```

Without relationship involvement, the storage engine time is approximately **5,531 ms** for large models .

### With Single Relationship

```dax
// Filter propagates through relationship
EVALUATE
{
    COUNTROWS (
        FILTER ( Sales, RELATED ( Product[ProductKey] ) >= 1 )
    )
}
```

Adding one relationship increases storage engine time to **7,266 ms** - a **31% increase** for a simple count .

### With Multiple Relationships

```dax
// Three relationships in chain
EVALUATE
{
    COUNTROWS (
        FILTER (
            Sales,
            RELATED ( Customer[CustomerKey] ) >= 1 &&
            RELATED ( Product[ProductKey] ) >= 1 &&
            RELATED ( Store[StoreKey] ) >= 1
        )
    )
}
```

Time increases to **21,531 ms** - each relationship adds incremental cost .

## Factors Affecting Chain Latency

### 1. Cardinality of Dimension Tables

The cost of a relationship strongly depends on the number of unique values in the dimension table :

| Dimension | Rows | Query Time |
|-----------|------|-------------|
| Customer | 1.8M | ~15,000 ms |
| Product | 2.5K | ~8,500 ms |

Larger dimensions require more work to translate filters.

### 2. Filter Selectivity

The exact time required depends on filter selectivity. A relationship with millions of rows is slower than a small relationship with a few hundred values .

### 3. Number of Relationships in Chain

Each relationship adds translation overhead. In tests with three relationships, time increased from **5,531 ms** to **21,531 ms** - nearly **4x** the baseline .

## Limited vs. Regular Relationships

### Regular Relationships (Optimized)

- One-to-many or one-to-one within same source group
- VertiPaq creates internal data structures at refresh time
- Can be pushed down to storage engine as single query
- Enable table expansion with LEFT OUTER JOIN semantics 

### Limited Relationships (Performance Impact)

Limited relationships occur when :
- **Many-to-many cardinality** (even if columns contain unique values)
- **Cross-source group** relationships in Composite models
- **Cross-storage mode** relationships (Import to DirectQuery)

**Performance comparison** :

| Relationship Type | SE CPU Time | Relative Cost |
|-------------------|-------------|----------------|
| Regular (one-to-many) | 15,000 ms | 1x |
| Limited (many-to-many) | 138,000 ms | ~9x |

The many-to-many relationship makes calculations almost **20 times more expensive** compared to regular relationships .

### Why Limited Relationships Are Slower

1. **No internal data structures** - Power BI resolves joins at query time 
2. **Multiple storage engine queries** - Cannot be pushed down as single query
3. **No table expansion** - Uses INNER JOIN semantics
4. **Fusion optimization disabled** - Cannot merge multiple measures into one query 

## Bidirectional Relationships

### Performance Impact

Bidirectional relationships use the same physical structures but require additional subqueries when filter propagates from many-side to one-side .

**Example: Distinct Count with Bidirectional Filter**

```dax
// Using bidirectional relationship
EVALUATE
SUMMARIZECOLUMNS (
    Product[Brand],
    "Customers", CALCULATE (
        COUNTROWS ( Customer ),
        CROSSFILTER ( Sales[CustomerKey], Customer[CustomerKey], BOTH )
    )
)
```

This requires materialization of intermediate tables and more complex query plans .

### Best Practice

Avoid bidirectional filtering unless absolutely necessary . Single-directional relationships are always faster.

## Query Plan Optimization

### CALCULATE vs. RELATED

Different formulations produce different query plans :

```dax
// Slower - forces relationship use
EVALUATE
{
    COUNTROWS (
        FILTER ( Sales, RELATED ( Product[ProductKey] ) >= 1 )
    )
}

// Faster - allows optimizer to choose plan
EVALUATE
CALCULATETABLE (
    Sales,
    Product[ProductKey] >= 1
)
```

The optimizer may detect small dimension tables and use bitmap indexes instead of relationship traversal, reducing time by **20% or more** .

### Key Column Inclusion

When the key column of a dimension is included in query results, VertiPaq generates **two storage engine queries** :

1. First query: joins fact and dimension, returns key + aggregated measure
2. Second query: retrieves all other dimension columns

This reduces memory footprint but adds overhead. Excluding the key column produces a single query.

## Measuring Chain Latency in Your Model

### Using Performance Analyzer

1. Run queries with varying relationship chain lengths
2. Compare storage engine CPU time
3. Isolate relationship cost by testing equivalent non-relationship queries

### Server Timings in DAX Studio

Key metrics to watch :
- **SE CPU**: Total storage engine processing time (best measure of real cost)
- **SE Queries**: Number of storage engine queries
- **SE Cache matches**: Indicates repeated scans

## Optimization Strategies

### 1. Reduce Chain Length

```dax
// Instead of chaining through multiple dimensions
CALCULATE (
    [Sales Amount],
    'Product'[Category] = "Electronics",
    'Customer'[Region] = "Europe"
)

// Consider denormalizing frequently used attributes
// Add Region directly to Sales table if practical
```

### 2. Use Regular Relationships

| Relationship Type | Performance Impact |
|-------------------|-------------------|
| One-to-many (regular) | Best - optimized |
| One-to-one (regular) | Good - optimized |
| Many-to-many (limited) | Poor - 9x slower |
| Cross-source (limited) | Poor - variable |

### 3. Limit Bidirectional Filtering

Disable bidirectional cross-filtering on all relationships unless specifically required .

### 4. Consider Key Column Strategy

For large dimension tables frequently queried with non-key columns, consider :
- Hiding the actual key column
- Creating a copy of the key (non-key column) for display
- Trade-off: double storage cost vs. single query execution

### 5. Optimize Filter Application

```dax
// Better - multiple filters in one CALCULATE
CALCULATE (
    [Sales Amount],
    'Product'[Category] = "Electronics",
    'Customer'[Region] = "Europe",
    'Store'[Country] = "USA"
)

// Worse - nested CALCULATE statements
CALCULATE (
    CALCULATE (
        [Sales Amount],
        'Product'[Category] = "Electronics"
    ),
    'Customer'[Region] = "Europe"
)
```

## Real-World Example

### Problem Scenario

A model with chain: Category → Subcategory → Product → Sales (4 relationships)

Query grouping by Category with 5 measures took **23 seconds**.

### Analysis

- Baseline Sales scan: 3 seconds
- Each relationship added ~2 seconds
- Limited relationship between Product and Sales (many-to-many) added 12 seconds

### Solution

1. Replaced many-to-many with bridge table (regular relationships)
2. Denormalized Category into Product table
3. Reduced chain to 2 relationships
4. Query time improved to **4 seconds**

## Best Practices Summary

### Do's
✓ **Keep chains short** - Minimize relationships between filter and fact
✓ **Use regular one-to-many** - Most optimized path
✓ **Test with large data** - Small models hide relationship costs 
✓ **Monitor SE CPU** - Best measure of relationship cost
✓ **Consider denormalization** - Trade storage for performance
✓ **Use CALCULATE** - Let optimizer choose best plan

### Don'ts
✗ **Avoid many-to-many** - 9-20x performance penalty 
✗ **Limit bidirectional** - Additional subqueries required 
✗ **Don't nest CALCULATE** - Multiple context transitions 
✗ **Avoid cross-source** - Limited relationships are slower 
✗ **Don't ignore key columns** - Affect query plan selection 

## Summary

**Key Points for Relationship Chain Latency:**

| Factor | Impact |
|--------|--------|
| **Regular relationships** | Optimized, ~30% overhead per hop |
| **Limited relationships** | 9-20x slower than regular |
| **Bidirectional filtering** | Additional subqueries |
| **Dimension cardinality** | Larger = slower translation |
| **Chain length** | Each hop adds cost |
| **Query formulation** | CALCULATE better than FILTER+RELATED |

**The Golden Rule:**
> Every relationship adds latency. Measure the cost in your model, keep chains short, use regular one-to-many relationships exclusively, and let the VertiPaq optimizer choose the best plan by using CALCULATE rather than forcing relationship traversal with RELATED.
