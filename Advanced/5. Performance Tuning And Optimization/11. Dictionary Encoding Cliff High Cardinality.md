# Dictionary Encoding Cliff: High Cardinality in Power BI

## Overview

Dictionary encoding is VertiPaq's primary compression technique, where unique column values are stored in a dictionary and replaced with numeric codes (0, 1, 2, etc.) in the column. The "dictionary encoding cliff" refers to the dramatic performance degradation that occurs when column cardinality exceeds certain thresholds, causing compression to fail and memory usage to spike.

```
Low Cardinality (<10K) → High Cardinality (10K-1M) → Extreme Cardinality (>1M)
        ↓                          ↓                          ↓
   Excellent                   Good but                   Memory
   compression                 larger                     explosion
```

## How Dictionary Encoding Works

### The Basic Mechanism

For each column, VertiPaq:
1. Scans all values to identify unique entries
2. Creates a dictionary mapping each unique value to an integer ID
3. Replaces original values with their integer IDs in the column segment

**Example - Product Category (Low Cardinality):**
```
Original: "Electronics", "Clothing", "Electronics", "Home"
Dictionary: {0:"Electronics", 1:"Clothing", 2:"Home"}
Stored as: 0, 1, 0, 2
```

**Example - Product Name (High Cardinality):**
```
Original: "Premium Wireless Headphones Black", "Premium Wireless Headphones White", ...
Dictionary: Millions of entries
Stored as: 0, 1, 2, 3, 4, 5, ... (very few repeats)
```

## The Cardinality Thresholds

### Memory Impact by Cardinality Range

| Cardinality Range | Compression Ratio | Memory per 1M rows | Performance |
|-------------------|-------------------|-------------------|-------------|
| **< 10,000** | 10-100x | 1-5 MB | Excellent |
| **10K - 100K** | 5-10x | 5-20 MB | Good |
| **100K - 1M** | 2-5x | 20-100 MB | Acceptable |
| **1M - 5M** | 1-2x | 100-500 MB | Poor |
| **> 5M** | No compression | 500+ MB | Critical failure |

### The "Cliff" Explained

The cliff occurs at multiple levels:

**Level 1: Value Encoding (2^16 = 65,536)**
- Codes stored as 2-byte integers up to 65,535 unique values
- Beyond this, codes become 4-byte integers, doubling memory

**Level 2: Segment Encoding (2^20 = ~1M)** 
- Segment encoding optimizations break down around 1M unique values
- Dictionary lookups become cache-inefficient

**Level 3: Memory Pressure (>5M)**
- Dictionary itself consumes significant memory
- Column data loses compression benefits
- May exceed available memory entirely

## Identifying High-Cardinality Columns

### Using VertiPaq Analyzer

```dax
// DAX query to analyze column cardinality
EVALUATE
SUMMARIZECOLUMNS (
    'Table'[Column],
    "Cardinality", DISTINCTCOUNT('Table'[Column])
)
ORDER BY [Cardinality] DESC
```

### Key Metrics to Monitor

```dax
// Memory usage by column
EVALUATE
SELECTCOLUMNS (
    INFO.STORAGETABLECOLUMNS(),
    "Table", [TableName],
    "Column", [ColumnName],
    "Cardinality", [Cardinality],
    "Data Size", [DataSize],
    "Dictionary Size", [DictionarySize],
    "Total Size (MB)", [DataSize] + [DictionarySize] / (1024 * 1024)
)
ORDER BY [Total Size (MB)] DESC
```

**Warning signs:**
- Dictionary size > Data size
- Cardinality > 50% of row count
- Column using > 100MB for < 10M rows

## The Dictionary Encoding Cliff in Action

### Memory Growth Pattern

```
Rows:     10M       10M       10M       10M       10M
Cardinality: 1K     10K       100K      1M        10M
Memory:    2MB      5MB       25MB      150MB     800MB+
           ↓        ↓         ↓         ↓         ↓
        Linear   Linear   Exponential   Cliff    Crash
```

The cliff appears between 100K and 1M, where memory usage suddenly explodes from linear to exponential growth.

### Performance Degradation

```dax
// Query on low-cardinality column (<10K)
Sales by Category = 
SUMX(VALUES('Product'[Category]), [Sales Amount])
// Time: 0.5 seconds

// Same query on high-cardinality column (>1M)
Sales by Product = 
SUMX(VALUES('Product'[ProductName]), [Sales Amount])
// Time: 30 seconds (60x slower)
```

## Optimization Strategies

### Strategy 1: Reduce Cardinality Through Grouping

```dax
// Instead of storing full transaction ID
// Create a grouped version
TransactionHour = 
VAR dt = 'Sales'[TransactionDateTime]
RETURN
    DATE(YEAR(dt), MONTH(dt), DAY(dt)) & " " & 
    FORMAT(HOUR(dt), "00") & ":00"

// Reduces cardinality from millions to <24*365
```

### Strategy 2: Split High-Cardinality Columns

```m
// Instead of one high-cardinality column
let
    Source = Sql.Database("server", "database"),
    Sales = Source{[Schema="dbo",Item="FactSales"]}[Data],
    
    // Split into multiple lower-cardinality columns
    WithSplit = Table.AddColumn(Sales, "Year", each Date.Year([OrderDate])),
    WithSplit2 = Table.AddColumn(WithSplit, "Month", each Date.Month([OrderDate])),
    WithSplit3 = Table.AddColumn(WithSplit2, "Day", each Date.Day([OrderDate])),
    
    // Remove original high-cardinality column
    RemoveOriginal = Table.RemoveColumns(WithSplit3, {"OrderDate"})
in
    RemoveOriginal
```

**Memory comparison:**
- Original DateTime column: 100MB (5M unique)
- Split columns (Year 10, Month 12, Day 31): <1MB total

### Strategy 3: Hash High-Cardinality Values

```dax
// Create hash of long text columns
ProductHash = 
VAR Text = 'Product'[ProductDescription]
RETURN
    CONVERT(
        RIGHT(
            CONCATENATE(
                "00000000",
                CONVERT(
                    ABS(Number.From(Text))/1000000,
                    STRING
                )
            ),
            8
        ),
        INTEGER
    )
// Reduces cardinality while maintaining ability to group
```

### Strategy 4: Remove Unnecessary High-Cardinality Columns

```m
// If TransactionID not used in measures or relationships
// Remove it entirely
let
    Source = Sql.Database("server", "database"),
    Sales = Source{[Schema="dbo",Item="FactSales"]}[Data],
    
    // Keep only necessary columns
    KeepColumns = Table.SelectColumns(
        Sales,
        {"ProductKey", "CustomerKey", "DateKey", "SalesAmount", "Quantity"}
    )
in
    KeepColumns
```

### Strategy 5: Use Value.Buffer for Repeated Values

```m
// For columns with repeated values in processing
let
    Source = Sql.Database("server", "database"),
    Sales = Source{[Schema="dbo",Item="FactSales"]}[Data],
    
    // Buffer to avoid repeated dictionary lookups
    BufferedColumn = Value.Buffer(Sales[HighCardinalityColumn])
in
    Sales
```

## Real-World Optimization Examples

### Example 1: E-Commerce Order IDs

**Before:**
- 50M rows, OrderID column (50M unique values)
- Memory: 400MB
- Relationship to Order dimension via OrderID

**After:**
- Removed OrderID from fact table
- Created OrderKey (integer surrogate key, 50M unique values)
- Memory: 150MB (62.5% reduction)
- Relationship now integer-to-integer, faster joins

### Example 2: Web Log URLs

**Before:**
- URL column with 20M unique values for 100M rows
- Memory: 800MB
- Queries by URL path extremely slow

**After:**
```m
let
    Source = Csv.Document(File.Contents("logs.csv")),
    
    // Extract domain and path components
    WithParts = Table.AddColumn(Source, "URLParts", each 
        let
            url = [URL],
            afterHttp = Text.AfterDelimiter(url, "://"),
            domain = Text.BeforeDelimiter(afterHttp, "/"),
            path = Text.AfterDelimiter(afterHttp, "/")
        in
            [Domain = domain, Path = path]
    ),
    
    // Expand components
    Expanded = Table.ExpandRecordColumn(WithParts, "URLParts", {"Domain", "Path"})
in
    Expanded
```
- Domain column: <10K unique values
- Path prefix column: <100K unique values
- Total memory: 50MB (94% reduction)

## Detecting Dictionary Encoding Issues

### VertiPaq Analyzer Script

```dax
-- High cardinality column detector
EVALUATE
VAR ColumnStats = 
    ADDCOLUMNS (
        INFO.STORAGETABLECOLUMNS(),
        "RowCount", INFO.STORAGETABLE()[RowCount],
        "CardinalityPct", 
            DIVIDE(
                [Cardinality],
                INFO.STORAGETABLE()[RowCount],
                0
            ) * 100
    )
RETURN
FILTER (
    ColumnStats,
    [CardinalityPct] > 50  -- Columns where cardinality > 50% of rows
)
ORDER BY [CardinalityPct] DESC
```

### Performance Monitor Queries

```dax
-- Identify columns with poor compression
EVALUATE
ADDCOLUMNS (
    INFO.STORAGETABLECOLUMNS(),
    "CompressionRatio",
    DIVIDE( [DataSize] + [DictionarySize], [Cardinality] ),
    "MB_per_MRows",
    DIVIDE( 
        ([DataSize] + [DictionarySize]) / (1024 * 1024),
        INFO.STORAGETABLE()[RowCount] / 1000000
    )
)
ORDER BY [MB_per_MRows] DESC
```

## The 2-Byte vs 4-Byte Threshold

### Technical Explanation

VertiPaq stores value IDs using the smallest possible data type:

| Unique Values | Bits | Bytes | Max ID | Memory Impact |
|---------------|------|-------|--------|---------------|
| < 65,536 | 16 | 2 | 65,535 | Baseline |
| 65,536 - 16.7M | 24 | 3 | 16,777,215 | 50% increase |
| > 16.7M | 32 | 4 | 4.29B | 100% increase |

The jump from 2 to 4 bytes doubles the column's storage, often causing the "cliff" effect.

### Monitoring the Threshold

```dax
-- Check columns near the threshold
EVALUATE
FILTER (
    INFO.STORAGETABLECOLUMNS(),
    [Cardinality] > 50000 AND [Cardinality] < 80000
)
```

## Best Practices Summary

### Do's
✓ **Monitor cardinality** - Regular VertiPaq Analyzer scans
✓ **Split datetime columns** - Date and time separate
✓ **Use surrogate keys** - Integers compress better than strings
✓ **Remove unused columns** - If not used in measures, drop it
✓ **Group high-cardinality text** - Extract meaningful components
✓ **Test compression** - Validate memory before production

### Don'ts
✗ **Don't store free text** - Comments, descriptions in fact tables
✗ **Don't use full URLs** - Extract domain and path components
✗ **Don't keep high-precision timestamps** - Round to hour/day
✗ **Don't ignore the 65K threshold** - 2-byte vs 4-byte matters
✗ **Don't load unnecessary columns** - Each column costs memory

## Decision Framework

```
Column identified with high cardinality (>50K)
        ↓
Is it used in relationships?
    ↓                       ↓
   Yes                      No
    ↓                       ↓
Can it be replaced      Can it be removed?
 by integer key?            ↓
    ↓                   Yes/No
   Yes → Replace        If No → Can it be grouped?
    ↓                           ↓
   Done                      Yes → Group
                               ↓
                              Done
```

## Summary

**Key Points for Dictionary Encoding Cliff:**

| Factor | Impact |
|--------|--------|
| **Cardinality < 65K** | Optimal, 2-byte encoding |
| **65K - 1M** | Acceptable, but 4-byte encoding doubles memory |
| **> 1M** | Poor compression, exponential memory growth |
| **> 5M** | Critical, likely memory failure |
| **High-cardinality text** | Worst case, minimal compression |
| **High-precision timestamps** | Can often be rounded |

**The Golden Rule:**
> The dictionary encoding cliff is real and unforgiving. Monitor column cardinality rigorously, keep high-cardinality columns out of your model when possible, and always test with representative data volumes before deployment. A single high-cardinality column can consume more memory than the rest of your model combined.
