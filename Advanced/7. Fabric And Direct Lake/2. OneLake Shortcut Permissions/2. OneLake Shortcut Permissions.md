# 2. OneLake Shortcut Permissions

## Overview

OneLake Shortcut Permissions is a critical concept in Power BI that enables efficient data access and management. It matters because it allows users to create shortcuts to frequently used data sources, models, and reports, streamlining their workflow and improving overall productivity. However, when misunderstood, it can lead to permission issues, data inconsistencies, and performance problems. The process of creating and managing OneLake Shortcut Permissions involves several stages:

```
Query → [Data Source] → [Model] → [Report] → Result
           ↓             ↓            ↓
      [Permission Check]    [Data Retrieval]   [Visualization]
```


## The Permission Framework

The permission framework of OneLake Shortcut Permissions consists of three major components:

### Data Source Permissions

Data Source Permissions determine what data sources a user can access and what actions they can perform on those sources. These permissions are crucial for ensuring data security and integrity.

**Characteristics:**
- **Read-only**: Users can only view data, not modify it.
- **Read-write**: Users can both view and modify data.
- **Admin**: Users have full control over data sources, including creating, editing, and deleting.

### Model Permissions

Model Permissions control what actions users can perform on data models, such as creating, editing, or deleting models. These permissions are essential for managing data modeling and ensuring data consistency.

**Characteristics:**
- **Model creator**: Users can create new models.
- **Model editor**: Users can edit existing models.
- **Model viewer**: Users can only view models, not modify them.

### Report Permissions

Report Permissions determine what actions users can perform on reports, such as viewing, editing, or deleting reports. These permissions are vital for managing report access and ensuring data security.

**Characteristics:**
- **Report creator**: Users can create new reports.
- **Report editor**: Users can edit existing reports.
- **Report viewer**: Users can only view reports, not modify them.


## The Permission Flow

The permission flow of OneLake Shortcut Permissions involves several stages:

### How Permissions Flow

```
[Data Source] → [Model] → [Report] → [User] → [Result]
    ↓            ↓            ↓            ↓
[Permission Check]    [Data Retrieval]   [Visualization]    [Access Control]
```

### When a User Requests Access

When a user requests access to a data source, model, or report, the system checks their permissions:

1. **Permission check**: The system checks the user's permissions on the data source, model, or report.
2. **Data retrieval**: If the user has the necessary permissions, the system retrieves the requested data.
3. **Visualization**: The system visualizes the data in a report or other format.
4. **Access control**: The system enforces access control, ensuring that the user can only perform actions they are permitted to do.


## DAX and M Language

OneLake Shortcut Permissions uses DAX and M languages to define permissions and access control. DAX is used for data modeling and calculation, while M is used for data transformation and loading.

### What is DAX?

DAX (Data Analysis Expressions) is a language used for data modeling and calculation in Power BI. It is used to define calculations, measures, and tables in a data model.

```sql
-- Example DAX query
EVALUATE
SUMMARIZE(
    'Sales',
    'Sales'[Region],
    "Total Sales", SUM('Sales'[Sales Amount])
)
```

### Example: Creating a Permission Table

```dax
-- Create a permission table
Permission Table =
SUMMARIZE(
    'Users',
    'Users'[Username],
    "Permission", 'Users'[Permission]
)
```

Then, the internal representation of this table would be:

```sql
-- Internal representation of the permission table
CREATE TABLE Permission Table (
    Username VARCHAR(255),
    Permission VARCHAR(255)
)
```
