# Merge Fact And Dimension with GDPR/Privacy constraints

**Description:**
This use case demonstrates how Power Query (M) can be used to merge fact and dimension tables while adhering to GDPR/privacy constraints, ensuring sensitive information is protected during data processing.

### Typical Scenario
In a real-world data pipeline, this transformation is crucial when dealing with customer data that must be anonymized or pseudonymized. For instance, a company might need to merge customer demographics (dimension) with transaction data (fact) without exposing sensitive personal information.

### Core M Functions Used
`Table.SelectRows`, `Table.Join`, `Table.SelectColumns`

### Code Source Execution
```powerquery
let
    // Sample fact table
    FactTable = Table.FromRows({
        {1, "2022-01-01", 100.00},
        {2, "2022-01-02", 200.00},
        {3, "2022-01-03", 300.00}
    }, {"CustomerID", "Date", "Amount"}),

    // Sample dimension table
    DimensionTable = Table.FromRows({
        {1, "John Doe", "johndoe@example.com"},
        {2, "Jane Doe", "janedoe@example.com"},
        {3, "Sam Smith", "samsmith@example.com"}
    }, {"CustomerID", "Name", "Email"}),

    // Applying GDPR/privacy constraints by removing sensitive columns
    ProtectedDimensionTable = Table.SelectColumns(DimensionTable, {"CustomerID"}),

    // Merging fact and protected dimension tables
    MergedTable = Table.Join(FactTable, ProtectedDimensionTable, "CustomerID", "CustomerID", JoinKind.Left)
    in
    MergedTable
```

### Example Execution Logic Step-By-Step
The transformation works as follows:
1. **Fact and Dimension Tables Creation**: First, sample fact and dimension tables are created. The fact table contains transactional data, while the dimension table contains customer information.
2. **Applying GDPR/Privacy Constraints**: To adhere to GDPR/privacy constraints, sensitive columns (like 'Name' and 'Email') are removed from the dimension table using `Table.SelectColumns`, leaving only the 'CustomerID'.
3. **Merging Tables**: The fact and protected dimension tables are then merged on the 'CustomerID' column using `Table.Join`. This ensures that the transactional data is combined with the non-sensitive customer information, protecting sensitive personal data.
