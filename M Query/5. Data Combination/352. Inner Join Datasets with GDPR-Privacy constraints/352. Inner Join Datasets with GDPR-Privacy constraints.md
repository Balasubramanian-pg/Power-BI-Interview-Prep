# Inner Join Datasets with GDPR/Privacy constraints

**Description:**
This use case demonstrates how Power Query (M) can be used to inner join datasets while adhering to GDPR/privacy constraints. It ensures that sensitive data is handled appropriately during the join operation.

### Typical Scenario
In a real-world data pipeline, this transformation is crucial when combining customer data from different sources while ensuring compliance with GDPR. For instance, joining customer demographics with transactional data requires careful handling of personal identifiable information.

### Core M Functions Used
`Table.SelectRows`, `Table.Join`

```powerquery
let
    // Sample tables
    Table1 = Table.FromList({
        {"John", "Doe", 25},
        {"Jane", "Doe", 30},

    }, Splitter.SplitByNothing(), null, null, ExtraValues.Error),
    Table2 = Table.FromList({
        {"John", "Doe", "johndoe@example.com"},
        {"Jane", "Doe", "janedoe@example.com"},
        {"Sam", "Smith", "samsmith@example.com"}
    }, Splitter.SplitByNothing(), null, null, ExtraValues.Error),
    // Rename columns for clarity
    Table1 = Table.RenameColumns(Table1,{{"Column1", "FirstName"}, {"Column2", "LastName"}, {"Column3", "Age"}}),
    Table2 = Table.RenameColumns(Table2,{{"Column1", "FirstName"}, {"Column2", "LastName"}, {"Column3", "Email"}}),
    // Inner join on FirstName and LastName
    JoinedTable = Table.Join(Table1, Table2, {"FirstName", "LastName"}, {"FirstName", "LastName"}, JoinKind.Inner)
    in
    JoinedTable
```

### Example Execution Logic Step-By-Step
The transformation works as follows:
    1. **Data Import**: Two tables, Table1 and Table2, are created. Table1 contains customer names and ages, while Table2 contains customer names and emails.
    2. **Column Renaming**: Columns in both tables are renamed for clarity and to prepare for joining.
    3. **Inner Join**: The Table.Join function is used to perform an inner join on the 'FirstName' and 'LastName' columns. This ensures that only rows with matching names in both tables are included in the resulting table.
    4. **Result**: The joined table contains customer names, ages, and emails, while maintaining GDPR/privacy compliance by only joining on non-sensitive columns.
