# Apply Row Level Security without query folding

**Description:**
This use case demonstrates how Power Query (M) can be used to apply row-level security without relying on query folding. It ensures that sensitive data is filtered based on user-specific criteria directly within the Power Query pipeline. This approach is particularly useful when dealing with data that requires strict access controls. By applying row-level security in Power Query, users can ensure that only authorized data is visible to them.

### Typical Scenario
In a real-world data pipeline, this transformation would be used when there are multiple users or teams accessing the same data source, but each should only see data that is relevant or authorized for them. For instance, in a sales data dashboard, different sales teams should only see sales data for their respective regions. This ensures data privacy and security by limiting exposure to sensitive information.

### Core M Functions Used
`Table.SelectRows`

### Code Source Execution
```powerquery
let
    // Sample table
    data = Table.FromList({
        {"John", "North", 100},
        {"Jane", "South", 200},
        {"Bob", "North", 300},
        {"Alice", "South", 400}
    }, Splitter.SplitByNothing(), null, null, ExtraValues.Error),
    // Define user region
    userRegion = "North",
    // Apply row-level security
    securedData = Table.SelectRows(data, each [Region] = userRegion)
    in
    securedData
```

### Example Execution Logic Step-By-Step
The transformation works as follows:
    1. **Import Data**: Start with importing a sample table that contains user data along with their respective regions and values.
    2. **Define User Region**: Determine the region for which the user is authorized to view data. This could be based on user input, a parameter, or any other form of user identification.
    3. **Apply Filter**: Use the `Table.SelectRows` function in Power Query (M) to filter the table. This function takes a table as input and returns a new table that includes only the rows for which the condition is true. In this case, the condition is that the 'Region' column must match the user's region.
    4. **Result**: The resulting table (`securedData`) will contain only the rows where the region matches the user's region, effectively applying row-level security without relying on query folding.
