# Validate Schema Changes with GDPR/Privacy constraints

**Description:**
This transformation ensures that any changes to the schema of a dataset comply with GDPR and other privacy constraints. It helps in validating that personally identifiable information (PII) is properly handled and protected. By applying this transformation, data pipelines can automatically detect and flag potential privacy breaches.

### Typical Scenario
In a real-world data pipeline, this transformation would be used when ingesting customer data from various sources into a centralized data warehouse. For instance, when integrating customer information from CRM systems, marketing databases, and support tickets, it's crucial to ensure that all personally identifiable information such as names, addresses, and phone numbers are properly anonymized or encrypted before being loaded into the warehouse for analytics and reporting purposes.

### Core M Functions Used
`Table.SelectRows`, `Table.SelectColumns`

```powerquery
    // Select rows where email is not null or empty
    FilteredRows = Table.SelectRows(Source, each [Email] <> "" and [Email] <> null),
    // Select specific columns to reduce PII exposure
    SelectedColumns = Table.SelectColumns(FilteredRows, {"FirstName", "LastName"})
    in
    SelectedColumns
```

### Example Execution Logic Step-By-Step
This transformation works by first defining a source table that contains customer data, including personally identifiable information. The Table.SelectRows function is then used to filter out any rows where the email is null or empty, ensuring that only valid and existing email addresses are processed further. Subsequently, the Table.SelectColumns function is applied to select only the necessary columns (e.g., FirstName and LastName), effectively reducing the exposure of sensitive PII. The result is a table that has been validated against GDPR/privacy constraints and is safe for further processing or analysis.
